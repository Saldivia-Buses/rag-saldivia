# Fase 3 — CI/CD Auto-deploy Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Pipeline de deploy automático via GitHub Actions que detecta pushes a `main`, despliega en Brev via SSH, hace health check, y revierte localmente si falla.

**Architecture:** GitHub Actions usa `webfactory/ssh-agent` para conectar a la instancia Brev vía deploy key ED25519. El workflow SSH-ejecuta `make deploy PROFILE=brev-2gpu`, luego llama `scripts/health_check.sh` (que verifica gateway:9000, RAG:8081, frontend:3000). Si el health check falla, el rollback step hace `git revert HEAD` localmente en Brev y redeploya — sin push al remoto para evitar loop infinito.

**Tech Stack:** GitHub Actions, bash, SSH (webfactory/ssh-agent@v0.9.0), curl, make

## Task 1: Crear `scripts/health_check.sh`

**Files:**
- Create: `scripts/health_check.sh`

El script verifica los 3 servicios críticos. Gateway tiene retry con backoff (los otros son checks únicos porque si el gateway está up, los demás deberían estarlo).

- [ ] **Step 1: Crear el script**

```bash
# scripts/health_check.sh
#!/bin/bash
set -e

MAX_RETRIES=5
SLEEP=10

echo "[health_check] Checking gateway (port 9000)..."
for i in $(seq 1 $MAX_RETRIES); do
  if curl -sf http://localhost:9000/health > /dev/null 2>&1; then
    echo "[health_check] ✓ Gateway OK"
    break
  fi
  echo "[health_check] Attempt $i/$MAX_RETRIES — waiting ${SLEEP}s..."
  sleep $SLEEP
  if [ $i -eq $MAX_RETRIES ]; then
    echo "[health_check] ✗ Gateway health check failed after ${MAX_RETRIES} attempts"
    exit 1
  fi
done

echo "[health_check] Checking RAG server (port 8081)..."
if ! curl -sf http://localhost:8081/health > /dev/null 2>&1; then
  echo "[health_check] ✗ RAG server not responding"
  exit 1
fi
echo "[health_check] ✓ RAG server OK"

echo "[health_check] Checking frontend (port 3000)..."
if ! curl -sf http://localhost:3000/ > /dev/null 2>&1; then
  echo "[health_check] ✗ Frontend not responding"
  exit 1
fi
echo "[health_check] ✓ Frontend OK"

echo "[health_check] All services healthy ✓"
```

## Task 3: Crear `.github/workflows/deploy.yml`

**Files:**
- Create: `.github/workflows/deploy.yml`

**IMPORTANTE:** El workflow asume que los 3 GitHub Secrets ya están configurados: `BREV_DEPLOY_KEY`, `BREV_HOST`, `BREV_USER`. Hacer el setup de la deploy key (ver sección arriba) antes de pushear esto.

- [ ] **Step 1: Crear el directorio y el workflow**

```bash
mkdir -p .github/workflows
```

Crear `.github/workflows/deploy.yml` con este contenido exacto:

```yaml
name: Deploy to Brev

on:
  push:
    branches: [main]
  workflow_dispatch:           # permite deploy manual desde GitHub UI

jobs:
  deploy:
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - name: Setup SSH
        uses: webfactory/ssh-agent@v0.9.0
        with:
          ssh-private-key: ${{ secrets.BREV_DEPLOY_KEY }}

      - name: Add Brev to known_hosts
        run: |
          ssh-keyscan -H ${{ secrets.BREV_HOST }} >> ~/.ssh/known_hosts

      - name: Deploy
        env:
          BREV_HOST: ${{ secrets.BREV_HOST }}
          BREV_USER: ${{ secrets.BREV_USER }}
        run: |
          ssh $BREV_USER@$BREV_HOST "
            set -e
            cd ~/rag-saldivia
            git fetch origin main
            git checkout main
            git pull origin main
            export VITE_GIT_SHA=\$(git rev-parse --short HEAD)
            make deploy PROFILE=brev-2gpu
          "

      - name: Health check
        env:
          BREV_HOST: ${{ secrets.BREV_HOST }}
          BREV_USER: ${{ secrets.BREV_USER }}
        run: |
          ssh $BREV_USER@$BREV_HOST "bash ~/rag-saldivia/scripts/health_check.sh"

      - name: Rollback on failure
        if: failure()
        env:
          BREV_HOST: ${{ secrets.BREV_HOST }}
          BREV_USER: ${{ secrets.BREV_USER }}
        run: |
          ssh $BREV_USER@$BREV_HOST "
            cd ~/rag-saldivia
            git revert HEAD --no-edit
            make deploy PROFILE=brev-2gpu
          "
          # NOTA: NO pusheamos el revert — queda local en Brev.
          # git revert --no-edit usa mensaje auto-generado "Revert '<commit>'"
          # No necesitamos [skip ci] porque este commit nunca llega a GitHub.

      - name: Notify result
        if: always()
        run: |
          STATUS=${{ job.status }}
          SHA=$(echo ${{ github.sha }} | cut -c1-8)
          echo "Deploy $STATUS — commit $SHA"
          # Conectar con webhook en Fase 13 cuando esté disponible
```

**Nota sobre escaping:** el `\$(git rev-parse --short HEAD)` lleva backslash para que bash lo evalúe en el servidor remoto (no en el runner de GitHub Actions).

## Troubleshooting común

### Error: "Host key verification failed"
El `ssh-keyscan` no encontró el host. Verificar que `BREV_HOST` tiene el valor correcto (IP sin espacios, sin `ssh://`).

### Error: "Permission denied (publickey)"
La public key no está en `~/.ssh/authorized_keys` de Brev, o el user en `BREV_USER` es incorrecto.

### Error: "make deploy" tarda >15 min
El timeout del workflow es 15 minutos. `make deploy` en Brev puede tardar si Docker descarga imágenes. Verificar que las imágenes ya están cacheadas en Brev.

### Health check falla pero servicios están up
Verificar que el gateway responde en `/health` (endpoint GET que devuelve 200). Revisar `curl -sf http://localhost:9000/health` directamente en Brev.

### Rollback no se ejecuta
El rollback solo corre cuando **algún step anterior falla**. Si el deploy succeed pero el health check tarda, el rollback no se dispara. El order es: Deploy → Health check (falla) → Rollback.
