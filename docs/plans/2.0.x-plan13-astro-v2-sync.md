## Fuera de scope

- **Semantic Router con embeddings**: el intent_router.py de Python usa FastEmbed ONNX (200MB+
  modelo). No vamos a embeber un modelo ONNX en el servicio Go. En cambio, expandimos el
  keyword matcher existente con las 1,817 utterances como reglas de matching, y dejamos
  preparada la interfaz para upgrade futuro via LLM o embedding service externo.
- **Agent audit (Haiku quality check)**: el auto-fix learning system de Python (Haiku evalua
  y ajusta system prompt) es Python-specific. Nuestro audit Go es deterministic (<10ms).
  Si necesitamos LLM-based audit, sera plan separado.
- **Company lookup (CUIT scraping)**: web scraping de AFIP/cuitonline es fragil y
  Argentina-specific. Se puede agregar como tool del Agent Runtime en plan futuro.

### Fase 2: Intent Router Upgrade (1,817 utterances)

**Que:** Reemplazar el keyword matcher basico con un sistema de utterance scoring
que usa las 1,817 utterances de Python como diccionario de matching.

**Approach:** NO usamos embeddings (seria agregar ONNX runtime al Go binary).
En cambio, usamos **TF-IDF-like scoring** sobre las utterances:

1. Pre-procesar las 1,817 utterances en startup: normalizar, tokenizar, construir
   inverted index de tokens → route
2. En runtime: tokenizar query del usuario, buscar tokens en el index, scoring por
   overlap ponderado (utterances que comparten mas tokens con la query ganan)
3. Confidence threshold: si score < 0.3, fallback al keyword matcher actual
4. Si keyword matcher tambien falla, fallback a "predictivo"

**Archivos:**
- `intelligence/intent.go` — refactor: extraer interface `IntentParser`, mantener
  `KeywordParser` como implementacion existente
- `intelligence/utterance_router.go` — NEW: TF-IDF utterance scorer, implementa `IntentParser`
- `intelligence/utterance_data.go` — NEW: 1,817 utterances via `//go:embed utterances.json`
  (embed es mejor que string literals para data plana — no afecta compilacion, permite
  updates sin recompilar, type safety via json.Unmarshal en init)
- `intelligence/utterances.json` — NEW: data file embebido (generado desde Python `data/utterances.json`)
- `intelligence/intelligence.go` — refactor: `Engine` recibe `IntentParser` en constructor,
  `Analyze()` llama `e.parser.Parse()` en vez de `ParseIntent()` directo (fix H2)
- `intelligence/utterance_router_test.go` — NEW: test matrix con queries ambiguas

**Interface para upgrade futuro:**
```go
type IntentParser interface {
    Parse(query string) Intent
}
```
Implementaciones: `KeywordParser` (actual), `UtteranceRouter` (nueva), futuro:
`EmbeddingRouter` (via llamada a servicio externo) o `LLMRouter` (via pkg/llm).

**Wiring (fix H2):**
```go
// intelligence.go — NewEngine ahora recibe parser
func NewEngine(registry *DomainRegistry, parser IntentParser, log *slog.Logger) *Engine

// main.go — wiring
parser := intelligence.NewUtteranceRouter(registry)
engine := intelligence.NewEngine(registry, parser, logger)
```

**Tests:**
- Matriz de 100+ queries ambiguas con dominio esperado (portadas de los test suites de Python)
- Accuracy target: >85% (Python con embeddings logra ~95%)
- Regression: queries que ya funcionaban con keyword no deben romper

### Fase 5: Brief Optimizations (3 mejoras)

**Que:** Portar las 3 optimizaciones de brief que reducen tokens 40-60%.

#### 5a: Brief Compressor (BCA)

Regex-based compression que preserva activaciones, meses, orbes, planetas, casas,
scores — pero remueve headers decorativos, patrones repetidos, titulos verbosos.
Python logra 40-60% reduccion en ~5ms.

**Archivos:**
- `intelligence/compressor.go` — expandir `CompressBrief()` con patterns de compresion BCA

#### 5b: Top-N Activation Filter

Limitar activaciones a top 5-10 por orbe across all techniques. Evita briefs con
50+ activaciones donde el LLM se pierde.

**Archivos:**
- `context/scoring.go` — agregar `FilterTopN(activations, maxPerTechnique)` function
- `context/builder.go` — invocar filtro ANTES de `BuildBrief()` (fix M1: el brief se
  genera al final de `Build()` en linea ~245, asi que el filtro debe aplicarse antes,
  no despues)

#### 5c: Domain-Aware Lazy Calc

Calcular solo tecnicas relevantes al dominio detectado. Si el dominio es "salud",
no calcular synastry, electional, horary, etc.

**Problema arquitectural (fix C4):** Actualmente el handler llama `context.Build(chart)`
ANTES de `intel.Analyze(query)`, asi que el dominio no esta resuelto cuando Build corre.
La funcion es `Build()` (no `BuildFullContext()`), signature:
`func Build(chart *natal.Chart, contactName string, birthDate time.Time, year int) (*FullContext, error)`

**Solucion: 2-pass approach.**
1. **Quick intent parse:** Agregar `QuickDomain(query string) string` en `intelligence/intent.go`
   que solo detecta dominio sin el pipeline completo de Analyze. Es rapido (regex/utterance).
2. **Handler refactor:** El handler llama `QuickDomain(query)` ANTES de `Build()`,
   y pasa el domainID a un nuevo `BuildWithDomain(chart, name, date, year, domainID)`.
3. **BuildWithDomain:** Si domainID != "" y != "predictivo", resuelve el dominio y
   skip tecnicas irrelevantes. `Build()` original sigue existiendo sin domainID (calcula todo).
4. `Analyze()` despues recibe el FullCtx ya filtrado — sin cambios.

**Archivos:**
- `context/builder.go` — agregar `BuildWithDomain()` que acepta `domainID string` +
  referencia a `*DomainRegistry` para resolver que tecnicas skip
- `intelligence/intent.go` — agregar `QuickDomain(query string) string`
- `handler/astro.go` — refactor linea 783: `QuickDomain()` → `BuildWithDomain()` → `Analyze()`

**Tests:**
- Brief BCA es <60% del tamano original
- Top-N: max 10 activaciones por tecnica en output
- Lazy calc: query "salud" no calcula synastry/electional/horary
- Lazy calc: query "predictivo" calcula todo (fallback)
- Lazy calc: query vacia calcula todo (fallback)

### Fase 7: Follow-Up Continuity

**Que:** Detectar mensajes cortos de follow-up y heredar contexto de la sesion anterior.

**Archivos:**

- `intelligence/intelligence.go` — integrar followup detection en pipeline Analyze()
- `handler/astro.go` — agregar `session_id` opcional al request body (fix C5) + cargar
  ultimos mensajes de sesion via `h.q.ListMessages(sessionID)`

**Request body change (fix C5):**
```go
var req struct {
    ContactName string `json:"contact_name"`
    Query       string `json:"query"`
    Year        int    `json:"year"`
    SessionID   string `json:"session_id,omitempty"` // NEW: optional, for follow-up detection
}
```
Si `session_id` no se envia, el handler intenta encontrar la sesion mas reciente
del contacto como fallback. Si no hay sesion, no hay follow-up detection.

**Logica de deteccion (fix M5 — mas robusta que solo longitud):**
1. Mensaje < 25 palabras
2. NO contiene datos natales (fecha, hora, lugar de nacimiento)
3. Contiene indicadores de continuidad: pronombres referenciales ("y", "tambien",
   "ademas", "pero"), preguntas sobre meses/planetas sin contexto nuevo
   ("en mayo?", "y Jupiter?", "que pasa con eso?"), o empieza con conjuncion
4. HAY sesion activa con al menos 1 mensaje assistant en historial

Solo si las 4 condiciones se cumplen → es follow-up.

**Herencia:**
- contactID del mensaje anterior
- domainID del analisis anterior (guardado en message metadata JSONB)
- brief cacheado de la sesion (via Redis context cache)
- LLM recibe: brief anterior + response anterior (ultimos 500 chars) + nuevo mensaje

**Tests:**
- "Y en mayo?" detectado como follow-up (continuacion, tiene "y")
- "Que pasa con Jupiter?" detectado como follow-up (referencia sin contexto)
- "Carta de Juan Garcia" NO detectado (nueva consulta, nombre propio)
- "Amor 2027" NO detectado (nueva consulta, dominio explicito)

### Fase 13: Proactive Alerts System

**Que:** Scanner de contactos que detecta SA/DP/eclipse exactos esta semana.

**Archivos:**
- `intelligence/alerts.go` — NEW: `ScanAlerts(contacts []repository.Contact, year int) []Alert`
- `handler/astro.go` — NEW endpoint: `GET /v1/astro/alerts` (permission: `astro.read`)
- `repository/queries.sql` — NEW query: `ListAllContactsForTenant` (fix C6: query necesaria
  para que el cron pueda escanear contactos sin user context)

**DB query nueva (fix C6):**
```sql
-- name: ListAllContactsForTenant :many
SELECT * FROM contacts
WHERE tenant_id = $1
  AND birth_year IS NOT NULL
  AND birth_month IS NOT NULL
  AND birth_day IS NOT NULL
ORDER BY updated_at DESC
LIMIT 50;
```
Nota: esto requiere `make sqlc` pero NO requiere migracion DB (la tabla contacts ya existe).

**Logica:**
1. Para cada contacto con datos natales, calcular SA + PD (solo esas 2 tecnicas, rapido)
2. Si orbe SA < 0.15° (≈8 semanas) → alerta urgente
3. Si orbe SA < 0.05° (≈1 semana) → alerta critica
4. Si eclipse de este ano cae en planeta natal (< 2°) → alerta eclipse
5. Max 10 contactos procesados por scan (limitar carga)

**Endpoint GET (fix M6):**
- `GET /v1/astro/alerts?limit=10` — escanea los 10 contactos mas recientes del user
  autenticado (usa `tenant_id + user_id` del JWT, como todos los endpoints)
- Response: `{alerts: [{contact_name, type, detail, urgency, orb}]}`

**Cron (fix C6):**
- `time.Ticker` en main.go, lunes 6am Argentina
- Usa `ListAllContactsForTenant` con tenant_id hardcoded (single-tenant = "saldivia")
- NO tiene user_id — escanea todos los contactos del tenant
- NATS publish en `tenant.{slug}.notify.astro.alert` sin user_id en el payload
  (el WS Hub broadcast a todos los users del tenant — aceptable para single-tenant)
- Si migran a multi-tenant, el cron necesitara iteracion por user

**Tests:**
