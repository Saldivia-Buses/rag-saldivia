# Plan 21 — RBAC granular + Admin superuser experience

> **Branch:** 1.0.x
> **Prerequisito:** Batch 3 (Admin Users básico ya funciona)
> **Sprint:** think → plan → execute → review → test → ship
> **Intensity:** thorough

## Contexto

Enzo quiere un sistema de permisos granular real: roles custom, permisos por feature,
usuarios con múltiples roles, y un panel admin que lo haga sentir superuser.
El sistema actual tiene 3 roles hardcodeados (`admin`, `area_manager`, `user`) en un
campo `users.role` y regex en el middleware. No es extensible.

**Decisión de Enzo:** "quiero control granular de permisos para cada rol, creación
de roles, múltiples roles por usuario, el rol más fuerte prevalece pero los permisos
laterales se suman."

## Arquitectura del RBAC

### Modelo de datos

```
roles (catálogo)              permissions (catálogo)
├── Admin (level 100)         ├── admin.access
├── Manager (level 50)        ├── users.view
├── User (level 10)           ├── users.manage
└── [custom...]               ├── roles.manage
                              ├── chat.use
     role_permissions         ├── chat.manage_all
     (N:M matrix)             ├── collections.read
                              ├── collections.manage
     user_role_assignments    ├── upload.use
     (N:M, multi-rol)        ├── templates.manage
                              └── settings.manage
```

### Resolución de permisos

```typescript
// Para un usuario con roles [Editor, Uploader]:
//   Editor.permissions    = { chat.use, collections.read, templates.manage }
//   Uploader.permissions  = { upload.use, collections.read }
//
//   Permisos efectivos = UNIÓN = { chat.use, collections.read, templates.manage, upload.use }
//
// El "level" más alto de sus roles determina el rol principal (para display).
// Pero los permisos siempre se suman — nunca se restan.
```

### Backward compatibility

El campo `users.role` se mantiene como "legacy fallback". El nuevo sistema usa
`user_role_assignments` pero si un usuario no tiene roles asignados, se lee `users.role`
y se mapea al rol default equivalente. Esto evita migración destructiva.

## Fases

### Fase 1: Schema + tablas + types

**Archivos:**
- `packages/db/src/schema.ts`

**Cambios:**

Agregar 4 tablas al final del schema (antes de relations):

```typescript
// ── RBAC: Roles ──────────────────────────────────────────────────────────

export const roles = sqliteTable("roles", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  name: text("name").notNull().unique(),
  description: text("description").notNull().default(""),
  level: integer("level").notNull().default(0), // higher = more powerful
  color: text("color").notNull().default("#6e6c69"), // hex for UI badges
  icon: text("icon").notNull().default("user"), // lucide icon name
  isSystem: integer("is_system", { mode: "boolean" }).notNull().default(false),
  createdAt: integer("created_at").notNull(),
})

// ── RBAC: Permission catalog ─────────────────────────────────────────────

export const permissions = sqliteTable("permissions", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  key: text("key").notNull().unique(),     // e.g. "users.manage"
  label: text("label").notNull(),          // e.g. "Gestionar usuarios"
  category: text("category").notNull(),    // e.g. "Usuarios"
  description: text("description").notNull().default(""),
})

// ── RBAC: Role ↔ Permission (many-to-many) ───────────────────────────────

export const rolePermissions = sqliteTable(
  "role_permissions",
  {
    roleId: integer("role_id").notNull().references(() => roles.id, { onDelete: "cascade" }),
    permissionId: integer("permission_id").notNull().references(() => permissions.id, { onDelete: "cascade" }),
  },
  (t) => ({
    pk: primaryKey({ columns: [t.roleId, t.permissionId] }),
  })
)

// ── RBAC: User ↔ Role (many-to-many) ─────────────────────────────────────

export const userRoleAssignments = sqliteTable(
  "user_role_assignments",
  {
    userId: integer("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
    roleId: integer("role_id").notNull().references(() => roles.id, { onDelete: "cascade" }),
    assignedAt: integer("assigned_at").notNull(),
  },
  (t) => ({
    pk: primaryKey({ columns: [t.userId, t.roleId] }),
  })
)
```

Agregar relations:

```typescript
export const rolesRelations = relations(roles, ({ many }) => ({
  rolePermissions: many(rolePermissions),
  userRoleAssignments: many(userRoleAssignments),
}))

export const permissionsRelations = relations(permissions, ({ many }) => ({
  rolePermissions: many(rolePermissions),
}))

export const rolePermissionsRelations = relations(rolePermissions, ({ one }) => ({
  role: one(roles, { fields: [rolePermissions.roleId], references: [roles.id] }),
  permission: one(permissions, { fields: [rolePermissions.permissionId], references: [permissions.id] }),
}))

export const userRoleAssignmentsRelations = relations(userRoleAssignments, ({ one }) => ({
  user: one(users, { fields: [userRoleAssignments.userId], references: [users.id] }),
  role: one(roles, { fields: [userRoleAssignments.roleId], references: [roles.id] }),
}))
```

Agregar a `usersRelations`:
```typescript
// Agregar al existente:
userRoleAssignments: many(userRoleAssignments),
```

Agregar type exports:
```typescript
export type DbRole = typeof roles.$inferSelect
export type NewRole = typeof roles.$inferInsert
export type DbPermission = typeof permissions.$inferSelect
export type DbRolePermission = typeof rolePermissions.$inferSelect
export type DbUserRoleAssignment = typeof userRoleAssignments.$inferSelect
```

**Verificación:**
- [ ] `bunx tsc --noEmit` → 0 errors
- [ ] Schema es consistente con las convenciones existentes

**Commit:** `feat(db): add RBAC schema — roles, permissions, mappings — plan21 f1`
