# Plan 24 — Admin System: Collections, Areas, Permissions, Config

> **Branch:** 1.0.x
> **Prerequisito:** Plan 23 (code hardening)
> **Sprint:** think -> plan -> execute -> review -> test -> ship
> **Intensity:** thorough
> **Orden:** Plan 23 -> Plan 24 -> Plan 22

---

## Contexto

Auditoria completa del sistema revelo que el admin esta a mitad de camino:

**Activo y funcionando (3/7 secciones):**
- `/admin` — Dashboard con stats, presencia, distribucion de roles
- `/admin/users` — CRUD de usuarios con asignacion de roles RBAC (575 lineas)
- `/admin/roles` — CRUD de roles con matriz de permisos completa (610 lineas)

**Infraestructura de DB existe pero sin UI (4/7 secciones):**
- **Areas** — queries: `listAreas`, `createArea`, `updateArea`, `deleteArea`,
  `setAreaCollections`, `countUsersInArea`. Actions archivadas.
- **Permissions** — queries: `setAreaCollections`, `addAreaCollection`,
  `removeAreaCollection`, `getUserCollections`, `canAccessCollection`. Sin UI.
- **RAG Config** — actions activas (`actionUpdateRagParams`, `actionResetRagParams`).
  Componente archivado. Sin pagina.
- **Collections** — solo lista basica. Sin detail, sin stats, sin historial.

**Bugs de colecciones:**
- DELETE no invalida cache Redis (60s stale)
- POST devuelve `apiOk()` cuando RAG server falla (error silencioso)
- Auth inconsistente entre rutas
- "tecpia" hardcoded en 3 lugares
- `RagGenerateRequest` no incluye `collection_names` en el tipo
- No hay validacion Zod de nombres de colecciones

**Codigo recuperable del archive:**
- `AreasAdmin.tsx` (177 lineas) — CRUD areas con table view
- `PermissionsAdmin.tsx` (169 lineas) — matriz area-coleccion read/write
- `RagConfigAdmin.tsx` (113 lineas) — sliders para params RAG
- `_archive/actions/areas.ts` — server actions de areas
- `CollectionSelector.tsx` — multi-collection selector para chat
- `AdminPanel.tsx` — sidebar nav con 10 items

**Conexion con el sistema completo:**
```
Users ←→ UserAreas ←→ Areas ←→ AreaCollections ←→ Collections (Milvus)
  ↓                                                        ↓
UserRoleAssignments ←→ Roles ←→ RolePermissions     ChatSessions
                                                     IngestionJobs
```
Todo esto existe en la DB. Este plan conecta las piezas faltantes.

### Fase 1: Fix collection bugs + type safety

**Archivos:**
- `apps/web/src/app/api/rag/collections/[name]/route.ts`
- `apps/web/src/app/api/rag/collections/route.ts`
- `apps/web/src/app/actions/collections.ts`

- `apps/web/src/lib/rag/client.ts`
- `packages/shared/src/schemas.ts`
- `apps/web/src/app/api/rag/generate/route.ts`

**Cambios:**

1. **Cache invalidation en DELETE route** — agregar `invalidateCollectionsCache()`
2. **Auth consistente en DELETE** — usar `requireAuth`+`requireAdmin` (no extractClaims raw)
3. **Error handling real:** eliminar `catch { return apiOk() }` silencioso.
   Si RAG server falla → `apiError("No se pudo crear la colección", 502)`
4. **Redis graceful degradation en collections-cache:** try/catch en get/set,
   bypass cache si Redis esta caido
5. **Fix mock mode en client.ts:** responses apropiadas por method/path
6. **CollectionNameSchema Zod:**
   ```typescript
   export const CollectionNameSchema = z.string()
     .min(1).max(64)
     .regex(/^[a-z0-9_-]+$/, "Solo minúsculas, números, guiones")
   ```
7. **Actualizar `RagGenerateRequest`:** agregar `collection_names?: string[]`
8. **Extender `GenerateBodySchema`** (creado por Plan 23 F3b): agregar
   `collection_names: z.array(CollectionNameSchema).optional()` al schema Zod existente
9. **Validar nombres** en POST create y DELETE con CollectionNameSchema

**Verificacion:**
- [ ] `bunx tsc --noEmit` -> 0 errors
- [ ] DELETE invalida cache inmediatamente
- [ ] POST con RAG down retorna 502
- [ ] Collection con nombre invalido retorna 400

**Commit:** `fix(rag): collection bugs, cache invalidation, zod schemas, types — plan24 f1`
