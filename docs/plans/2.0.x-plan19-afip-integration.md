## Contexto

Toda empresa argentina que factura necesita autorizar sus comprobantes ante
AFIP (Administracion Federal de Ingresos Publicos) via webservice. Histrix
tiene esto implementado en PHP con SOAP directo. SDA necesita su propia
integracion para:

1. **Emitir facturas electronicas** con CAE (Codigo de Autorizacion Electronico)
2. **Generar el Libro IVA Digital** en formato AFIP (CITI)
3. **Consultar el padron de contribuyentes** (validar CUIT, condicion IVA)

> **Out of scope V1:** SIFERE (retenciones federales) se postpone a V2.
> Razon: protocolo legacy, bajo uso, anade complejidad al plan sin valor
> inmediato. La arquitectura CITI se deja extensible para agregar SIFERE
> despues sin refactor.

**Sin AFIP, las facturas emitidas no tienen validez fiscal.**

## Principios

1. **AFIP es un sistema externo inestable.** Reintentos, timeouts, circuit breaker.
2. **Separar protocolo de logica.** `pkg/afip/` maneja SOAP/WSDL, `services/erp/` maneja logica.
3. **Ambiente de homologacion primero.** Todo se desarrolla contra el sandbox de AFIP.
4. **Certificados por tenant.** Cada empresa tiene su propio certificado digital AFIP.

### SOAP approach

AFIP usa SOAP/XML exclusivamente. Go no tiene soporte nativo de SOAP. Approach:

- **Manual XML marshaling con `encoding/xml`** (recomendado). Los WSDLs de AFIP
  son non-standard y rompen los code generators. Es mas confiable escribir los
  structs manualmente con tags `xml:` y hacer HTTP POST con el envelope construido.
- `xml.go` implementa un helper para construir SOAP envelopes genericos.
- NO usar `gowsdl` ni otros code generators — AFIP devuelve XML malformado en
  algunos casos edge que necesitan handling manual.

### Flujo de autenticacion AFIP (WSAA)

```
1. ERP tiene certificado digital (.crt) + clave privada (.key) por tenant
2. Firma un TRA (Ticket de Requerimiento de Acceso) con la clave privada
3. Envia el TRA firmado al WSAA
4. Recibe un TA (Ticket de Acceso) con token + sign (valido 12hs)
5. Usa token + sign en cada request a WSFE, padron, etc.
```

Los TAs se cachean en Redis (TTL = expiration del TA - 5 minutos).

### Flujo de encryption en PUT /config

El frontend envia el cert + key **en claro** via multipart form:
- `cert` (file, .crt PEM)
- `key` (file, .key PEM)
- `cuit`, `environment`, `sale_point` (form fields)

El **backend encripta** antes de guardar, usando `pkg/crypto.Encryptor.Encrypt(plaintext, aad)`
con `aad = []byte(slug)`. El handler nunca persiste bytes en claro — el cert/key solo
existen en memoria durante el scope del handler:

```go
func (h *AFIP) PutConfig(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    slug := tenant.FromContext(ctx)
    aad := []byte(slug)

    certBytes, _, _ := r.FormFile("cert")  // read fully
    certPlain, _ := io.ReadAll(certBytes)
    defer zeroize(certPlain)

    keyBytes, _, _ := r.FormFile("key")
    keyPlain, _ := io.ReadAll(keyBytes)
    defer zeroize(keyPlain)

    // Encrypt antes de persistir
    certEncDEK, certEncData, err := h.encryptor.Encrypt(certPlain, aad)
    if err != nil { ... }
    keyEncDEK, keyEncData, err := h.encryptor.Encrypt(keyPlain, aad)
    if err != nil { ... }

    h.svc.SaveConfig(ctx, slug, &AFIPConfig{
        CUIT: r.FormValue("cuit"),
        Environment: r.FormValue("environment"),
        SalePoint: parseInt(r.FormValue("sale_point")),
        CertEncDEK: certEncDEK, CertEncData: certEncData,
        KeyEncDEK: keyEncDEK, KeyEncData: keyEncData,
    })
}
```

**Transport security:** El PUT viaja sobre TLS (Traefik termina TLS, traffic interno
es plaintext en la red inhouse). Acceptable dado el threat model inhouse. Si en el
futuro el threat model cambia, el frontend puede pre-encriptar con una KEK publica
sin cambiar este handler — solo se saltea el `Encrypt` en el backend.

## Fase 1 — Facturacion electronica (WSFE + CAE)

**Objetivo:** Solicitar CAE a AFIP para cada factura emitida. El CAE es
obligatorio para que la factura tenga validez fiscal.

### Flujo

```
1. Usuario crea factura en SDA (status='draft')
2. Usuario clickea "Autorizar AFIP" (o se autoriza automaticamente al postear)
3. Service layer:
   a. Obtiene TA via WSAA (cache o nuevo)
   b. Construye FECAERequest con datos de la factura
   c. Llama a FECAESolicitar en WSFE
   d. Recibe CAE + fecha vencimiento
   e. Guarda CAE en erp_invoices (afip_cae, afip_cae_due)
   f. Genera QR con datos fiscales
   g. Marca factura como 'posted'
4. Si AFIP rechaza: guarda errores, factura queda en 'draft'
```

### Service layer

```go
func (s *AFIP) AuthorizeInvoice(ctx context.Context, slug string, invoiceID uuid.UUID) (*AuthorizeResult, error) {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    inv := s.repo.GetInvoice(ctx, tx, slug, invoiceID)
    if inv.Status != "draft" { return nil, errors.New("solo draft se puede autorizar") }
    if inv.Direction != "issued" { return nil, errors.New("solo comprobantes emitidos") }

    config := s.repo.GetAFIPConfig(ctx, tx, slug)

    // Decrypt cert + key con pkg/crypto/Encryptor (AAD = tenant_id).
    // ESTE es el unico lugar donde el cert/key existen en memoria en claro.
    // Variables locales se sobrescriben con zero bytes al final del scope via defer.
    aad := []byte(slug)
    cert, err := s.encryptor.Decrypt(config.CertEncDEK, config.CertEncData, aad)
    if err != nil { return nil, fmt.Errorf("decrypt cert: %w", err) }
    defer zeroize(cert)
    key, err := s.encryptor.Decrypt(config.KeyEncDEK, config.KeyEncData, aad)
    if err != nil { return nil, fmt.Errorf("decrypt key: %w", err) }
    defer zeroize(key)

    // Autenticar con cert/key desencriptados
    ta, err := s.wsaa.Authenticate(ctx, slug, "wsfe", cert, key)
    if err != nil { return nil, fmt.Errorf("WSAA auth: %w", err) }

    // Construir request
    cbteTipo := mapInvoiceType(inv.InvoiceType)
    cbteNro := s.getNextNumber(config, cbteTipo)
    req := s.buildFECAERequest(inv, config, cbteTipo, cbteNro)

    // Solicitar CAE
    resp, err := s.wsfe.SolicitarCAE(ctx, ta, req)

    // Log SOAP (siempre, aprobado o rechazado)
    s.repo.CreateAFIPLog(ctx, tx, slug, invoiceID, "FECAESolicitar", reqXML, respXML, resp)

    if err != nil || resp.Resultado == "R" {
        tx.Commit(ctx)
        return &AuthorizeResult{Approved: false, Errors: resp.Errores}, nil
    }

    // Guardar CAE
    s.repo.SetInvoiceCAE(ctx, tx, slug, invoiceID, resp.CAE, resp.CAEFchVto)
    s.repo.UpdateLastNumber(ctx, tx, slug, cbteTipo, cbteNro)

    tx.Commit(ctx)
    s.audit.Strict(ctx, "afip_authorize", ...)
    return &AuthorizeResult{Approved: true, CAE: resp.CAE}, nil
}
```

type CITIComprasLine struct {
    Fecha           string  // YYYYMMDD (8)
    Tipo            string  // codigo comprobante (3)
    PtoVenta        string  // 5 digits right-aligned
    NroComprobante  string  // 20 digits right-aligned

func (s *AFIP) GenerateCITI(ctx context.Context, slug string, period string, direction string) (string, error) {
    invoices := s.repo.ListInvoicesForCITI(ctx, slug, period, direction)
    var buf bytes.Buffer
    for _, inv := range invoices {
        line := formatCITILine(inv)
        buf.WriteString(line + "\r\n")
    }
    key := fmt.Sprintf("afip/%s/citi_%s_%s.txt", slug, direction, period)
    s.storage.Upload(ctx, key, buf.Bytes())
    s.repo.CreateTaxExport(ctx, slug, ...)
    return key, nil
}

func (s *AFIP) GetPadron(ctx context.Context, slug string, cuit string) (*PersonaReturn, error) {
    // 1. Buscar en cache (< 24hs)
    cached := s.repo.GetPadronCache(ctx, cuit)
    if cached != nil && cached.FetchedAt.After(time.Now().Add(-24*time.Hour)) {
        return cached.Data, nil
    }

    // 2. Consultar AFIP
    config := s.repo.GetAFIPConfig(ctx, slug)
    ta, err := s.wsaa.Authenticate(ctx, slug, "ws_sr_padron_a10", config.Certificate, config.PrivateKey)
    if err != nil { return nil, err }

    persona, err := s.padron.GetPersona(ctx, ta, parseCUIT(cuit))
    if err != nil { return nil, err }

    // 3. Guardar en cache
    s.repo.UpsertPadronCache(ctx, cuit, persona)
    return persona, nil
}
```

   CREATE OR REPLACE FUNCTION erp_afip_log_immutable() RETURNS trigger AS $$
   BEGIN
       RAISE EXCEPTION 'erp_afip_log is append-only';
   END;
   $$ LANGUAGE plpgsql;

   CREATE TRIGGER trg_afip_log_immutable BEFORE UPDATE OR DELETE ON erp_afip_log
       FOR EACH ROW EXECUTE FUNCTION erp_afip_log_immutable();
   ```

2. **Decrypt location:** El decrypt del cert/key se hace **en el service layer
   justo antes de llamar a `wsaa.Authenticate`**, NO en el handler ni en el repo.
   Ver pseudocodigo de `AuthorizeInvoice` en fase 1. Las variables locales se
   zero-ean con `defer zeroize(cert)` para evitar que queden en heap despues
   del scope. `pkg/crypto/Encryptor.Decrypt(encDEK, encData, aad)` ya existe.

3. **Multiples puntos de venta (out of scope V1):** `erp_afip_config.sale_point`
   es unico por tenant. Para V2, convertir en tabla separada `erp_afip_sale_points`
   con FK a config. El schema actual NO bloquea esta migracion — agregar la tabla
   y migrar el valor default es suficiente.

4. **QR endpoint:** `GET /invoices/{id}/qr` esta listado en fase 1 endpoints.
   Implementacion: el handler lee el CAE + metadata de la invoice, construye el
   `QRData`, serializa a JSON, encodea en base64, y genera el QR PNG con
   `github.com/skip2/go-qrcode`. El URL codificado es
   `https://www.afip.gob.ar/fe/qr/?p={base64}`. Content-Type `image/png`.

5. **SIFERE V2 extension path:** Cuando se implemente SIFERE, agregar `export_type`
   valores `'sifere_retenciones'`, `'sifere_percepciones'` y crear el generator
   en `services/erp/internal/service/afip_sifere.go`. No requiere migration.

6. **Anulacion AFIP via NCA (no hay cancel CAE):** AFIP **no tiene un endpoint
   para cancelar un CAE ya emitido**. La unica forma de anular una factura con
   CAE ante AFIP es **emitir una Nota de Credito (NCA o NCB segun el tipo)** que
   referencie la factura original. El CAE de la factura original queda valido
   para siempre — lo que cambia es que existe una NCA que la contrarresta.

   Implicancia: cuando Plan 18 fase 2 (`VoidInvoice`) se ejecuta sobre una
   factura con CAE, el workflow debe:
   1. Crear la NCA inversa (`erp_invoices` nuevo con `invoice_type = 'NCA'`
      o `'NCB'`, referenciando via `voided_by`).
   2. **Solicitar un nuevo CAE para la NCA** via `AuthorizeInvoice` de este plan.
   3. Marcar la factura original como `cancelled` (trigger de Plan 18 lo permite).
   4. El contra-asiento, reversa IVA, etc. son efectos de Plan 18.

   **El CAE de la factura original NO se toca.** Si AFIP rechaza el CAE de la NCA,
   toda la TX de VoidInvoice se rollbackea (incluido el contra-asiento). Este
   comportamiento se documenta en Plan 18 fase 2 como dependencia de Plan 19.

## Consideraciones

- **Certificado AFIP:** cada empresa necesita tramitar su certificado digital
  ante AFIP. El proceso es manual y tarda 1-3 dias habiles.
- **Ambiente homologacion:** AFIP tiene un sandbox completo. Todo este plan se
  desarrolla y testea contra homologacion antes de apuntar a produccion.
- **Rate limits:** AFIP no documenta rate limits oficiales, pero en la practica
  rechaza mas de ~50 requests/minuto. El circuit breaker en `pkg/afip/client.go`
  maneja esto.
- **Downtime AFIP:** El webservice se cae regularmente. Las facturas quedan en
  `draft` hasta que AFIP responda. El frontend muestra el estado claramente.
