# Plan 21 — Data Migration: Histrix MySQL → SDA PostgreSQL

> **Prerequisito:** Plan 17 + 18 + 19 completados (ERP funcional con paridad).
> Plan 20 es opcional (no bloquea la migracion).
> **Fecha:** 2026-04-10
> **Estado:** Draft — pendiente aprobacion
> **Version target:** 2.0.10
> **Sprint:** think > plan > execute > review > test > ship (por fase)
> **Servicios afectados:** `tools/cli/`, `db/tenant/migrations/`, documentacion
> **Estimacion:** 6 fases, ~7-9 dias (agregada fase de pre-validation contra constraints Plan 18)
> **Migrations:** 040 (id mapping + progress tracking + validation issues). 039 la usa plan 20 (`erp_pending_exports`).
> **Direccion:** One-way (MySQL → PostgreSQL). Sin rollback automatico — si algo sale mal
> se dropean las tablas SDA del tenant y se re-corre la migracion. Este plan asume que
> la branch 2.0.x aun no fue desplegada a produccion, por lo que no hay datos del cliente
> que proteger mas alla de lo ya en MySQL legacy.

## Contexto

Histrix tiene 15+ anos de datos operativos en MySQL. Reemplazar el sistema sin
migrar los datos no es una opcion: la contadora necesita el historial de
asientos, el jefe de compras necesita las OC anteriores, RRHH necesita los
legajos completos.

**Este plan migra los datos historicos de MySQL a PostgreSQL.** El sistema nuevo
(SDA) ya esta funcionando. Histrix sigue read-only como referencia hasta que
la migracion este validada.

## Principios

1. **MySQL es read-only.** Nunca escribir en la base legacy. Solo SELECT.
2. **Migrar por dominio, en orden de FK.** Catalogos primero, luego entidades,
   luego documentos que referencian entidades.
3. **IDs cambian.** Legacy usa INT auto-increment, SDA usa UUID. La tabla de
   mapping es la verdad.
4. **Idempotente.** Correr la migracion 2 veces produce el mismo resultado.
   Implementado con `INSERT ... ON CONFLICT DO NOTHING` en el batch writer
   y la tabla de mapping que evita regenerar UUIDs en re-ejecuciones.
5. **Validacion automatica.** Counts, sumas de control, integridad referencial.
6. **Dry-run obligatorio antes de prod.** Toda ejecucion contra un tenant real
   debe correr `--dry-run` primero. La fase 0 lo enforza: el CLI rehusa correr
   sin `--dry-run` si el tenant target no esta en la allowlist `--skip-dry-run-for`.
7. **Resume por tabla, no por dominio.** Si falla en la tabla N del dominio, al
   `--resume` continua desde la tabla N — no reinicia el dominio entero. Tracking
   en `erp_migration_table_progress` con `(domain, table, last_legacy_id, status)`.
8. **Pre-validacion contra constraints Plan 18 (nueva fase 1).** Antes de insertar
   nada, se corren queries legacy contra los triggers/CHECKs/FKs de Plan 17+18
   y se reportan filas que romperian. El operador decide fix-at-source o skip.

## Arquitectura

```
tools/cli/
  cmd/
    migrate_legacy.go      ← Comando: sda migrate-legacy
  internal/
    legacy/
      connection.go        ← MySQL connection pool
      reader.go            ← Generic MySQL reader
      catalogs.go          ← Leer lookups legacy
      entities.go          ← Leer personal/clientes/proveedores
      stock.go             ← Leer articulos/depositos/movimientos
      accounting.go        ← Leer plan de cuentas/asientos
      treasury.go          ← Leer bancos/movimientos/cheques
      purchasing.go        ← Leer OC/recepciones
      sales.go             ← Leer cotizaciones/pedidos
      invoicing.go         ← Leer facturas/libro IVA/retenciones
      production.go        ← Leer OPs/centros
      hr.go                ← Leer empleados/licencias
    migration/
      orchestrator.go      ← Orden de ejecucion + resume
      mapper.go            ← ID mapping (INT → UUID)
      transformer.go       ← Data transformation rules
      validator.go         ← Post-migration checks
      writer.go            ← Batch writer a PostgreSQL
```

### Flujo general

```
1. Conectar a MySQL (read-only) y PostgreSQL (write)
2. Por cada dominio (en orden de FK):
   a. Leer registros de MySQL
   b. Transformar (tipos, formatos, normalizacion)
   c. Generar UUID, guardar mapping INT→UUID
   d. Resolver FKs usando mappings de dominios anteriores
   e. Batch INSERT en PostgreSQL
   f. Validar counts
3. Al final: validacion integral
4. Generar reporte
```

### Resume per-table

```go
// orchestrator.go
func (o *Orchestrator) Resume(ctx context.Context, runID string) error {
    // Cargar run previo
    run, err := o.repo.GetMigrationRun(ctx, runID)
    if err != nil { return err }

    // Listar tablas pendientes/fallidas del run (NO saltar dominios enteros)
    pending, err := o.repo.ListPendingTables(ctx, runID)
    if err != nil { return err }

    slog.Info("resuming migration", "run_id", runID, "pending_tables", len(pending))

    for _, t := range pending {
        if err := o.runTable(ctx, run, t); err != nil {
            return fmt.Errorf("table %s failed: %w", t.LegacyTable, err)
        }
    }
    return o.repo.MarkRunCompleted(ctx, runID)
}

// runTable procesa una tabla legacy, continuando desde last_legacy_key.
// Cada batch exitoso actualiza last_legacy_key atomicamente.
// El reader de cada tabla conoce el formato de su key — el orchestrator
// lo trata como opaco TEXT.
func (o *Orchestrator) runTable(ctx context.Context, run *MigrationRun, t *TableProgress) error {
    o.repo.MarkTableInProgress(ctx, t.ID)

    reader := o.legacy.GetReader(t.LegacyTable)
    for {
        rows, nextKey, err := reader.ReadBatch(ctx, t.LastLegacyKey, 500)
        if err != nil { return err }
        if len(rows) == 0 { break }

        written, skipped, err := o.writer.WriteBatch(ctx, rows, run.ID)
        if err != nil {
            o.repo.MarkTableFailed(ctx, t.ID, err.Error())
            return err
        }

        t.LastLegacyKey = nextKey  // reader retorna la key del ultimo row del batch
        t.RowsWritten += written
        t.RowsSkipped += skipped
        o.repo.UpdateTableProgress(ctx, t)
    }
    return o.repo.MarkTableCompleted(ctx, t.ID)
}

// Interfaz del Reader — cada tabla legacy implementa esta signature.
type LegacyReader interface {
    // ReadBatch lee hasta N filas con key > resumeKey.
    // resumeKey="" significa empezar desde el principio.
    // Retorna las filas y la key del ultimo row (para el proximo ReadBatch).
    ReadBatch(ctx context.Context, resumeKey string, limit int) ([]LegacyRow, string, error)
}

// Ejemplo: reader con PK simple (BIGINT)
func (r *JournalEntryReader) ReadBatch(ctx context.Context, resumeKey string, limit int) ([]LegacyRow, string, error) {
    lastID := int64(0)
    if resumeKey != "" { lastID, _ = strconv.ParseInt(resumeKey, 10, 64) }
    rows, _ := r.db.Query(ctx, "SELECT ... FROM CTB_MOVIMIENTOS WHERE IDMOVIMIENTO > ? ORDER BY IDMOVIMIENTO LIMIT ?", lastID, limit)
    var out []LegacyRow
    var last int64
    for rows.Next() {
        var row LegacyRow
        rows.Scan(&row.IDMOVIMIENTO, ...)
        out = append(out, row)
        last = row.IDMOVIMIENTO
    }
    return out, strconv.FormatInt(last, 10), nil
}

// Ejemplo: reader con PK compuesta
func (r *TaxEntryReader) ReadBatch(ctx context.Context, resumeKey string, limit int) ([]LegacyRow, string, error) {
    // Parse resumeKey "FACTURA_ID=X,TIPO_IVA=Y"
    lastFactura, lastTipo := parseCompositeKey(resumeKey)
    rows, _ := r.db.Query(ctx, `
        SELECT ... FROM IVACOMPRAS
        WHERE (FACTURA_ID, TIPO_IVA) > (?, ?)
        ORDER BY FACTURA_ID, TIPO_IVA LIMIT ?`, lastFactura, lastTipo, limit)
    // ... parse rows, compute last composite key
    return out, fmt.Sprintf("FACTURA_ID=%d,TIPO_IVA=%s", lastFacturaOut, lastTipoOut), nil
}
```

### ID Mapper

```go
type Mapper struct {
    repo     *repository.Queries
    cache    map[string]map[int64]uuid.UUID  // domain+table → legacy_id → uuid
    tenantID string
}

// Map retorna el UUID para un legacy ID. Si no existe, genera uno nuevo.
func (m *Mapper) Map(ctx context.Context, tx pgx.Tx, domain, table string, legacyID int64) (uuid.UUID, error) {
    key := domain + ":" + table
    if cached, ok := m.cache[key][legacyID]; ok {
        return cached, nil
    }

    // Check DB
    sda, err := m.repo.GetLegacyMapping(ctx, tx, m.tenantID, domain, table, legacyID)
    if err == nil {
        m.cache[key][legacyID] = sda
        return sda, nil
    }

    // Generate new
    newID := uuid.New()
    m.repo.CreateLegacyMapping(ctx, tx, m.tenantID, domain, table, legacyID, newID)
    m.cache[key][legacyID] = newID
    return newID, nil
}

// Resolve busca un mapping existente (para FKs). Error si no existe.
func (m *Mapper) Resolve(ctx context.Context, domain, table string, legacyID int64) (uuid.UUID, error)
```

### Batch writer

```go
type BatchWriter struct {
    pool      *pgxpool.Pool
    batchSize int  // default 500
}

// WriteBatch inserts rows using INSERT ... ON CONFLICT DO NOTHING for idempotency.
// CopyFrom is faster but fails on duplicate keys on re-run. Since idempotency is
// a hard requirement (--resume must work), we use batch INSERT instead.
// For ~500k total records this adds ~30% time vs CopyFrom but guarantees safety.
func (w *BatchWriter) WriteBatch(ctx context.Context, table string, columns []string, conflictColumn string, rows [][]any) (int, error) {
    if len(rows) == 0 {
        return 0, nil
    }
    // Build: INSERT INTO table (col1, col2) VALUES ($1,$2), ($3,$4) ... ON CONFLICT (id) DO NOTHING
    var sb strings.Builder
    sb.WriteString(fmt.Sprintf("INSERT INTO %s (%s) VALUES ", table, strings.Join(columns, ",")))
    args := make([]any, 0, len(rows)*len(columns))
    for i, row := range rows {
        if i > 0 { sb.WriteString(",") }
        sb.WriteString("(")
        for j, val := range row {
            if j > 0 { sb.WriteString(",") }
            args = append(args, val)
            sb.WriteString(fmt.Sprintf("$%d", len(args)))
        }
        sb.WriteString(")")
    }
    sb.WriteString(fmt.Sprintf(" ON CONFLICT (%s) DO NOTHING", conflictColumn))

    tag, err := w.pool.Exec(ctx, sb.String(), args...)
    return int(tag.RowsAffected()), err
}
```

## Fase 1 — Pre-validation contra constraints Plan 17+18

**Objetivo:** Antes de insertar nada, detectar filas legacy que romperian los
constraints de SDA (triggers de Plan 18, CHECKs, FKs). Reportarlas para que
el operador decida: fix manual en legacy, skip, o transformacion en el mapper.

> **Por que esta fase existe:** Legacy Histrix puede tener estados que los triggers
> de Plan 18 rechazan (ej: `'anulado'` en vez de `'cancelled'`, `entry_type` no-enum,
> fechas fuera de rango, referencias rotas). Insertar sin pre-validar genera errores
> mid-migration y deja estado inconsistente. Esta fase corre **en dry-run exclusivamente**
> y reporta a `erp_migration_validation_issues`.

### Validaciones a ejecutar

Cada validacion es una query MySQL que detecta filas incompatibles:

### Legacy → SDA status mapping (tabla de verdad)

Este mapping se aplica en `transformer.go` cuando `Transform = "auto_fix"`:

| Dominio | Legacy value | SDA value |
|---|---|---|
| invoices.status | `anulada` | `cancelled` |
| invoices.status | `pagada` | `paid` |
| invoices.status | `pendiente` | `posted` |
| invoices.status | `borrador` | `draft` |
| invoices.status | `confirmada` | `posted` |
| journal_entries.entry_type | `MA` (manual) | `manual` |
| journal_entries.entry_type | `AU` (automatico) | `auto` |
| journal_entries.entry_type | `AJ` (ajuste) | `adjustment` |
| journal_entries.entry_type | `RV` (reversa) | `reversal` |
| treasury_movements.status | `confirmado` | `confirmed` |
| treasury_movements.status | `anulado` | `cancelled` |
| fechas `0000-00-00` | NULL |

**Regla:** Si aparece un valor legacy no listado aqui, el transformer tira error
fatal — prefiere fallar que insertar data corrupta. El operador agrega el mapping
y re-corre.

### Service layer

```go
func (o *Orchestrator) RunPreValidation(ctx context.Context, run *MigrationRun) (*ValidationReport, error) {
    report := &ValidationReport{RunID: run.ID}
    for _, rule := range validationRules {
        rows, err := o.legacy.ExecQuery(ctx, rule.Query)
        if err != nil { return nil, err }
        for _, r := range rows {
            o.repo.CreateValidationIssue(ctx, &ValidationIssue{
                RunID: run.ID,
                Domain: rule.Domain,
                LegacyTable: rule.LegacyTable,
                LegacyID: r.LegacyID,
                ConstraintName: rule.Constraint,
                Details: r.ToJSON(),
                Resolution: rule.Transform,  // default
            })
            report.AddIssue(rule.Name, r)
        }
    }
    // Si hay issues 'pending' o 'fix_manual' sin resolver, el run se marca como
    // dry_run_failed y el prod run se rehusa a correr hasta que se resuelvan.
    return report, nil
}
```

```go
        {"clientes", "SELECT COUNT(*) FROM REG_CUENTA WHERE TIPO=1", "erp_entities WHERE type='customer'"},
        {"proveedores", "SELECT COUNT(*) FROM REG_CUENTA WHERE TIPO=2", "erp_entities WHERE type='supplier'"},
        {"empleados", "SELECT COUNT(*) FROM PERSONAL", "erp_entities WHERE type='employee'"},
    }
    for _, c := range checks {
        mysqlCount := queryCount(mysql, c.MySQL)
        pgCount := queryCount(pg, "SELECT COUNT(*) FROM "+c.PGTable+" AND tenant_id=$1", tenant)
        if mysqlCount != pgCount {
            return fmt.Errorf("%s: mysql=%d pg=%d", c.Name, mysqlCount, pgCount)
        }
    }
    return nil
}
```

```go
        Number:      fmt.Sprintf("%04d-%08d", row.Int("PTOVENTA"), row.Int64("NUMERO")),
        Date:        safeDate(row.Time("FECHA")),
        InvoiceType: invoiceType,
        Direction:   "issued",
```

```go
// Stock levels: Histrix calcula stock actual con query, no con tabla.
// Migramos recalculando desde movimientos para garantizar consistencia.
func migrateStockLevels(ctx context.Context, pg *pgxpool.Pool, tenant string) error {
    // En vez de copiar STK_STOCKACTUAL (que puede estar desincronizado),
    // recalculamos desde los movimientos ya migrados
    _, err := pg.Exec(ctx, `
        INSERT INTO erp_stock_levels (tenant_id, article_id, warehouse_id, quantity)
        SELECT tenant_id, article_id, warehouse_id,
            SUM(CASE WHEN movement_type = 'in' THEN quantity ELSE -quantity END)
        FROM erp_stock_movements
        WHERE tenant_id = $1
        GROUP BY tenant_id, article_id, warehouse_id
        ON CONFLICT (tenant_id, article_id, warehouse_id)
        DO UPDATE SET quantity = EXCLUDED.quantity, updated_at = now()
    `, tenant)
    return err
}
```

### Transformacion de `erp_fiscal_years.result_account_id`

Legacy `CTB_EJERCICIOS` **no tiene el concepto de "cuenta de resultado"** — Histrix
lo preguntaba al usuario en el momento del cierre. La migracion replica este
comportamiento:

```go
func transformFiscalYear(row mysql.Row) (*FiscalYear, *LegacyMeta, error) {
    return &FiscalYear{
        Year:           row.Int("ANOFISCAL"),
        StartDate:      row.Time("FECHA_INICIO"),
        EndDate:        row.Time("FECHA_FIN"),
        Status:         mapFiscalStatus(row.String("ESTADO")),  // 'abierto'→'open', 'cerrado'→'closed'
        ResultAccountID: nil,  // NULL — se setea post-migration via PATCH
    }, meta, nil
}
```

Los fiscal years ya cerrados en legacy se migran con `status='closed'` pero
**no se pueden volver a cerrar** (el trigger de Plan 18 bloquea `closed → closed`).
El operador solo necesita setear `result_account_id` en los ejercicios con
`status='open'` — es decir, el actual + eventuales futuros.

## Post-plan: que queda reutilizable

- **tools/cli/internal/migration/** — framework de migracion reutilizable.
  Si hay otro sistema legacy en el futuro, el orchestrator + mapper + writer
  se reusan.
- **erp_legacy_mapping** — tabla permanente que permite trazar cualquier
  registro de SDA a su origen en Histrix. Util para debugging post-migracion.

## Consideraciones

- **Tiempo de ejecucion:** Con ~500k registros totales en MySQL, la migracion
  deberia completar en <45 minutos (batch INSERT ON CONFLICT, ~30% mas lento
  que CopyFrom pero garantiza idempotencia).
- **Encoding:** MySQL de Histrix usa latin1. Convertir a UTF-8 en el reader
  configurando `?charset=utf8mb4&parseTime=true` en el DSN del driver
  `go-sql-driver/mysql` (hace la conversion transparente).
- **Fechas:** MySQL tiene fechas invalidas (`0000-00-00`). `safeDate()` en el
  transformer las mapea a NULL. Las FKs a `fiscal_year` que dependen de una
  fecha valida se validan en pre-validation (fase 1).
- **IDs de usuario:** Los `user_id` en tablas de negocio se setean como
  `"legacy-import"` (constante). La trazabilidad del usuario Histrix original
  vive en `erp_legacy_mapping.legacy_created_by` — hacer JOIN para obtenerlo.
  No se agregan columnas a las tablas de negocio.
- **Dry-run obligatorio:** `--dry-run` corre fases 0-1 completas (incluyendo
  pre-validation) + transformaciones de fases 2-5 sin hacer INSERTs. Al terminar
  marca el run como `dry_run_ok` o `dry_run_failed`. El run prod rehusa correr
  sin un `dry_run_ok` previo para el mismo tenant.
- **Resume per-table:** Si la migracion falla, `--resume --resume-run-id=<uuid>`
  continua desde el `last_legacy_id` de la tabla que fallo. No reinicia el
  dominio entero. Ver `orchestrator.Resume()` en fase 0.
- **Rollback:** One-way. Si algo sale catastroficamente mal post-migration,
  el procedimiento es: `DROP` de las tablas de negocio del tenant + re-run.
  No hay rollback automatico porque agregarlo duplicaria la complejidad del
  orchestrator sin beneficio claro (branch 2.0.x pre-prod).
- **Rate limiting Mysql:** Los reads de MySQL legacy estan en el mismo pool
  que Histrix read-only. Si Histrix sigue sirviendo trafico, limitar el read
  batch a 100-200 rows/s para no impactar. Configurable via `--mysql-rps=N`.
