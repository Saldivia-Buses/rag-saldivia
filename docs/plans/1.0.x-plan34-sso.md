## Contexto

SSO es blocker para enterprise. Hoy el auth es JWT custom con email/password
(login route + proxy middleware + Redis revocation). Los campos `ssoProvider` y
`ssoSubject` ya existen en la tabla `users` pero no estan conectados a nada.

El deploy model es single-tenant: cada empresa tiene su servidor. Esto simplifica
SSO porque no hay multi-IdP por deployment -- cada empresa configura su IdP.

### Decision de libreria: `arctic` (lucia-auth)

**Evaluacion:**

| Criterio | arctic | next-auth (Auth.js v5) | Manual (raw fetch) |
|---|---|---|---|
| Peso | ~30KB, zero deps | ~200KB, muchas deps | 0 deps |
| Lock-in | Ninguno (funciones puras) | Alto (middleware, callbacks, adapters) | Ninguno |
| Conflicto con auth existente | Ninguno -- solo genera URLs y parsea tokens | Alto -- quiere manejar sesiones, JWT, DB | N/A |
| Providers OIDC | 50+ built-in | 40+ built-in | Manual por provider |
| SAML | No | Parcial (enterprise plan) | Requiere `@node-saml/node-saml` |
| Curva de aprendizaje | Baja -- funciones puras | Alta -- framework dentro del framework | Media |

**Decision: `arctic` para OIDC + `@node-saml/node-saml` para SAML.**

Razones:
1. `arctic` son funciones puras que generan authorization URLs y parsean callbacks.
   No conflictua con nuestro JWT existente -- despues del callback, seguimos usando
   `createAccessToken()` + `createRefreshToken()` exactamente igual.
2. `next-auth` quiere ser el auth system completo. Integrarlo implicaria reescribir
   proxy.ts, jwt.ts, safe-action.ts, y todos los tests de auth. No vale la pena.
3. SAML se posterga a Fase 4 porque es infrecuente en el target market inicial
   (Saldivia Buses). Si se necesita, `@node-saml/node-saml` es la unica opcion
   viable en Node.js.

### Flujo SSO

```
1. Admin configura SSO provider en /admin/sso (client_id, client_secret, tenant)
   -> Guardado en tabla `sso_providers` (cifrado con SYSTEM_API_KEY)
2. Usuario visita /login -> ve botones SSO segun providers configurados
3. Click "Iniciar con Google" -> GET /api/auth/sso/[provider]
   -> arctic genera authorization URL con state + PKCE
   -> state guardado en cookie HttpOnly (5 min TTL)
   -> redirect a IdP
4. IdP autentica -> redirect a /api/auth/callback/[provider]
   -> arctic parsea code + verifica state
   -> arctic intercambia code por tokens (access + id_token)
   -> Extrae claims: email, name, sub
   -> Busca usuario por (ssoProvider, ssoSubject)
   -> Si no existe + auto-provisioning ON -> crea con rol default
   -> Si no existe + auto-provisioning OFF -> error "Contacta al admin"
   -> Si existe + email match -> actualiza lastLogin
   -> createAccessToken() + createRefreshToken() -> Set-Cookie -> redirect /chat
```

### Seguridad

- **CSRF:** state parameter en cookie HttpOnly + verificacion en callback
- **PKCE:** code_verifier en cookie HttpOnly (standard para OAuth 2.0 public clients)
- **Account linking:** por email match. Si el email SSO ya existe como usuario local,
  se actualiza `ssoProvider`/`ssoSubject` en el primer login SSO (one-time link).
  Si ya esta linkeado a otro provider -> error.
- **Token storage:** NO almacenar refresh tokens del IdP. Solo necesitamos el
  id_token para extraer claims en el callback. El usuario se autentica con
  nuestro JWT propio, no con tokens del IdP.
- **Secrets:** `client_secret` cifrado en DB con AES-256-GCM usando `SYSTEM_API_KEY`

## Riesgos y mitigacion

| Riesgo | Probabilidad | Mitigacion |
|---|---|---|
| arctic no soporta un provider especifico | Baja | Tiene 50+ providers; fallback a `oidc_generic` |
| IdP tokens tienen claims no standard | Media | `extractUserInfo()` normaliza por provider |
| Account linking crea conflictos | Media | Locking: una vez linkeado, no se puede re-linkear a otro provider |
| PKCE no soportado por IdP legacy | Baja | arctic maneja esto per-provider |
| Secret rotation en DB | Baja | Re-encrypt action en admin (futuro) |
| Provider desactivado post-login | Baja | JWTs existentes siguen validos (max 15min). Refresh funciona normal (JWT no tiene info de provider). El usuario solo no puede re-autenticar via SSO. |
| IdP elimina cuenta del usuario | Baja | JWT sigue valido hasta expirar. Proximo refresh funciona. Solo el re-login SSO falla. El admin puede desactivar el usuario manualmente. |
