## Fases

### Fase 1: Security — FTS5 MATCH sanitization

**Archivo:** `packages/db/src/queries/search.ts`

**Problema:** FTS5 MATCH acepta operadores especiales (`*`, `"`, `OR`, `NOT`, `AND`, `NEAR`)
que un usuario podría inyectar. Drizzle parametriza el SQL pero FTS5 interpreta el contenido
del parámetro como query syntax.

**Cambio:** Sanitizar input antes de pasarlo a MATCH — escapar comillas dobles,
strip de operadores booleanos.

**Verificación:**
- [ ] `bun test packages/db/src/__tests__/search.test.ts` → green
- [ ] `bun run test` → green
- [ ] `bun run lint` → clean

**Commit:** `fix(db): sanitize FTS5 MATCH input against special characters — plan28 f1`

### Fase 2: Security — cifrado de external_sources credentials

**Archivo:** `packages/db/src/queries/external-sources.ts`

**Problema:** `external_sources.credentials` almacena JSON en texto plano.
El schema ya tiene el comentario "cifrar con SYSTEM_API_KEY en prod" pero no está implementado.

**Cambios:**
- Helper `encrypt(plaintext)` / `decrypt(ciphertext)` con AES-256-GCM
- Key derivada de `SYSTEM_API_KEY` (generada con `openssl rand -base64 32` = 32 bytes)
- Validar formato de key al iniciar — si no cumple, log warning y NO cifrar (no romper en dev)
- Cifrar al escribir (`createExternalSource`, `updateExternalSource`)
- Descifrar al leer (`getExternalSource`)
- **Migración de datos existentes:** Si hay credentials en texto plano (no base64),
  detectar y cifrar en la primera lectura (lazy migration)

**Verificación:**
- [ ] `bun test packages/db/src/__tests__/external-sources.test.ts` → green
- [ ] `bun run test` → green
- [ ] `bun run lint` → clean

**Commit:** `fix(db): encrypt external_sources credentials with AES-256-GCM — plan28 f2`

### Fase 6: Error UX — toast + feedback system

**NOTA: Esta fase es una feature nueva, no refactoring.**

**Estado actual:** Los componentes admin ya manejan errores con `setError()` inline
(banner visible en la página). Esto funciona pero no hay forma de que el usuario
envíe el error al equipo con contexto.

**Archivos nuevos:**
- `apps/web/src/lib/error-feedback.ts` — `showErrorFeedback(error, context)` abre dialog
- `apps/web/src/components/ui/error-feedback.tsx` — dialog con:
  - Error amigable (no stack trace)
  - Campo de texto: "¿Qué estabas haciendo?"
  - Botón "Enviar reporte" → POST /api/feedback
- `apps/web/src/app/api/feedback/route.ts` — POST endpoint protegido con auth.
  Guarda en `audit_log` con `action="error_feedback"`, `queryPreview` = JSON con
  error + comentario del usuario. Campos existentes de audit_log son suficientes.

**Archivos modificados:**
- Verificar `sonner.tsx` — usa clases shadcn (`bg-background`, `text-foreground`)
  que pueden no coincidir con los tokens custom del proyecto (`bg-bg`, `text-fg`).
  Ajustar si hay discrepancia visual.
- En componentes admin donde tenga sentido, agregar botón "Reportar" en el catch
  existente (complementa el `setError()` inline, no lo reemplaza).

**Patrón:**
```typescript
} catch (err) {
  setError("Error al crear rol")
  // Opcionalmente, el toast con botón de reporte
  toast.error("Error al crear rol", {
    action: { label: "Reportar", onClick: () => showErrorFeedback(err, "Crear rol") },
  })
}
```

**Verificación:**
- [ ] `bunx tsc --noEmit` → 0 errors
- [ ] `bun run test` → green
- [ ] `bun run lint` → clean
- [ ] Manual: provocar error → toast → click "Reportar" → dialog → enviar

**Commit:** `feat(web): add error feedback system with toast + report dialog — plan28 f6`
