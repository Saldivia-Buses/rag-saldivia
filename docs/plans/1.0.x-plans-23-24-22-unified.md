# Sprint Unificado — Plans 23 → 24 → 22

> **Branch:** 1.0.x
> **Objetivo:** Llevar Saldivia RAG de "prototipo funcional" a "producto solido
> con admin completo y messaging interno" en 3 planes secuenciales.

---

## Narrativa

El sistema tiene un problema clasico: la mitad del codigo esta construido pero
desconectado. La DB tiene 25+ tablas con queries funcionales. El archive tiene
componentes de admin listos. Pero solo 3 de 7 secciones admin estan conectadas.

Los 3 planes forman un pipeline que completa todo:

```
Plan 23: Code Hardening          Plan 24: Admin + Collections    Plan 22: Messaging
━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━━━      ━━━━━━━━━━━━━━━━━━
Arregla seguridad (XSS)   →     Fix bugs colecciones      →    Schema messaging
React Compiler             →     Admin nav 3→7 tabs        →    WebSocket sidecar :3001
Async bcrypt               →     Areas CRUD + miembros     →    Canales + DMs
Extract hooks              →     Permission matrix          →    Threading + reacciones
next-safe-action           →     Collection management      →    @mentions + pins
ConfirmDialog              →     RAG Config (del archive)   →    FTS5 search
Aria-live + a11y           →     Kill "tecpia" para siempre →    Command palette
+50 tests                  →     Collection selector chat   →    Notifications
                                 +30 tests                  →    +20 tests
+ CHANGELOG                →     + CHANGELOG                →    + CHANGELOG + CLAUDE.md
```

---

## Por que este orden

1. **Plan 23 primero:** Arregla XSS real, crea `authAction`/`adminAction`,
   extrae hooks (`useLocalStorage`), crea `ConfirmDialog`. Todo lo que los
   planes 24 y 22 necesitan como foundation.

2. **Plan 24 segundo:** Completa el admin (de 3 a 7 secciones), arregla
   colecciones, conecta users→areas→collections→permissions. Messaging
   necesita que este flujo funcione para poder referenciar colecciones
   y permisos en canales.

3. **Plan 22 ultimo:** El mas grande. Se beneficia de: React Compiler
   (componentes auto-optimizados), next-safe-action (messaging actions
   type-safe desde el dia 1), hooks reutilizables, colecciones funcionando,
   permisos correctos.

## Flujo integrado post-sprint

```
Admin crea Area "Operaciones"
  → Asigna colecciones "manuales-buses", "normativas" con permiso read
  → Asigna usuario "Papa Saldivia" al area (NUEVO — P24 F3)
  → Papa Saldivia entra, ve solo sus 2 colecciones
  → Selecciona ambas en el chat (CollectionSelector), hace multi-collection query
  → Admin ajusta RAG config (temperatura, top_k) desde /admin/config
  → Papa Saldivia abre /messaging, ve canales #general y #random
  → Manda mensaje, el tio lo recibe instantaneamente via WebSocket
  → @menciona a Enzo, Enzo recibe notificacion via WS
  → Busca un mensaje viejo con Cmd+K (FTS5 search)
```
