# Plan 2.0.5 — Modular Docs Rewrite

> **Date:** 2026-04-15
> **Branch:** `2.0.5-docs-modular` (off `2.0.5`)
> **Status:** Approved, ready to execute
> **Driver:** Current docs are monolithic (`bible.md`, `CRITICAL_FLOWS.md`, `2.0.x-plan01`), hard to maintain, drift silently. Audience is AI — needs small, focused, indexable files.

---

## Goal

Replace monolithic docs with one-aspect-per-file structure. Each file is ≤200 lines, English, AI-optimized, kept in sync automatically by a `doc-sync` agent.

## Non-goals

- Human onboarding guides (not the audience — those are in service READMEs).
- Migrating `docs/plans/*` — plans stay as-is (they're historical records).
- Renaming code or reorganizing the repo itself.

## Doc format (enforced by invariants)

```markdown
---
title: JWT Authentication
audience: ai
last_reviewed: 2026-04-15
related:
  - ../services/auth.md
  - ../flows/login-jwt.md
  - ../packages/jwt.md
---

One paragraph: what this is and when to read it.

## Sections...
```

**Rules:**
1. Frontmatter fields: `title`, `audience: ai`, `last_reviewed: YYYY-MM-DD`, `related: [relative links]`.
2. Max 200 lines including frontmatter and blank lines.
3. No redundancy — each fact lives in one doc, others link.
4. Code references: `path/to/file.go:42` (path + line), not embedded blocks of source.
5. Use relative links between docs.
6. First paragraph after frontmatter: "what + when to read".

## `doc-sync` agent

**Purpose**: keep modular docs in sync with code automatically.

### Files created

| Path | Role |
|---|---|
| `.claude/agents/doc-sync.md` | Agent definition (prompt, tools, allowlist) |
| `.claude/hooks/doc-sync.sh` | Pre-commit hook dispatching the agent |
| `.claude/schedules/doc-drift-daily.md` | Daily scheduled drift audit (via `schedule` skill) |

### Pre-commit behavior

1. `git diff --cached --name-only` — what's staged
2. For each staged file, resolve target docs:
   - `services/X/...` → `docs/services/X.md` + possibly `docs/architecture/*.md` if endpoint/NATS-event changed
   - `pkg/Y/...` → `docs/packages/Y.md`
   - `db/*/migrations/*.sql` → `docs/conventions/migrations.md`
   - `.claude/agents/Z.md` → `docs/ai/agents.md`
3. Dispatch `doc-sync` subagent with the queue + diffs
4. Agent reads only target docs + diffs, updates stale sections, bumps `last_reviewed`
5. Stages doc changes, commit continues
6. If any target doc exceeds 200 lines after edit → blocks commit with error

### Daily audit

- Cron-triggered via `schedule` skill at 06:00 local
- Scans every doc against `last_reviewed`, related code mtime, broken links
- Writes report to `docs/.drift-report.md` (gitignored)
- Does **not** auto-commit — human reviews and triggers update

### Guardrails

- Agent tool allowlist: `Read`, `Edit`, `Glob`, `Grep` (no `Write`, no `Bash` except `git diff`)
- Only edits files under `docs/` — never touches code
- Cannot create new docs (structure changes are human decisions)
- Skipped if `git commit --no-verify`

## Risks

| Risk | Mitigation |
|---|---|
| 200-line cap forces too-terse docs | Use it as a forcing function — if a topic needs more, split the topic, not the doc |
| Service/pkg docs duplicate info | Services describe endpoints + behavior, packages describe API. Cross-link, don't repeat |
| `doc-sync` agent loops or bloats cost | Pre-commit hook has timeout (30s), agent has read-only tool allowlist except for `Edit` in `docs/` |
| Migrating 81 docs is a lot of work | Delegate F3b and F3c to `doc-writer` subagents, one service/pkg per dispatch |
| Plan takes multiple sessions | Each phase self-contained, commit checkpoints after each |
| Stale plan docs still reference `bible.md` | Acceptable — plans are historical records, not live docs |
