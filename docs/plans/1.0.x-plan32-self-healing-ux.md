# Plan 32 — Self-Healing Error UX

> **Branch:** 1.0.x
> **Prerequisito:** Plan 28 (Error UX + toast sistematico completado)
> **Sprint:** think -> plan -> execute -> review -> test -> ship
> **Intensity:** standard

## Contexto

Plan 28 establecio error handling sistematico: toast con sonner, ErrorBoundary
global, ErrorFeedbackMount para reportes, y errores tipados en `lib/rag/client.ts`
(`UNAVAILABLE`, `TIMEOUT`, `FORBIDDEN`, `UPSTREAM_ERROR`). Tambien existe
`packages/logger/src/suggestions.ts` con `getSuggestion()` que mapea errores
conocidos a sugerencias tecnicas.

El problema: las sugerencias actuales estan pensadas para devs ("Verifica con:
make status"), no para usuarios finales. Y el frontend muestra errores genericos
("Algo salio mal") sin contexto ni acciones de recuperacion.

Este plan transforma los errores en experiencias de recuperacion: mensajes en
espanol, acciones sugeridas con botones clickeables, y contexto dinamico (ej:
cuanto falta para el proximo slot de rate limit, que colecciones tienen cache).

### Por que ahora

Los testers (Enzo, su papa, su tio) van a encontrar errores reales. En vez de
una pantalla rota que los deja perdidos, necesitan mensajes que expliquen que
paso y que pueden hacer.

### Fase 5: Rate limit context en API + enriquecer generate errors

**Archivos:**
- `apps/web/src/app/api/rag/generate/route.ts` (modificar)

**Cambios:**

Enriquecer la respuesta de rate limit para incluir datos de contexto que el frontend pueda usar para mostrar "Proximo slot en X minutos" (buscar el bloque de rate limit con `grep -n "rate\|429" apps/web/src/app/api/rag/generate/route.ts`):

Antes:
```typescript
return apiError(`Limite de ${maxQph} queries/hora alcanzado. Intenta mas tarde.`, 429)
```

Despues:
```typescript
const oldestQueryTs = await getOldestQueryInWindow(userId)
const retryAfterMs = oldestQueryTs ? (oldestQueryTs + 60 * 60 * 1000) - Date.now() : 60 * 60 * 1000
return apiError(
  `Limite de ${maxQph} queries/hora alcanzado.`,
  429,
  { code: "RATE_LIMITED", retryAfterMs, currentCount: count, maxCount: maxQph }
)
```

Esto requiere agregar `getOldestQueryInWindow` en `packages/db` — **pero** para evitar scope creep, podemos estimar el retry con `60 * 60 * 1000 - (Date.now() - oneHourAgo)` directamente en la ruta sin nueva query de DB. La estimacion es: "tu query mas vieja en la ventana de 1 hora va a expirar en ~60 min menos lo que ya paso".

Simplificacion (sin nueva query DB):
```typescript
const retryAfterMs = Math.max(0, 60 * 60 * 1000 - 0) // worst case: 60 min
// El frontend mostrara "~60 minutos" — es una estimacion conservadora
```

Mejor: usar `countQueriesLastHour` que ya existe — si el count == max, el peor caso es 60 min. El frontend puede decir "hasta ~60 minutos".

Tambien enriquecer el error de `ragGenerateStream` para pasar el `error.code` en details:

```typescript
return apiError(result.error.message, status, {
  code: result.error.code,
  suggestion: result.error.suggestion,
})
```

(Esto ya se hace parcialmente — la linea actual ya pasa `{ suggestion }`. Solo agregar `code`.)

## Decisiones de diseno

1. **`getUserSuggestion` separado de `getSuggestion`** — el primero es para logs del server (en ingles tecnico para devs), el segundo es para la UI (en espanol para usuarios). No mezclar audiencias.

2. **`getErrorRecovery` en `lib/error-recovery.ts`**, no en `packages/logger`** — la logica de recovery incluye acciones de UI (rutas de navegacion, callbacks). Eso es dominio del frontend, no del logger.

3. **No polling de health check** — monitorear servicios en background (Redis, Milvus, RAG) agrega complejidad significativa. Para v1, el error se detecta cuando el usuario hace una accion. Si un servicio se cae entre acciones, el proximo request lo va a detectar. El polling se puede agregar en un plan futuro.

4. **Variantes `inline` y `page`** — el mismo componente se usa en dos contextos: dentro del chat (compacto, sin borde, se descarta al reintentar) y como pagina completa (centrado, con borde, boton reset de Next.js). Una sola abstraccion, dos presentaciones.

5. **Rate limit sin query extra de DB** — estimacion conservadora ("hasta 60 minutos") en vez de agregar `getOldestQueryInWindow` a packages/db. YAGNI — la estimacion es suficiente para el usuario.

6. **No modificar `apiError` ni `api-utils.ts`** — `apiError` ya acepta `details` como tercer argumento. Solo pasamos mas datos en `details`.
