# Plan 16 --- Backend Polish (version 2.0.5)

> **Prerequisito:** Plan 14 (deploy hardening) y Plan 15 (bigbrother) completados
> **Fecha:** 2026-04-09
> **Estado:** Draft --- pendiente aprobacion
> **Version target:** 2.0.5
> **Sprint:** think > plan > execute > review > test > ship
> **Servicios afectados:** TODOS (12 Go services + 1 Python service + infra)
> **Estimacion:** 6 fases, ~8-10 dias

## Contexto

Una auditoria exhaustiva del codebase encontro 25 findings (6 critical, 4 high,
6 medium, 9 low). Los criticals son production blockers --- Docker builds que
fallan, servicios que no pueden conectarse a NATS, env vars faltantes que
deshabilitan funcionalidad esencial.

Este plan es CERO features nuevas. Es polish puro: arreglar todo, conectar
todo, testear todo, maximizar reutilizacion de codigo. Al terminar este plan,
cada servicio debe compilar en Docker, conectarse a todas sus dependencias,
tener tests, y compartir el maximo codigo posible.

## Principio rector

**Si no compila en Docker y no pasa tests, no existe.** Cada fase termina con
un servicio que funciona end-to-end en el entorno target.

### Fase 3: pkg/server --- Eliminar Boilerplate de main.go

**Objetivo:** Extraer ~50 lineas de boilerplate repetido en 12 main.go a un
package compartido. Cada main.go queda en ~30-50 lineas de config especifica.

**Archivos nuevos:**
- `pkg/server/server.go`
- `pkg/server/server_test.go`

**Archivos modificados:**
- Todos los 12 `services/*/cmd/main.go`

**Diseno de pkg/server:**

El boilerplate comun encontrado en TODOS los main.go:

1. Logger init (slog.NewJSONHandler, slog.SetDefault) --- 2 lineas x12
2. Signal context (signal.NotifyContext) --- 3 lineas x12
3. OTel setup (sdaotel.Setup con ServiceName/ServiceVersion) --- 8 lineas x12
4. Blacklist init (security.InitBlacklist) --- 1 linea x12
5. Chi router base (chi.NewRouter + 5 middleware) --- 6 lineas x12
6. Health check registration (hc.Handler on "/health") --- 2 lineas x12
7. Build info endpoint ("/v1/info") --- 1 linea x12
8. HTTP server config (timeouts) --- 8 lineas x12
9. Graceful shutdown (srv.Shutdown) --- 8 lineas x12

**API target (minimal — no service locator):**

```go
package main

func main() {
    app := server.New("sda-auth", server.WithPort("AUTH_PORT", "8001"))

    // app provides: Context(), Logger(), Router() (pre-wired with base middleware)
    ctx := app.Context() // signal-aware context
    r := app.Router()    // chi.NewRouter + RequestID + RealIP + Recoverer + SecureHeaders + Timeout

    // Service-specific setup — NOT absorbed into pkg/server
    pool, err := database.NewPool(ctx, config.Env("POSTGRES_PLATFORM_URL", ""))
    nc, err := natspub.Connect(config.Env("NATS_URL", nats.DefaultURL))
    blacklist := security.InitBlacklist(ctx, config.Env("REDIS_URL", ""))

    // Register routes
    r.Get("/health", hc.Handler())
    r.Get("/v1/info", build.Handler("sda-auth"))
    r.Post("/v1/auth/login", authHandler.Login)

    // Blocks until SIGINT/SIGTERM, then graceful shutdown
    app.Run()
}
```

**Regla importante:** pkg/server solo elimina las ~30 lineas identicas
(logger, signal context, OTel, chi base middleware, HTTP server, graceful
shutdown). NO absorbe DB, NATS, blacklist, ni gRPC — esos quedan en cada
main.go. Esto evita el service locator anti-pattern y permite que servicios
complejos (auth con dual JWT, astro con CGO, chat/search con gRPC) usen
el mismo package sin friction.

Servicios que NO encajan en un pattern mas agresivo:
- auth: dual JWT keys, dual DB mode
- astro: CGO init, 4 middleware groups, LLM + intelligence engine
- bigbrother: scanner loop, encryption, cleanup goroutine
- chat/search: dual server (HTTP + gRPC)

**M2 --- ServiceVersion hardcodeada:**

Como parte de este refactor, pkg/server debe leer la version del VERSION file
del servicio (via pkg/build), no de un string hardcodeado. Eliminar todas las
constantes `serviceVersion` y strings `"1.0.0"` de los main.go.

pkg/build.Version ya se puede inyectar via ldflags. Para dev, implementar
fallback que lee `VERSION` file del servicio si ldflags no set. Alternativa
mas simple: el Makefile ya deberia pasar `-ldflags` con el contenido del
VERSION file.

Actualizar `pkg/build/info.go` para agregar un `ReadVersionFile(path)` como
fallback cuando `Version == "dev"`.

**NOTA:** El Makefile actual NO pasa ldflags. Los `go build` son bare:
`cd $(SERVICES_DIR)/$$svc && go build -o $(GOBIN)/$$svc ./cmd/...`.
Esta fase DEBE actualizar tanto el Makefile como los Dockerfiles para pasar
`-ldflags "-X pkg/build.Version=$(cat VERSION)"`. Sin esto, el fallback
ReadVersionFile es necesario para dev, y ldflags para Docker builds.

**Orden de migracion:**

Migrar servicios de menor a mayor complejidad:
1. traces (mas simple, solo DB + NATS consumer)
2. notification (DB + NATS consumer + SMTP)
3. platform (DB + NATS publisher)
4. ingest (DB + NATS bidireccional + worker)
5. feedback (dual DB + NATS consumer + aggregator)
6. chat (DB + NATS + gRPC server)
7. search (DB + gRPC server + LLM)
8. ws (NATS + gRPC client, no DB)
9. agent (NATS + LLM + tools, no DB)
10. auth (dual mode platform/tenant, mas complejo)
11. astro (CGO, ephemeris, intelligence engine)
12. bigbrother (scanner, encryption, cleanup)

Cada servicio se migra individualmente, se compila, y se testea antes de pasar
al siguiente. Si un servicio tiene particularidades que no encajan en
pkg/server, se mantiene su setup custom y se documenta por que.

**Verificacion:**
- [ ] `make build` --- todos los 12 servicios compilan
- [ ] `make test` --- todos los tests green
- [ ] `make lint` clean
- [ ] Cada main.go tiene <= 60 lineas (excluyendo comentarios)
- [ ] Ningun main.go tiene string hardcodeado de version
