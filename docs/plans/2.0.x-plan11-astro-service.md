## Contexto

`astro-v2` es un motor de astrología predictiva escrito en Python (FastAPI + pyswisseph).
43k LOC, 215 archivos .py, con 19 duplicaciones de `_ang_diff` y 28 de `_pos_to_str`.
Funciona bien pero es un monolito Python standalone fuera de SDA.

Este plan lo reescribe como `services/astro` — un microservicio Go que sigue los patrones
SDA (chi, sqlc, NATS, pkg/*) con cálculos determinísticos via Swiss Ephemeris (cgo) y
verificación de exactitud via golden files contra la salida Python.

### Por qué ahora

1. El motor de astrología es un módulo de negocio — debe vivir dentro de SDA, no afuera
2. Los agents de SDA necesitan acceso a las herramientas astrológicas via tool manifest
3. Contacts deben estar en PostgreSQL con tenant isolation, no en un JSON suelto
4. La deuda técnica del Python (19x duplicaciones) se resuelve gratis en el port

### Notas técnicas

- **CGO:** Este servicio requiere `CGO_ENABLED=1` (swephgo wraps Swiss Ephemeris C).
  Desviación del patrón estándar (`CGO_ENABLED=0` + distroless). Dockerfile usa `alpine` + `gcc musl-dev`.

- **swephgo:** `github.com/mshafiee/swephgo` v1.1.0, GPL-3.0, effectively unmaintained
  (last release Apr 2023, 0 issues closed). GPL aceptable (comercialización target: 2028).
  La librería NO embebe fuentes C — requiere `libswe.a` pre-compilada.
  Para compilar localmente: extraer fuentes C de pyswisseph sdist y compilar con cc/ar.

- **API quirks de swephgo:** `SetEphePath([]byte)` no `string`; `Julday` gregflag es `int32`;
  `Revjul` output via slices; `SetTopo` modifica estado global (requiere session mutex);
  eclipse constants: `SeEclTotal=4`, `SeEclAnnular=8`, `SeEclPartial=16` (NO 1/2/4).

## Fases pendientes — resumen

### Fase 6: Primary Directions (Polich-Page)
- Source: `primary_directions.py` ~600 lines
- **Most complex technique** — Polich-Page mundane directions with oblique ascension
- Golden file: 57 activations para Adrian 2026
- Incluye direcciones directas y conversas, aspectos mayores

### Fase 7: Secondary Progressions
- Source: `secondary_progressions.py`
- Day-for-year: `progressed_jd = natal_jd + age_days`
- Ingress detection: sign/house changes between consecutive years

### Fase 8: Solar/Lunar Return
- Uses `SolcrossUT` (already implemented in Phase 2)
- Solar Return: find JD when transiting Sun = natal Sun longitude
- Lunar Return: iterate monthly for Moon crossing

### Fase 10: Eclipses + ZR + Fixed Stars
- Eclipses: `SolEclipseWhenGlob` / `LunEclipseWhen` iteration
- ZR: from Fortune AND Spirit, with Egyptian bounds sub-periods
- Fixed Stars: 14 stars × ~15 natal points = 210 conjunction checks

### Fase 11: Context Builder + Intelligence Brief
- Orchestrates all techniques, produces structured text for LLM
- Convergence matrix: monthly scores based on technique overlap

### Fase 12-15: Repository, Handler, Integration, Manifest
- sqlc contacts CRUD, SSE streaming, Traefik routing, tools.yaml

## Notas de ejecución

- **Ephemeris data files** must be available at `EPHE_PATH` env var (or Moshier fallback)
- **CGO_ENABLED=1** is required for ALL `go test` and `go build` commands
- **libswe.a** must be compiled from Swiss Ephemeris C sources and linked via CGO_LDFLAGS
- **Port:** 8011 (next sequential after search=8010)
- **Dependency order:** ephemeris → astromath → natal → techniques → context → handler
- **CalcMu:** BuildNatal holds it for entire duration. CalcPlanetFullLocked for nested calls.
- **Golden files:** Generated from Python with Moshier fallback. Go tests must use same mode.
