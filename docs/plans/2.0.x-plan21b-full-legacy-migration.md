# Plan 21B — Full Legacy Migration: Histrix MySQL → SDA PostgreSQL

> **Prerequisito:** Plan 21 completado (28 migrators existentes funcionando).
> **Fecha:** 2026-04-13
> **Estado:** Draft v2 — corregido post-review (7C, 7H, 6M findings resueltos)
> **Migration prerequisito:** `054_migration_schema_fixes.up.sql`
> **Version target:** 2.0.5+
> **Sprint:** think > plan > execute > review > test > ship (por fase)
> **Servicios afectados:** `tools/cli/`, `services/erp/`, `services/auth/`
> **Direccion:** One-way (MySQL → PostgreSQL). Sin rollback automatico.

## Contexto

Plan 21 migro 28 tablas (~3.5M filas) cubriendo catalogos, entidades, contabilidad
core, stock basico, compras basicas, cotizaciones, produccion basica, y RRHH basico.

Este plan migra **TODO lo demas**: 62 migrators nuevos cubriendo ~12.4M filas
adicionales (facturacion, tesoreria completa, cuentas corrientes, BOM, calidad,
mantenimiento, seguridad laboral, auth, y datos historicos). Al completarse,
el 89% de los 17.1M de filas legacy estaran en SDA. El 11% restante son
logs de sistema, tablas deprecated, y config UI legacy — descartados justificadamente.

| Categoria | Tablas | Filas | Accion |

### Decisiones clave (aprobadas 2026-04-13)

1. **Todo el historial** — sin cutoff date, cada fila importa
2. **saldivia + saldivia_sa = 1 tenant** — misma empresa, pre/post conversion SA (~2025)
3. **Password reset OK** — usuarios Histrix se crean en SDA con password reset forzado
4. **JSONB metadata** — tablas secundarias van como metadata en registro padre SDA
5. **Prioridad: mas critico primero** — finanzas → stock → ventas → calidad → RRHH → mant → seguridad

## Principios (heredados de Plan 21)

1. **MySQL es read-only.** Solo SELECT.
2. **Migrar por dominio, en orden de FK.** Respeta dependencias.
3. **IDs cambian.** INT → UUID via erp_legacy_mapping.
4. **Idempotente.** INSERT ... ON CONFLICT DO NOTHING.
5. **Dry-run obligatorio.** Antes de cada prod run.
6. **Resume per-table.** Checkpoint por batch (5000 rows).
7. **JSONB metadata debe ser funcional.** Estructura con keys significativas,
   queryable via `->` operators. GIN index donde se consulte frecuentemente.

### 10.1 Pedidos internos (PEDIDOINT → erp_purchase_orders)

**Legacy:** PEDIDOINT (384,345 rows)

PEDIDOINT son requisiciones internas de materiales. Se migran a `erp_purchase_orders`
con `metadata: {"source": "internal_requisition"}` para distinguirlos de las OC
externas (CPS_MOVIMIENTOS).

> **C5 fix:** `supplier_id` ahora es nullable (migration 054). Requisiciones internas
> sin proveedor (37,474 filas = 9.7%) se insertan con `supplier_id = NULL`.

> **H2 fix:** Cardinalidad verificada en MySQL: 245,237 nrocom distintos en 384,345 filas
> = promedio 1.57 lineas por pedido. Cada fila es una linea de pedido.
> Se agrupan por `nrocom` para generar header + lines. El header toma fecha/estado
> del primer renglon del grupo. Si hay colision de nrocom con distinta fecha,
> se desambigua con sufijo `-{fechaemi}`.

```
PEDIDOINT (agrupado por nrocom) →  erp_purchase_orders (header)
  nrocom                       →  number (prefijo "RI-{nrocom}")
  MIN(fechaemi)                →  date
  codprv (del primer renglon)  →  supplier_id (FK entity, NULLABLE)
  MAX(estado)                  →  status (0=draft, 1=approved, 2=received)

PEDIDOINT (cada fila)          →  erp_purchase_order_lines
  artcod                       →  article_id (FK via hashCode)
  cant                         →  quantity
  pencps                       →  unit_price
```

### 11.5 Inspecciones de produccion

**Legacy:** PROD_CONTROL_MOVIM (1,747,678) + PROD_CONTROL_HOMOLOG (454,804) +
PROD_CONTROLES (1,999) + PROD_CONTROL_RETRABAJO (19,417)

**Estrategia (H4 fix — 2-hop FK resolution):**

1. PROD_CONTROLES (1,999 headers) → erp_production_inspections con order_id directo.
2. PROD_CONTROL_MOVIM (1.7M) tiene `prodcontrol_id` FK a PROD_CONTROLES.
   Se resuelve: PROD_CONTROL_MOVIM → PROD_CONTROLES.orden_id → erp_production_orders.
   El migrator hace JOIN en MySQL: `SELECT m.*, c.orden_id FROM PROD_CONTROL_MOVIM m
   JOIN PROD_CONTROLES c ON m.prodcontrol_id = c.id`.
3. PROD_CONTROL_HOMOLOG (454K) sigue mismo patron.

**Result mapping:** PROD_CONTROL_MOVIM tiene campos `conforme`, `no_conforme`, `no_aplica`:
```
conforme=1    → result='pass'
no_conforme=1 → result='fail'
no_aplica=1   → result='pass' (metadata: {"not_applicable": true})
ninguno=1     → result='rework' (default)
```

### saldivia_sa (M1 fix)

Solo existe la database `saldivia` en el Docker actual (verificado: `saldivia_sa`
no tiene tablas). Los datos post-conversion SA ya estan en `saldivia` gracias a
los triggers de sync. **No se necesita DSN separado ni merge/dedup.**

Si en el futuro se descubre data exclusiva de `saldivia_sa`, se agrega un segundo
pass con `--mysql-dsn` apuntando a esa DB. La idempotencia (ON CONFLICT DO NOTHING)
garantiza que los duplicados no se inserten dos veces.

### Encoding

MySQL legacy tiene mix latin1/utf8. El driver MySQL con `charset=utf8mb4` ya esta
configurado en Plan 21. Los transformers manejan caracteres invalidos.

### JSONB metadata — consideraciones

**Indices recomendados (en migration 054 o posterior):**

```sql
-- Para buscar articulos por atributo
CREATE INDEX idx_erp_articles_metadata ON erp_articles USING GIN (metadata jsonb_path_ops);

-- Para buscar vehiculos por numero de chasis en historia
CREATE INDEX idx_erp_units_metadata ON erp_units USING GIN (metadata jsonb_path_ops);
```

**M2 — Tamaño JSONB en erp_units:** Con 3,993 vehiculos recibiendo ~75K child records,
algunos vehiculos podrian tener JSON >500KB. Mitigacion: para `accessories[]` (37K filas)
y `drawing_folders[]` (10K filas), guardar solo los campos esenciales (id, nombre, fecha),
no todas las columnas legacy. Estimar ~200 bytes por accessorio = ~2KB promedio por vehiculo.

**M5 — hashCode collision:** Con FNV-1a sobre ~25K article codes, la probabilidad de
colision es ~0.007%. Post-validacion verifica: `COUNT(DISTINCT artcod) en MySQL ==
COUNT(DISTINCT code) en erp_articles`. Si hay mismatch, se reporta y se resuelve manual.

### Triggers de inmutabilidad (M4 fix)

La migracion usa `SET LOCAL session_replication_role = 'replica'` que desactiva
ALL triggers (incluidos los de inmutabilidad de account_movements, treasury_movements,
invoices). Esto es correcto porque estamos insertando datos historicos, no modificando
datos existentes. Al finalizar la migracion se restaura el role.

Fase 17 (metadata UPDATE) tambien corre con triggers desactivados. Esto es seguro
porque solo modifica la columna `metadata` que no es protegida por los triggers.

### Rollback

Si algo sale mal: DELETE FROM erp_* WHERE tenant_id = $1 y re-correr.
La migracion es idempotente — no se duplican registros.
