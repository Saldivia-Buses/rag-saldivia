# Plan 08 — Backend Hardening: de critico a bajo

> **Prerequisito:** Plan 07 implementado y mergeado
> **Sprint:** hardening (seguridad, performance, reliability, consistency)
> **Fecha:** 2026-04-05
> **Estado:** borrador v2 (post-review gateway + security audit)
> **Reviews:** `docs/artifacts/plan08-backend-hardening-review.md`,
> `docs/artifacts/plan08-backend-hardening-security-audit.md`
> **Trigger:** auditoria completa del backend/DB/infra revelo hallazgos
> que separan el sistema de "bien arquitectado" a "production-ready"

---

## Principio

> El sistema tiene fundamentos excelentes. Las decisiones grandes son correctas.
> Lo que falta es cerrar la brecha entre lo que el spec promete y lo que el
> codigo hace. Este plan no agrega features — cierra gaps.

**Regla:** cada fase se ataca en orden de severidad. Ningun hallazgo queda
sin resolver. Al terminar, el sistema esta listo para usuarios reales.

**Scope:** solo hardening. gRPC (protos existentes, 1182 lineas, cero uso)
se implementa en Plan 09 por separado — es un feature, no un fix.

**Workflow:** fase → branch `feat/plan08-phase-N` → PR a `2.0.1` → review → merge

**Por que:** hallazgos criticos que bloquean produccion. Sin estos,

`pkg/audit` existe y 4 servicios ya lo usan (auth: 7 Write calls, chat: 3,
search: queries, notification: ReadAll). Pero 5 servicios no tienen
audit logging: **ingest, platform, agent, feedback, traces**.

**Que hacer:**

  - platform: create/update tenant, enable/disable module, update config

  - traces: (consumer-only, audit si se borran traces)

**Verificacion:** audit_log tiene entries de los 5 servicios nuevos

### C2. Fix Traefik configs — routes desactualizados

Traefik configs (dev + prod) referencian servicio `rag` en puerto 8004 que
ya no existe (deprecated, reemplazado por `agent`). Faltan routes para

**Que hacer:**

**Verificacion:** `curl localhost/v1/agent/health`, `/v1/search/health`, `/v1/traces/health`

**Verificacion:** `make sqlc` genera sin error. `git diff` muestra cero cambios en generated code

### C4. Habilitar SSL en conexiones PostgreSQL

`sslmode=disable` en todas las connection strings, dev y prod.

**Que hacer:**

- En `pkg/tenant/resolver.go`, usar `url.Parse` para appendear sslmode de forma segura:

**Verificacion:** servicio auth conecta a PG con SSL en prod compose

## Fase 2: High — Security Depth

**Por que:** hallazgos de severidad alta que son fixes puntuales de seguridad.
Todos son pequenos (30min-1h cada uno) y se agrupan en un solo PR.

### H-NEW. Rate limiting aplicativo

Traefik prod ya tiene rate limit global (100 req/s per IP, burst 200). Lo que
falta es rate limiting **granular a nivel aplicacion**: por usuario, por
endpoint sensible, por cuenta.

**Que hacer:**

```
pkg/middleware/ratelimit.go
```

- In-memory para MVP single-node. Documentar migration path a Redis para scale
- Cuatro niveles:
  - **Auth login:** 5/min por IP en `/v1/auth/login`
  - **Auth refresh:** 10/min por IP en `/v1/auth/refresh` (previene token flooding)
  - **MFA verify:** 5/min por IP en `/v1/auth/mfa/verify` (TOTP tiene solo 1M posibilidades)
  - **AI:** 30/min por user en `/v1/agent/query` y `/v1/search/query`
- Retornar `429 Too Many Requests` con header `Retry-After`
- Override per-tenant via config resolver (para power users)

**Riesgo:** rate limit de AI puede bloquear power users. Mitigacion: override
configurable per-tenant/user via `pkg/config` resolver.

**Verificacion:** test que envia 10 requests rapidos a login y recibe 429 en el 6to

### H-NEW. ReadHeaderTimeout en todos los servicios

Ninguno de los 10 servicios Go tiene `ReadHeaderTimeout` configurado.
Sin esto, un client puede enviar headers byte a byte indefinidamente (slowloris).

**Que hacer:**

Agregar a cada `&http.Server{}`:
```go
ReadHeaderTimeout: 10 * time.Second,
```

### H8. Ownership checks en queries

`GetSession` (chat.sql) filtra solo por `id` sin `user_id`. El service layer
compensa, pero defense-in-depth requiere filtrar en la query. Mismo patron
en `TouchSession` y `GetJob` (ingest.sql).

**Que hacer:**
- `GetSession`: agregar `AND user_id = $2`
- `TouchSession`: agregar `AND user_id = $2`
- `GetJob` (ingest): agregar `AND user_id = $2`
- Documentar que documents son shared-within-tenant (design decision, no bug)

### H11. JWT JTI — defense in depth

El service layer ya genera JTI en los 7 paths de token creation. Agregar
dos capas de proteccion adicionales:

1. En `pkg/jwt/jwt.go`: si `claims.ID` esta vacio, generar con `uuid.NewString()`
2. En `pkg/middleware/auth.go`: rechazar tokens sin JTI:
```go
if cfg.Blacklist != nil && claims.ID == "" {
    writeJSONError(w, http.StatusUnauthorized, "invalid token")
    return
}
```

### H-NEW. Blacklist fail-open configurable

`pkg/middleware/auth.go:71-74` fail-open cuando Redis cae: tokens revocados
vuelven a ser validos. Hacer configurable:

```go
type AuthConfig struct {
    Blacklist *security.TokenBlacklist
    FailOpen  bool // true = allow on Redis error, false = reject
}
```

- Auth service: `FailOpen: false` (seguridad > disponibilidad)
- Otros servicios: `FailOpen: true` (disponibilidad > seguridad)

### M9. Security scans bloquean CI

En `.github/workflows/ci.yml`:
- Cambiar gosec de `|| true` a exit real
- Cambiar Trivy de `exit-code: '0'` a `exit-code: '1'`

### M11. Agent WriteTimeout + todos los servicios

- Agent: cambiar `WriteTimeout: 0` a `WriteTimeout: 5 * time.Minute`
- Todos los servicios: verificar que tienen `ReadHeaderTimeout: 10 * time.Second`
  (cubierto por H-NEW ReadHeaderTimeout)

### M14. API key no se cachea en Redis

En `pkg/config/resolver.go`, crear `CachedModelConfig` sin `APIKey`.
Cachear solo el struct reducido. APIKey se lee de DB per-request al LLM.
No encriptar (si Redis se compromete, el atacante probablemente tiene la
encryption key tambien — esta en el mismo host).

### L4. writeJSONError — usar json.Marshal

En `pkg/middleware/auth.go:122-126`, reemplazar concatenacion de strings
por `json.NewEncoder(w).Encode(map[string]string{"error": msg})`.
Previene JSON injection si algun caller futuro pasa datos no-hardcoded.

**Verificacion:** tests existentes pasan. Test nuevo: JWT sin JTI es rechazado.
Test: 429 en login despues de 5 intentos. gosec/trivy fallan CI si hay findings

## Fase 3: High — Database & Query Hardening

**Por que:** queries sin pagination, subqueries O(N), sin batch inserts,
sin cleanup. Problemas que no se notan con 3 usuarios pero explotan con 100.

**Riesgo:** pagination rompe frontends que asumen lista completa. Coordinar
con frontend: agregar params `?page=1&page_size=50` y manejar responses
paginadas. Agregar header `X-Total-Count` para que el frontend sepa el total.

### H4. Pagination universal

Crear helper en pkg:

```go
// pkg/pagination/pagination.go
type Params struct {
    Page     int // default 1
    PageSize int // default 50, max 100
}
func (p Params) Offset() int { return (p.Page - 1) * p.PageSize }
func (p Params) Limit() int  { return p.PageSize }
func Parse(r *http.Request) Params // lee ?page=&page_size= con defaults y max cap
```

Agregar pagination a:
- `ListActiveUsers` (auth.sql)
- `ListSessionsByUser` (chat.sql)
- `ListMessages` (chat.sql) — usar cursor-based (`WHERE created_at > $cursor`)
  en vez de OFFSET para este caso (mensajes pueden ser miles, OFFSET es O(N))
- `GetAllDocumentTrees` (documents.sql)
- `GetCollectionDocumentTrees` (documents.sql)
- `ListTenants` (tenants.sql)
- `ListNotifications` — cap limit a max 100 (actualmente sin upper bound)

### H9. ListActiveUsers — eliminar correlated subquery

Reescribir con `LATERAL JOIN`:

```sql
SELECT u.id, u.email, u.name, u.created_at,
       COALESCE(r.name, 'user') AS role
FROM users u
LEFT JOIN LATERAL (
    SELECT r.name FROM roles r
    JOIN user_roles ur ON ur.role_id = r.id
    WHERE ur.user_id = u.id
    ORDER BY CASE r.name WHEN 'admin' THEN 1 WHEN 'manager' THEN 2 WHEN 'user' THEN 3 ELSE 4 END
    LIMIT 1
) r ON true
WHERE u.is_active = true
ORDER BY u.created_at ASC
LIMIT $1 OFFSET $2;
```

### M10. Batch insert para document pages

Reemplazar `InsertDocumentPage` (1 row/call) con `pgx.CopyFrom`.
PDF 200 paginas: 200 round-trips → 1 round-trip.

### L5. DeleteJobByID — agregar ownership check

En `services/ingest/db/queries/ingest.sql`, agregar `AND user_id = $2`.
Actualmente solo se usa internamente para cleanup, pero defense-in-depth.

### L6. Notification cleanup/purge

Agregar query `PurgeOldNotifications`:
```sql
DELETE FROM notifications WHERE is_read = true AND created_at < now() - interval '90 days';
```

### L7. FK documents.uploaded_by → users(id)

```sql
ALTER TABLE documents ADD CONSTRAINT fk_documents_uploaded_by
    FOREIGN KEY (uploaded_by) REFERENCES users(id);
```

### L8. PerformancePercentiles — generated column

```sql
ALTER TABLE feedback_events ADD COLUMN latency_ms NUMERIC
    GENERATED ALWAYS AS ((context->>'latency_ms')::numeric) STORED;
CREATE INDEX idx_feedback_events_latency ON feedback_events(latency_ms)
    WHERE category = 'usage';
```

### L9. CountHistoricalUsage — agregar filtro de tiempo

Agregar parametro `since TIMESTAMPTZ` al query.

### L10. Categorias como constantes

Crear constantes Go en `pkg/feedback/categories.go` y referenciar
en queries para evitar strings hardcoded repetidos en 4+ queries.

### L11. Eliminar indice redundante

```sql
DROP INDEX IF EXISTS idx_refresh_tokens_hash;
```

**Verificacion:** `make test` pasa. Queries con pagination retornan subsets correctos.
Bulk insert de 100 pages es >5x mas rapido. Frontend notificado de cambios de API

## Fase 7: Infrastructure — Backups, CrowdSec, Hardening

**Por que:** sin backups, sin IDS, sin CPU limits, sin TLS en otel.
El ultimo paso antes de declarar el sistema production-ready.

### L15. Backup strategy

**PostgreSQL:**
- Script `deploy/scripts/backup.sh` que:
  - Lista todos los tenants de Platform DB
  - `pg_dump` cada tenant + Platform DB
  - Comprime con gzip
  - **Encripta con `age` o `gpg`** antes de upload (dumps contienen password
    hashes, MFA secrets, connection URLs)
  - Sube a MinIO bucket `sda-backups/` con usuario **write-only** (proteccion
    contra ransomware — no puede borrar backups existentes)
  - Retiene: 30 dias diario, 12 meses mensual
  - Genera SHA-256 checksums junto a cada backup
- Cron job en la workstation: diario a las 3 AM
- Agregar restore script `deploy/scripts/restore.sh`:
  - Verifica checksum antes de restaurar
  - NO se deploya a containers prod — solo corre manual desde workstation

**Redis:**
- `BGSAVE` + copiar `.rdb` al mismo bucket (encrypted)
- Nota: blacklist tiene TTL, es inherentemente efimero. Worst case post-restore:
  tokens revocados son validos hasta expirar naturalmente (15 min max)

**NATS JetStream:**
- Incluir `/data` en el backup

**Verificacion:** backup → drop DB → restore → verificar datos intactos

### M12. CrowdSec — deployar

```
Fase 1 (critical) ←── bloquea todo
  ├── Fase 2 (parallel OK)
  └── Fase 3 (parallel OK, solo depende de C3)
Fases 2 + 3 completadas
  └── Fase 4 (feature wiring + NATS)
Fase 4
  └── Fase 5 (testing — features wired, testeables)
Fase 5
  └── Fase 6 (build/CI — tests cubren codigo, CI los corre)
Todo completado
  └── Fase 7 (infra — cierre)
```

Fases 2 y 3 son completamente independientes y pueden ejecutarse en paralelo.

## Riesgos

| # | Riesgo | Mitigacion |
|---|---|---|
| R1 | Pagination rompe frontends que asumen lista completa | Coordinar con frontend antes de Fase 3. Header X-Total-Count |
| R2 | Rate limiting bloquea power users | Override per-tenant via config resolver |
| R3 | NATS per-service auth rompe dev workflow | Dev mantiene auth simple (documentado) |
| R4 | sqlc schema change genera structs diferentes | Verificar con `git diff` post-regeneracion |
| R5 | Security scans bloquean CI con false positives | Agregar `// nosec` annotations donde sea justified |
