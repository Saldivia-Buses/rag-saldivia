> **Estado:** implementado (fases 1-11, pendiente: tests detallados + business module)

> **Scope:** Port COMPLETO de astro-v2 a Go/SDA + preparar endpoints para frontend futuro

---

## Contexto

El Plan 11 porto las 11 tecnicas predictivas base de astro-v2 Python a Go. Pero astro-v2
es un sistema de **477 archivos con 80+ modulos Python** que incluye:

- **55+ tecnicas astrologicas** pre-calculadas deterministicamente
- **Intelligence Layer** — 47 dominios con routing, technique gating, cross-references
- **Business/Enterprise module** — dashboard ejecutivo con timing, riesgos, roles
- **Quality/Audit system** — validacion deterministica de respuestas
- **Session management** — chat con historial, memoria, prediccion tracking
- **System prompt experto** — 55 tecnicas simultaneas con narración por dominio

El Go service actual tiene ~20% de las capacidades. Este plan porta el 80% restante
(todo excepto frontend, que se conectara despues via endpoints preparados).

---

## Tenant model

**Single-tenant: Saldivia Buses.** Un proceso astro, una DB de tenant, un slug.
Se usa `POSTGRES_TENANT_URL` como ahora. Las columnas `tenant_id` en tablas son
defense-in-depth (preparacion para multi-tenant futuro, no se usa para routing).
No se necesita `pkg/tenant.Resolver`.

---

## Fuera de scope

- **Frontend** — no se porta el frontend de astro-v2, pero SI se preparan todos los
  endpoints, WebSocket y SSE para que el frontend de SDA (Next.js) lo consuma
- **Semantic Router (FastEmbed ONNX)** — el intent parser de Python usa embeddings;
  el Go port usa keyword+regex (suficiente para 90% de queries, LLM upgrade futuro)
- **gRPC** — no se agrega facade gRPC en este plan; si agent runtime lo necesita, plan futuro

## Decisiones de diseno (de los 3 agentes de arquitectura)

### Calculo
1. **`technique/` queda flat** — 40+ archivos esta bien en Go, sub-packages crearian import cycles
2. **Multi-chart en `technique/`** con tipos `ChartPair`/`ChartSet` (no package separado)
3. **Build() sigue explicito** — no registry pattern, no interfaces — phased por grupo
4. **No goroutines** para tecnicas — CalcMu serializa todo; parallelizar no ayuda
5. **Electional usa two-pass**: coarse (1 dia) → fine (1 hora) dentro de regiones prometedoras
6. **Rectification/Electional: lock-per-chart** — NO hold CalcMu durante todo el scan loop,
   solo durante cada BuildNatal individual. Semaforo dedicado: max 1 concurrent rectification/electional.
   Rate limit separado: 2/min para estos endpoints (vs 10/min para tecnicas normales)

### Intelligence Layer
1. **`Engine` struct** creado una vez en startup, stateless `Analyze()` calls
2. **Gate usa inspeccion de structs** (no regex como Python) — `len(ctx.Transits) > 0`
3. **4 algoritmos de cross-ref**: ruler, point, chain, temporal — set-intersection sobre datos tipados
4. **Domain registry como Go struct literals** (type-safe), YAML tags para migracion futura
5. **Intent parser keyword-only** — LLM upgrade futuro via `WithLLM()` option
6. **Brief inteligente** reemplaza brief "dump everything" solo cuando hay query — backward compatible
7. **Integration minima**: handler gana `intel *intelligence.Engine`, `Query()` llama `intel.Analyze()`

### API + Data
1. **Tablas prefijadas `astro_`** para evitar colision con chat service en misma tenant DB
2. **SSE para streaming, WS via NATS** para state changes — no duplicar stream por WS
3. **Cache Redis opcional** — degrada gracefully si Redis no esta.
   Keys: `astro:{type}:{contact_id}:{year}`. TTL: natal=24h, transits=1h, business=15min.
   Invalidacion: contact update borra todas las cache entries del contacto.
4. **Chart registry in-memory** LRU (500 charts ~1MB). Key: `{tenant_id}:{contact_id}` (tenant-aware)
5. **Handler split** en multiples archivos del mismo package (sessions.go, chat.go, business.go)
6. **Service layer** entre handler y repository para domain logic
7. **Permissions**: `astro.read` (GET + stateless technique POSTs), `astro.write` (session/prediction/feedback CRUD), `astro.business` (business module)
8. **Role filter** (ventas/operaciones/rrhh) es view filter, no auth gate
9. **Sessions separadas de chat service** — justificacion: astro sessions llevan contact_id,
   techniques[], predictions tracking, metadata JSONB que chat no tiene. Duplicacion intencional.
10. **NATS publishing** siempre via `natspub.Notify()` o `natspub.Broadcast()` — nunca raw `nc.Publish()`
11. **Chat handler**: `http.MaxBytesReader` obligatorio, max 50 mensajes de historial para LLM,
    token budget total que trunca context si brief + history + query excede ventana del modelo
12. **Audit logging** (`audit.Write()`) en TODAS las operaciones de escritura nuevas
    (session create/update/delete, prediction create/verify, feedback, followup create/update)

con SA/DP que estaran exactos esta semana. Max 10 contactos por scan, tecnicas limitadas
a SA + PD (no full Build). Publica via `natspub.Notify()` → WS Hub → frontend.
