# SDA Framework — Spec

> La verdad del sistema. Si lo leés, tiene que describir el sistema completo.
> Un agente que abre este documento entiende la arquitectura, los servicios,
> los módulos, la seguridad, y cómo todo encaja — sin leer nada más.

## Qué es esto

**SDA Framework** — plataforma SaaS multi-tenant que combina servicios de IA
con módulos de negocio para empresas de cualquier industria. No es "un sistema
de RAG". Es un sistema operativo empresarial donde la IA potencia cada módulo:
gestión documental, chat inteligente, agentes autónomos, procesamiento de
documentos, workflows de aprobación, y módulos verticales por industria.

La plataforma funciona tanto para una PyME de 5 usuarios como para una empresa
de 500. Los módulos se habilitan por tenant — cada empresa paga por lo que usa.
La IA no es un feature más: es la capa que hace que cada módulo sea 10x más
útil que la alternativa genérica.

**Modelo de negocio:** Backend, DB y servicios de IA corren inhouse en una
workstation con RTX PRO 6000 (96GB VRAM) y 256GB RAM. Frontends corren en la
nube. El costo marginal por query de IA es electricidad — no hay factura por
token a OpenAI. Esto permite ofrecer más IA por menos plata que cualquier
competidor cloud-based.

**Ventaja competitiva:** Soberanía de datos. Los datos del cliente nunca salen
de la infraestructura controlada. Sin dependencia de AWS/Azure/GCP. Atractivo
para empresas reguladas y contratos con gobierno.

**Performance target:** MacMaster-Carr level — la app se siente como software
local, no como una web. Zero loading spinners.

**Security target:** El sistema es una fortaleza. El usuario pasa por un login
profesional que es la única puerta de entrada. Nada del sistema es visible
sin autenticación. Capas de ciberseguridad en cada nivel.

```
CLOUD (compartido)
┌──────────────────┐
│  DB Backups (S3)  │  ← backups automáticos de todos los tenants
```

**Por qué este split:**
- **Frontend en la nube** — no necesita GPU, se beneficia del CDN global, bajo costo
- **Backend inhouse** — acceso directo a GPU (RAG), baja latencia a DB, control total
- **RAG compartido** — un solo servicio proxy al NVIDIA Blueprint, colecciones Milvus aisladas por tenant
- **Cloudflare Tunnel** — expone servicios sin abrir puertos, DDoS protection, soporta WebSocket
- **DB backups en cloud** — disaster recovery para todos los tenants + propia

## Microservicios

### Filosofía: plataforma extensible, no lista fija

El repo está diseñado para soportar **cualquier cantidad de microservicios**. Agregar un servicio nuevo es: crear un directorio en `services/`, seguir la estructura estándar, agregar al Docker Compose. `make new-service NAME=x` genera el scaffold completo.

Los servicios se dividen en dos categorías:

**Servicios core** (siempre activos, columna vertebral de la plataforma):

| Servicio | Qué hace |
|---|---|
| **Auth Gateway** | La puerta de entrada. Login profesional, JWT, refresh tokens, RBAC, MFA/2FA, brute force protection. Nada del sistema es accesible sin pasar por acá. |
| **WebSocket Hub** | Corazón del real-time. Mantiene conexiones, protocolo subscribe/mutation/event, rutea a servicios via gRPC, pushea eventos NATS al browser. |
| **Chat** | Sesiones, mensajes, historial, tags, favoritos, fork. Es la interfaz principal de interacción del usuario con el sistema. |
| **RAG** | Proxy al NVIDIA Blueprint. Streaming via WS. Colecciones Milvus por tenant. Compartido entre tenants. |
| **Notification** | In-app (bell icon), email (SMTP), WebSocket push. Consume eventos NATS de todos los demás servicios. |
| **Platform** | Control total del sistema para los dueños (Saldivia). Gestión de tenants, billing, feature flags, config global, modelos RAG, deploy, Grafana. Solo accesible por platform admins. |

**Servicios modulares** (activables por tenant via feature flags):

| Servicio | Qué hace |
|---|---|
| **Ingest** | Upload de documentos, pipeline de procesamiento via NATS, conectores (Google Drive, OneDrive, etc.). |
| **Document AI** | Extracción estructurada de PDFs/imágenes (facturas, contratos, planos). Docling + nv-ingest + VLM. |
| **Agent** | Agentes empresariales configurables por tenant. NeMo Agent Toolkit + tools per-tenant. |
| **Workflow** | Motor de automatización (n8n o Temporal + UI custom). Trigger → condición → acción. Sin código. |
| **Speech** | ASR (Whisper) + TTS (Piper/XTTS). Transcripción de reuniones, dictado, asistente de voz. |
| **Vision** | Detección de objetos (YOLO). Seguridad en obra, análisis de fotos, fleet dashcam. |
| **Optimization** | cuOpt para rutas de flota, scheduling, logística. GPU-acelerado. |
| **Translation** | LibreTranslate. Traducción offline multi-idioma para tenants internacionales. |
| **Forecast** | TimesFM/Chronos. Series temporales para predicción de demanda, costos, mantenimiento. |
| **Feedback** | Event consumer cross-service. Calidad de IA, error reporting, NPS, data flywheel. |
| *Futuros* | CRM, WhatsApp, email analysis, e-commerce, HR, invoicing. |

Los módulos se habilitan por tenant en la Platform DB. Cada empresa paga por lo que usa. Chat y RAG son core — siempre disponibles para todos los tenants.

### Dos tipos de admin — distintos poderes

| | Platform Admin (Saldivia) | Tenant Admin (jefe de empresa) |
|---|---|---|
| **Quiénes son** | Los dueños del sistema (Enzo, equipo Saldivia) | El admin de cada empresa que contrata |
| **Qué ven** | TODOS los tenants, todo el sistema | Solo su empresa, sus usuarios |
| **Qué controlan** | Tenants, billing, feature flags, deploys, modelos RAG, config global, Grafana, audit log cross-tenant | Usuarios de su empresa, roles, permisos, config RAG de su tenant, sus colecciones |
| **Acceso a** | Platform Service + todo lo demás | Auth + Chat + RAG + Notification + módulos habilitados |
| **Dónde operan** | Panel platform (`platform.sda.app`) + CLI + MCP | Panel admin dentro de la app de su tenant |
| **Seguridad** | MFA obligatorio, IP allowlist, acceso a Grafana | MFA configurable por el platform admin |

El Platform Service tiene su propia DB (la Platform DB). Los tenant admins operan dentro de la DB de su tenant.

### Agregar un servicio nuevo

```
make new-service NAME=billing
→ Crea services/billing/ con estructura estándar
→ Agrega al docker-compose
→ Registra en Platform DB como servicio disponible
→ Listo para implementar lógica de negocio
```

### Auth Gateway — login separado del sistema

El login no es "una página más" del sistema. Es una **barrera total**. El usuario llega a la URL y ve únicamente un login profesional, limpio, branded. No hay hints sobre qué hay detrás. No hay nav, no hay sidebar, no hay nada. Solo email, password, y el logo.

Una vez autenticado, recién ahí se carga el sistema completo. La separación es real:

```
Usuario → URL
  → Auth Gateway verifica sesión
    → NO tiene sesión → Login page (aislada, sin leaks del sistema)
    → SÍ tiene sesión → App shell + WebSocket + datos preloaded
```

El login page no comparte layout, componentes, ni assets con la app. Es una app mínima que solo hace una cosa: autenticar. Esto previene information leakage y da una imagen profesional.

### Comunicación entre servicios

```
Frontend ──REST/WS──► WebSocket Hub ──gRPC──► Auth / Chat / RAG Service
                                     
Ingest Service ──NATS──► Notification Service (ingesta completada)
Chat Service   ──NATS──► Notification Service (nuevo mensaje)
Auth Service   ──NATS──► Notification Service (evento de seguridad)
WebSocket Hub  ◄──NATS── todos los servicios (pushea eventos al browser)
```

**Principios:**
- REST para login, upload, health checks. WebSocket para todo lo demás (90% de la interactividad).
- gRPC para comunicación interna entre servicios. Nunca expuesto al frontend.
- NATS JetStream para eventos async con persistencia. Subjects namespaceados por tenant (`tenant.saldivia.chat.new_message`) para aislamiento.

### Tenant context — cómo viaja el tenant por todo el sistema

El tenant se identifica por **subdomain** (`saldivia.app.com`) y se propaga así:

```
Browser → saldivia.app.com
  → Cloudflare Tunnel
    → Traefik extrae "saldivia" del subdomain → agrega header X-Tenant-ID
      → WebSocket Hub / REST API lee X-Tenant-ID
        → Resuelve connection info del tenant desde Platform DB (cacheado en Redis)
          → Conecta al PostgreSQL y Redis del tenant correcto
            → Todas las queries van a la DB del tenant
```

El JWT también contiene el `tenant_slug` como claim. Doble verificación: el subdomain y el JWT deben coincidir. Si no coinciden → request rechazado.

### Fault tolerance — qué pasa cuando un servicio falla

**JWT validation es local.** Cada servicio verifica la firma JWT localmente (clave pública compartida en `pkg/jwt/`). No llama a Auth Service en cada request. Auth Service solo se necesita para login, refresh, y RBAC changes. Si Auth se cae, los usuarios ya logueados siguen funcionando.

**WebSocket Hub redundancy.** El Hub se puede escalar a múltiples instancias con sticky sessions (Traefik) y estado compartido via Redis pub/sub. Una instancia es suficiente para ~300 usuarios concurrentes.

**NATS JetStream garantiza entrega.** Si Notification Service se cae cuando Chat publica un evento, NATS persiste el mensaje. Cuando Notification vuelve, consume los mensajes pendientes. Ningún evento se pierde.

**Operaciones cross-service.** Enviar un mensaje toca Chat (guardar) + RAG (generar respuesta) + Notification (avisar). Estas operaciones son **eventualmente consistentes**: Chat guarda el mensaje sincrónicamente (gRPC), RAG streamea la respuesta via WebSocket, y Notification recibe un evento via NATS (async, puede fallar sin afectar al usuario). No necesitamos saga pattern — el mensaje se guardó, la respuesta llegó, la notificación es best-effort.

## NVIDIA RAG Blueprint — integración como microservicio

### Qué es el Blueprint

El [NVIDIA RAG Blueprint v2.5.0](https://github.com/NVIDIA-AI-Blueprints/rag) es un pipeline de retrieval-augmented generation preconfigurado que incluye:

- **RAG Server** (:8081) — orquestador LangChain que recibe queries, recupera chunks de Milvus, re-rankea, y genera respuestas via LLM
- **NV-Ingest** (:8082) — pipeline GPU-acelerado de extracción de documentos (texto, tablas, imágenes, charts)
- **Milvus** (:19530) — base de datos vectorial para embeddings
- **NIMs** — microservicios de inferencia para embedding, reranking, y LLM

### Endpoints que expone

| Endpoint | Método | Qué hace |
|---|---|---|
| `/v1/chat/completions` | POST (SSE) | Generación con streaming. Formato OpenAI-compatible. |
| `/v1/chat/completions` | POST | Generación sin streaming (sugerencias, reportes). |
| `/v1/collections` | GET | Listar colecciones Milvus. |
| `/v1/collections` | POST | Crear colección (`{ collection_name: string }`). |
| `/v1/collections/{name}` | DELETE | Eliminar colección. |
| `/v1/documents` | POST (multipart) | Upload de documentos al ingestor (:8082). |
| `/v1/documents/{name}` | GET | Descargar documento original. |
| `/health` | GET | Health check. |

### Request de generación

```json
{
  "messages": [{ "role": "user", "content": "¿Qué dice el contrato sobre plazos?" }],
  "stream": true,
  "use_knowledge_base": true,
  "collection_name": "contratos-saldivia",
  "temperature": 0.2,
  "top_p": 0.7,
  "max_tokens": 1024,
  "vdb_top_k": 10,
  "reranker_top_k": 5
}
```

### Response SSE (formato OpenAI-compatible)

```
data: {"choices":[{"delta":{"content":"El contrato"}}]}
data: {"choices":[{"delta":{"content":" establece"}}]}
data: {"choices":[{"delta":{"sources":[{"document_name":"contrato.pdf","content":"...","score":0.95}]}}]}
data: [DONE]
```

### Modelos en la workstation

| Componente | Modelo | Endpoint | VRAM |
|---|---|---|---|
| LLM | Nemotron-Super-49B v1.5 | NVIDIA API (externo) | 0 (API) |
| Embeddings | llama-nemotron-embed-1b-v2 | `nemotron-embedding-ms:8000` | ~4GB |
| Reranker | llama-nemotron-rerank-1b-v2 | `nemotron-ranking-ms:8000` | ~2GB |
| VLM (ingesta) | Qwen3-VL-8B | `qwen3-vl-8b:8000` | ~16GB |

**Modo switching** (1 GPU): QUERY mode (~46GB: embed+rerank) vs INGEST mode (~90GB: embed+rerank+VLM). Un mode manager automático cambia según la operación.

### Cómo encaja en SDA

El Blueprint corre como un stack Docker separado. El RAG Service de Go es un **proxy delgado** que:

1. Recibe requests del WebSocket Hub (via gRPC)
2. Inyecta tenant context (colección correcta, permisos, system prompt)
3. Proxea al Blueprint (:8081) con las colecciones Milvus del tenant
4. Transforma el SSE a protocolo WebSocket y lo pushea al browser
5. Registra la query en audit log del tenant

**Aislamiento por tenant:** Cada tenant tiene sus propias colecciones Milvus (namespaceo: `{tenant_slug}-{collection_name}`). El RAG Service verifica permisos antes de proxear. Un tenant nunca puede acceder a las colecciones de otro.

**Config per-tenant:** Temperature, top_k, reranker, system prompt, y guardrails son configurables por tenant admin en la Platform DB.

## Sistema de agentes empresariales

### Por qué agentes, no solo chat

El chat con RAG responde preguntas. Los agentes **hacen cosas**: procesan facturas automáticamente, monitorean vencimientos, clasifican emails, generan reportes, y ejecutan workflows multi-paso. Un agente convierte al sistema de "herramienta de consulta" a "empleado digital".

### Stack de agentes

**NeMo Agent Toolkit** (v1.5, Apache 2.0) es el framework elegido:

- **Config-driven**: agentes definidos en YAML (tools, LLM, embedder, guardrails)
- **MCP-compatible**: puede actuar como MCP server (exponer agentes) y MCP client (consumir tools externos)
- **Multi-RAG**: un agente puede consultar múltiples colecciones RAG como tools
- **Evaluadores**: accuracy, relevance, groundedness — medición automática de calidad
- **Framework-agnostic**: funciona con LangChain, LlamaIndex, CrewAI, Semantic Kernel
- **Deploy**: FastAPI microservice o CLI

**NemoClaw** (NVIDIA, anunciado GTC 2026-03-16) es prometedor pero **alpha/single-player**. NO apto para multi-tenant hoy. Watchlist para cuando madure. Conceptos valiosos a adoptar:

- **Privacy Router**: queries sensibles → modelo local, no-sensibles → cloud. Clasificación automática de sensibilidad de datos
- **OpenShell sandbox**: aislamiento kernel-level para agentes (Landlock + Seccomp BPF)
- **Policies YAML**: guardrails declarativos out-of-process (el agente no puede overridearlos)
- **Audit estructurado**: toda acción del agente logueada para compliance

### Arquitectura de agentes en SDA

```
Tenant Admin configura agente (YAML via UI)
  → Agent Service (Go) almacena config en tenant DB
  → NeMo Agent Toolkit levanta agente con tools asignados
  → Agente accede a:
      ├─► RAG (colecciones del tenant)
      ├─► Document AI (procesar documentos)
      ├─► Tenant DB (leer/escribir datos del negocio)
      ├─► APIs externas (Google, WhatsApp, email)
      └─► Otros servicios del tenant (cuOpt, Vision, etc.)
  → Resultados via WebSocket al browser
  → Audit log de cada acción
```

### Agentes pre-configurados por vertical

| Agente | Industria | Qué hace |
|---|---|---|
| **Asistente de documentos** | Todas | Responde preguntas sobre documentos, extrae datos, resume |
| **Procesador de facturas** | Todas | Recibe factura → extrae datos → carga en sistema → notifica |
| **Monitor de vencimientos** | Todas | Escanea documentos con fechas → alerta cuando se acercan |
| **Clasificador de email** | Todas | Categoriza, prioriza, sugiere respuestas |
| **Analista de seguridad** | Construcción | Analiza fotos de obra → detecta falta de PPE → genera reporte |
| **Optimizador de rutas** | Transporte | Recibe pedidos del día → calcula rutas óptimas → asigna a choferes |
| **Asistente de mantenimiento** | Transporte | Analiza historial → predice fallas → programa mantenimiento |
| **Analista de contratos** | Legal/Construcción | Extrae cláusulas clave, identifica riesgos, compara versiones |

### Guardrails por tenant

Cada tenant define políticas que restringen a sus agentes:

```yaml
# Ejemplo: política para tenant "saldivia-buses"
guardrails:
  blocked_topics: ["política", "religión", "competencia"]
  data_classification: "confidential"
  max_actions_per_minute: 10
  require_approval:
    - "delete_document"
    - "send_email"
    - "modify_user"
  allowed_external_apis:
    - "maps.googleapis.com"
    - "api.whatsapp.com"
```

## Feedback Service — data flywheel

### Qué es

Event consumer cross-service que recolecta señales de calidad de toda la
plataforma. No es un CRUD — es el motor que hace que la IA mejore con el uso.

### Arquitectura

```
Chat Service ──NATS "feedback.response_quality"──►┐
Agent Service ──NATS "feedback.agent_quality"────►│
```

### Data flywheel — cómo mejora la IA

```
1. Usuario da thumbs down a respuesta RAG
   └─► Feedback Service guarda: query, respuesta, sources, score=negative

2. Se acumulan N negativos similares en la misma colección
   └─► Feedback Service detecta patrón:
       "30% de queries sobre 'plazos' en colección 'contratos' son negativas"

3. Alerta al platform admin (via Notification Service)
   └─► "Colección 'contratos' de tenant 'saldivia' tiene calidad baja en tema 'plazos'"

4. Platform admin investiga:
   └─► El chunk_size es muy chico para contratos largos
   └─► O faltan documentos relevantes en la colección
   └─► O el system prompt no guía bien al LLM

5. Ajusta → calidad mejora → feedback lo confirma → loop cerrado
```

## Frontend modular — una app, N módulos

### Principio: monolito frontend con code splitting

El frontend es **UN solo Next.js**. No hay micro-frontends, no hay Module
Federation. Cada módulo es un directorio dentro de la app que se carga con
`next/dynamic` (lazy loading). Si un tenant no tiene el módulo activo, ese
código **nunca se descarga** al browser.

El backend de cada módulo (Go) vive en `services/`. El frontend de cada
módulo vive en `apps/web/`. No están juntos. Esto es intencional: un solo
build de frontend, un solo deploy, un solo punto de entrada. La complejidad
de micro-frontends no se justifica hasta tener 10+ módulos y múltiples
developers.

## Sistema de módulos — qué ve cada tenant

### Principio: cada tenant ve solo lo que tiene habilitado

No todos los tenants son iguales. Saldivia Buses necesita gestión de flota pero
no le importa el módulo de construcción. Una empresa de arquitectura necesita
planos pero no rutas de colectivos. El sistema se adapta: el platform admin
habilita módulos por tenant, y el tenant **solo ve lo que tiene activo**.
Todo lo demás no existe para ese usuario — ni en la UI, ni en la API, ni en
la navegación.

### Endpoint de módulos — quién lo sirve

El Auth Service incluye la lista de módulos activos en el **payload inicial
del WebSocket** (preload). No hay fetch separado — cuando el usuario conecta,
ya sabe qué módulos tiene. Si un módulo se activa/desactiva en vivo, el
WebSocket Hub pushea un evento y el frontend invalida el cache.

```
WebSocket connect → Auth verifica JWT → Platform DB: tenant_modules
  → Hub envía preload: { modules: ["chat","rag","fleet","docai"], ... }
  → Frontend: TanStack Query cache hydrated → NavRail renderiza
```

Fallback REST (por si el WebSocket no está listo): `GET /v1/modules/enabled`
servido por el Auth Service (lee de Platform DB, cacheado en Redis, TTL 60s).

### Gestión documental — módulo core o platform?

La gestión documental (carpetas, permisos, versionado, búsqueda) es un
**módulo platform** (no core). No todos los tenants necesitan carpetas —
algunos solo usan chat + RAG con colecciones. Pero es el módulo platform
más común, habilitado por defecto en todos los planes excepto Starter básico.

El Ingest Service maneja el **upload y procesamiento** de documentos (pipeline
a Milvus). La gestión documental es la **capa de organización** sobre esos
documentos: carpetas, metadata, permisos de acceso, búsqueda por nombre/tipo/fecha.
Son concerns separados: Ingest procesa, Docs organiza.

Ruta frontend: `/documents` (dentro de `app/(modules)/documents/`).
Protegida por `ModuleGuard("docs")`.

### Lifecycle de migrations de módulos

```
Platform Admin activa módulo "fleet" para tenant "saldivia"
  → Platform Service:
    1. Verifica dependencias (fleet requires docs → ¿docs está activo?)
    2. Inserta en tenant_modules (enabled=true)
    3. Conecta a PostgreSQL del tenant
    4. Corre migrations UP del módulo (services/fleet/db/migrations/)
    5. Registra canales WS del módulo en el Hub
    6. Publica evento NATS: tenant.saldivia.module.enabled
    7. Responde OK

Platform Admin desactiva módulo "fleet"
  → Platform Service:
    1. Marca tenant_modules.enabled = false
    2. NO corre migrations DOWN (datos se preservan)
    3. Desregistra canales WS
    4. Publica evento NATS: tenant.saldivia.module.disabled

Platform Admin ELIMINA módulo "fleet" (acción destructiva)
  → Platform Service:
    1. Requiere confirmación ("escribí DELETE FLEET para confirmar")
    2. Corre migrations DOWN (borra tablas del módulo)
    3. Elimina registro de tenant_modules
    4. Limpia archivos del tenant asociados al módulo
    5. Registra en audit log
```

### Cómo funciona — flujo completo

```
1. PLATFORM ADMIN habilita módulo "fleet" para tenant "saldivia"
   └─► Platform Service:
       ├── Inserta en tenant_modules
       ├── Corre migrations del módulo en la DB del tenant
       ├── Registra los canales WS en el Hub
       └── Publica evento NATS: tenant.saldivia.module.enabled {module: "fleet"}

2. FRONTEND recibe el evento via WebSocket
   └─► TanStack Query invalida la query de módulos
       └─► El NavRail se re-renderiza con el nuevo item "Flota"
           └─► Las rutas /fleet/* se habilitan en el router

3. USUARIO de Saldivia navega a /fleet
   └─► Frontend carga el componente de Flota
       └─► Hace fetch a /v1/fleet/vehicles
           └─► Backend verifica: ¿tenant "saldivia" tiene módulo "fleet" activo?
               ├── SÍ → responde con datos
               └── NO → 403 Forbidden

4. USUARIO de otro tenant navega a /fleet
   └─► Frontend: la ruta no existe en su router (módulo no habilitado)
       └─► Redirect a /chat (home)
   └─► Si intenta llamar la API directamente:
       └─► 403 Forbidden
```

### Enforcement — 3 capas de protección

**Capa 1: Frontend (UX)**
- El NavRail solo muestra items de módulos activos
- Las rutas de módulos inactivos no se registran en el router
- Si alguien tipea la URL manualmente → redirect a home
- Los componentes de módulos se cargan con `next/dynamic` (code splitting) — no se descargan si el módulo no está activo

```typescript
// apps/web/src/lib/modules.ts
export function useEnabledModules() {
  const { data } = useQuery({
    queryKey: ['modules'],
    queryFn: () => api.get('/v1/modules/enabled'),
  })
  return data ?? []
}

export function useHasModule(moduleId: string) {
  const modules = useEnabledModules()
  return modules.some(m => m.id === moduleId)
}
```

**Capa 2: API Gateway (seguridad)**
- Middleware Go que intercepta cada request y verifica módulo activo
- Los endpoints de cada módulo están registrados en el manifiesto
- Si el tenant no tiene el módulo → 403 sin llegar al handler

```go
// pkg/middleware/module_check.go
func RequireModule(moduleID string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            tenantID := tenant.FromContext(r.Context())
            if !platform.HasModule(tenantID, moduleID) {
                http.Error(w, "module not enabled", http.StatusForbidden)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}
```

**Capa 3: WebSocket Hub (real-time)**
- Al conectarse, el Hub solo suscribe al usuario a canales de módulos activos
- Si se desactiva un módulo, el Hub desuscribe a todos los usuarios de ese tenant de los canales del módulo
- Eventos NATS del módulo no se pushean si el módulo está inactivo

### Desactivar un módulo — qué pasa

```
Platform Admin desactiva módulo "fleet" para tenant "saldivia"
  └─► Platform Service:
      ├── Marca tenant_modules.enabled = false
      ├── NO borra las tablas ni los datos (soft-disable)
      ├── Publica evento NATS: tenant.saldivia.module.disabled {module: "fleet"}
      └─► WebSocket Hub:
          ├── Desuscribe usuarios de canales fleet.*
          └── Pushea evento al browser → NavRail se actualiza en vivo
              └── Item "Flota" desaparece inmediatamente

Si se reactiva después:
  └─► Los datos siguen ahí. Todo vuelve a aparecer.

Si se elimina definitivamente (solo platform admin):
  └─► Migrations down + borrado de datos + cleanup
      └─► Requiere confirmación explícita ("escribí DELETE FLEET para confirmar")
```

## Base de datos — PostgreSQL

### Estrategia: una instancia Docker por tenant + platform DB

Cada tenant tiene su propia instancia PostgreSQL en un container dedicado. Aislamiento total desde el día 1. Overhead: ~30-50MB RAM por instancia (insignificante en 256GB).

```
PostgreSQL platform  ← constants, registry de tenants, billing, feature flags, config global
PostgreSQL tenant-1  ← todos los datos de Saldivia Buses
PostgreSQL tenant-2  ← todos los datos de Empresa 2
PostgreSQL tenant-3  ← todos los datos de Empresa 3
```

Backups automáticos de cada instancia se replican a cloud (S3/R2).

### Platform Database

La base de datos de control. Contiene todo lo que es cross-tenant:

- **tenants** — registry de todos los tenants con su connection info, plan, estado
- **plans** — planes de suscripción (basic, pro, enterprise) con features y límites
- **global_config** — configuración global del sistema
- **rag_models** — registry de modelos RAG disponibles y sus endpoints
- **feature_flags** — flags por tenant o globales, habilitables on-the-fly
- **billing** — uso mensual por tenant, montos, estado de pago

Desde acá se controla TODO: qué features tiene cada tenant, cuántos usuarios puede tener, qué modelos RAG puede usar, y cuánto paga.

### Tenant Database

Cada instancia de tenant tiene el mismo schema:

- **users, roles, permissions, user_roles, role_permissions** — auth + RBAC completo
- **sessions, messages** — chat con historial
- **tags, feedback** — metadata de chat
- **notifications, notification_preferences** — sistema de notificaciones
- **audit_log** — registro de todas las acciones
- **ingest_jobs, connectors** — pipeline de ingesta
- **rag_config** — configuración RAG específica del tenant

El schema se define una vez y se aplica a todos los tenants via migrations automáticas.

## WebSocket-first — real-time como canal principal

El 90% de la interactividad va por WebSocket. REST se mantiene solo para operaciones one-shot (login, upload de archivos, health checks).

### Protocolo

Tres tipos de mensaje:

| Tipo | Dirección | Qué hace |
|---|---|---|
| **subscribe** | Client → Server | Suscribirse a datos live (ej: "dame la lista de sesiones, actualizada") |
| **mutation** | Client → Server | Ejecutar acción (ej: "crear sesión", "borrar usuario") |
| **event** | Server → Client | Push de datos o notificaciones sin que el client pregunte |

### Canales

| Canal | Qué datos fluyen |
|---|---|
| `chat.messages:{sessionId}` | Streaming de mensajes + respuestas RAG en tiempo real |
| `sessions` | Lista de sesiones — se actualiza automáticamente cross-tab |
| `notifications` | Notificaciones in-app push |
| `admin.stats` | Dashboard admin con métricas live |
| `ingest.jobs` | Progreso de ingesta en tiempo real |
| `presence` | Quién está online en el sistema |
| `collections` | Estado de colecciones (nuevas, actualizadas) |

### WebSocket Hub — cómo funciona

El Hub es el intermediario entre el browser y todos los microservicios. Cuando un usuario se conecta, el Hub:

1. Valida su JWT
2. Lo registra en sus canales según sus permisos
3. Le envía proactivamente los datos que va a necesitar (preloading)
4. Escucha NATS por eventos de cualquier servicio y los pushea al browser

El browser nunca habla directo con Auth/Chat/RAG — siempre pasa por el Hub. Esto centraliza la lógica de conexión, reconnect, y autenticación en un solo punto.

## Performance — MacMaster-Carr level

### Qué hace MacMaster-Carr

Su sitio se siente instantáneo porque las páginas cargan en <100ms percibido, no hay loading spinners, y navegar se siente como una app local. Logramos lo mismo con 4 técnicas concretas.

### Pilar 1: WebSocket Preloading

El truco más importante. Cuando el usuario se loguea y conecta al WebSocket, el Hub le envía **automáticamente** los datos de las rutas más usadas: sesiones recientes, colecciones, count de notificaciones, preferencias. Esos datos se inyectan directo en el cache de TanStack Query.

Cuando el usuario navega a `/chat`, TanStack Query busca las sesiones en cache — **ya están ahí**. No hay fetch, no hay loading, no hay spinner. Los datos llegaron antes que el usuario.

### Pilar 2: Optimistic Updates

Cuando el usuario ejecuta una acción (crear sesión, enviar mensaje, borrar tag), la UI se actualiza **inmediatamente** sin esperar al server. TanStack Query guarda el estado anterior por si el server falla — en ese caso, revierte automáticamente y muestra un error.

El resultado: cada acción se siente instantánea. El usuario clickea "Nueva sesión" → aparece en la lista en <50ms → el server confirma en background.

### Pilar 3: Streaming SSR + Suspense

En vez de esperar a que todos los datos carguen para mostrar la página, Next.js envía el HTML del shell (layout, nav, estructura) inmediatamente. Los datos aparecen progresivamente via Suspense boundaries.

```
Navegación tradicional:  [── esperar datos ──][─ render ─] = 800ms hasta ver algo
Streaming SSR:           [─ shell ─]                        = 50ms hasta ver layout
                         [── datos streamean ──]            = aparecen progresivamente
```

En la práctica, los skeletons se muestran por <100ms (o ni se ven) porque el WebSocket preload ya inyectó los datos en cache.

### Pilar 4: Prefetch inteligente

Dos niveles de prefetch:

1. **Por ruta** — cuando estás en `/chat`, Next.js ya prefetched `/collections` y la sesión más reciente. Basado en un mapa de navegación probable.
2. **Por hover** — cuando el usuario pasa el mouse sobre un link, se prefetchean tanto la ruta (Next.js) como los datos (via WebSocket). Cuando clickea, todo está listo.

### Performance budget

Hay dos contextos de performance distintos:

**Primera carga** (usuario abre la app por primera vez):

| Métrica | Target | Nota |
|---|---|---|
| First Contentful Paint | <400ms | Frontend en CDN, shell HTML es estático |
| Largest Contentful Paint | <800ms | Datos viajan por Cloudflare Tunnel (agrega ~50-100ms) |
| Time to Interactive | <500ms | Hydration + WebSocket connect |
| JS bundle (gzipped) | <100KB | Code splitting agresivo |

**Navegación subsiguiente** (usuario ya conectado, WebSocket activo):

| Métrica | Target | Nota |
|---|---|---|
| Navegación entre rutas | <100ms percibido | Datos ya en cache via WebSocket preload |
| Acción → feedback visual | <50ms | Optimistic updates, sin round-trip |
| WebSocket message latency | <80ms | Inhouse → Cloudflare Tunnel → cloud |

La primera carga es más lenta porque todo pasa por el tunnel. Pero una vez conectado el WebSocket, todo se siente instantáneo porque los datos ya están en cache. El 95% del tiempo del usuario es navegación subsiguiente.

Monitoreo automático: Lighthouse CI en cada push, Web Vitals reportados a Grafana, latencia de WebSocket en Prometheus. Si alguna métrica supera el budget → alerta.

## Ciberseguridad — defensa en profundidad

Seguridad en cada capa del sistema. No es un feature — es una restricción.

### Capa 1: Perímetro (internet → sistema)

| Protección | Tecnología | Qué hace |
|---|---|---|
| **DDoS protection** | Cloudflare (via Tunnel) | Absorbe ataques volumétricos antes de que lleguen al hardware |
| **WAF** | Cloudflare WAF | Bloquea SQL injection, XSS, y OWASP Top 10 a nivel de request |
| **TLS everywhere** | Traefik + Let's Encrypt | Encripción en tránsito obligatoria. No hay HTTP, solo HTTPS. |
| **No puertos abiertos** | Cloudflare Tunnel | La workstation no expone puertos al internet. Todo pasa por el tunnel. |

### Capa 2: Autenticación (quién sos)

| Protección | Implementación |
|---|---|
| **Login aislado** | Página separada sin información del sistema. Zero leakage. |
| **Brute force protection** | Rate limiting por IP + cuenta. 5 intentos fallidos → lockout temporal (15 min). 20 intentos → lockout permanente hasta admin reset. |
| **MFA/2FA** | TOTP (Google Authenticator, Authy). Opcional por tenant, obligatorio para admins. |
| **JWT seguro** | Access token corto (15 min), refresh token largo (7 días), rotación en cada refresh. HttpOnly + Secure + SameSite cookies. |
| **Session management** | Límite de sesiones concurrentes configurable por tenant. Lista de sesiones activas visible para el usuario. "Cerrar todas las sesiones" disponible. |
| **Password policy** | Mínimo 12 caracteres, bcrypt con cost 12, check contra lista de passwords comprometidos (HaveIBeenPwned API). |

### Capa 3: Autorización (qué podés hacer)

| Protección | Implementación |
|---|---|
| **RBAC granular** | Roles → permisos → recursos. Admin, manager, user, viewer, custom. |
| **Tenant isolation** | JWT contiene tenant_slug. Doble verificación: subdomain + JWT deben coincidir. Imposible acceder a datos de otro tenant. |
| **Feature gating** | Servicios habilitados por tenant en Platform DB. Si no pagaste por RAG, no ves RAG. |
| **Audit log** | TODA acción queda registrada: quién, qué, cuándo, desde dónde (IP, user agent). Inmutable. |

### Capa 4: Red interna (entre servicios)

| Protección | Implementación |
|---|---|
| **Network segmentation** | Docker networks aisladas. Cada tenant tiene su propia network. Servicios solo pueden hablar con lo que necesitan. |
| **mTLS entre servicios** | Comunicación gRPC interna encriptada con certificados mutuos. Un servicio comprometido no puede impersonar a otro. |
| **NATS authorization** | Subjects restringidos por servicio. Chat Service solo puede publicar en `tenant.*.chat.*`, no en `tenant.*.auth.*`. |
| **Secrets management** | Variables sensibles (DB passwords, JWT keys, API keys) en Docker secrets o HashiCorp Vault. Nunca en env vars planos ni en código. |

### Capa 5: Datos (en reposo)

| Protección | Implementación |
|---|---|
| **Encryption at rest** | PostgreSQL con volúmenes encriptados (LUKS). Backups encriptados con AES-256 antes de subir a cloud. |
| **Backup aislados** | Cada tenant tiene su propio backup. Un tenant no puede acceder al backup de otro. |
| **Data retention** | Políticas configurables por tenant. Audit logs mínimo 1 año. |

### Capa 6: Código y containers

| Protección | Implementación |
|---|---|
| **Static analysis** | gosec en cada PR — detecta vulnerabilidades en código Go. |
| **Dependency scanning** | trivy escanea Docker images y dependencias por CVEs conocidos. |
| **Runtime scanning** | OWASP ZAP en CI — pruebas de penetración automatizadas contra el stack de test. |
| **Minimal containers** | Base image `scratch` o `distroless` para Go services. Sin shell, sin package manager, superficie de ataque mínima. |
| **No root** | Containers corren como usuario no-root. |
| **CrowdSec** | Intrusion detection colaborativa. Detecta patrones de ataque y comparte blocklists con la comunidad. |

### Capa 7: Monitoreo de seguridad

| Protección | Implementación |
|---|---|
| **Alertas de seguridad** | Login desde IP nueva → notificación. Múltiples 401 → alerta. Cambio de password → notificación. |
| **Anomaly detection** | Patrones inusuales de uso (100 queries en 1 minuto de un usuario que normalmente hace 5) → flag + log. |
| **Grafana security dashboard** | Panel dedicado: intentos de login fallidos, IPs bloqueadas, acciones de admin, sesiones activas por tenant. |

## Observability — debugging de microservicios

### El stack (Grafana LGTM)

| Componente | Qué hace |
|---|---|
| **OpenTelemetry** | Se instala en cada servicio Go. Genera trace IDs automáticos que viajan por todos los servicios. Captura latencia, errores, y logs con contexto. |
| **Grafana Tempo** | Almacena traces. Buscás por trace ID y ves exactamente qué pasó en cada servicio, cuánto tardó, dónde falló. |
| **Prometheus** | Métricas numéricas: requests/segundo, latencia p50/p95/p99, errores, uso de recursos. Por servicio y por tenant. |
| **Loki** | Logs centralizados de todos los servicios. Buscables por trace ID, tenant, nivel, servicio. |
| **Grafana** | Dashboard único que integra logs + traces + métricas. Clickeás un spike de errores → ves los logs → clickeás un log → ves el trace completo. |

### Cómo se ve en la práctica

**"El chat anda lento"** → Grafana → latencia p99 por servicio → RAG Service a 2s → click en trace → NVIDIA Blueprint tardando en similarity search → colección sin índice HNSW.

**"Un usuario no puede loguearse"** → Grafana → buscar por email → trace → JWT expired + refresh falló → Redis del tenant caído.

**"Algo anda mal para tenant empresa2"** → Grafana → filtrar por `tenant=empresa2` → ver TODO lo que pasa en ese tenant aislado.

### Costo: ~500MB RAM, ~2GB disco/mes. El 0.2% de la workstation.

```
  ├── fix/auth-jwt-refresh     → PR → CI → review → squash merge
```

Feature branches cortas (1-3 días máx). Nombres: `feat/`, `fix/`, `refactor/`, `docs/`.

### Commits

Formato: `tipo(servicio): descripción`

```
feat(auth): add TOTP-based MFA flow

Implement TOTP generation, verification, and recovery codes.
Uses Go standard crypto/hmac for HOTP generation.
```

**Tipos:** `feat`, `fix`, `refactor`, `test`, `docs`, `ci`, `chore`

**Scopes:** `auth`, `chat`, `rag`, `ingest`, `notification`, `ws`, `platform`, `web`, `cli`, `mcp`, `agent`, `docai`, `vision`, `feedback`, `infra`

El subject explica el problema y la solución. Si alguien lee solo el subject,
entiende qué cambió y por qué. El body agrega contexto técnico si es necesario.

### Pull Requests

**Todo cambio a `main` pasa por PR.** Sin excepciones.

Proceso:
1. Crear feature branch desde `main`
2. Implementar + tests + docs (en el mismo PR)
3. Push → CI corre automáticamente
4. Review: Opus revisa el código (agent reviewer)
5. CI verde + review aprobado → squash merge a `main`

**PR template:**

```markdown
## Summary
<!-- 1-3 bullet points: qué cambia y por qué -->

## Changes
<!-- Lista de cambios técnicos -->

## Test plan
<!-- Cómo verificar que funciona -->

## Docs updated
<!-- Qué docs se actualizaron (checklist) -->
- [ ] OpenAPI spec (si hay endpoints nuevos/modificados)
```


### Microversioning — semver por servicio

Cada servicio tiene su propio versionado semver independiente. El frontend
también tiene su propia versión. No hay "versión del sistema" — hay versiones
de servicios.

**Versión bump automático desde commit type:**

| Commit type | Bump | Ejemplo |
|---|---|---|
| `fix(auth): ...` | **patch** (1.2.3 → 1.2.4) | Bug fix, corrección menor |
| `feat(auth): ...` | **minor** (1.2.3 → 1.3.0) | Feature nueva, endpoint nuevo |
| `feat(auth)!: ...` o body con `BREAKING CHANGE:` | **major** (1.2.3 → 2.0.0) | Cambio breaking en API/proto |
| `refactor`, `test`, `docs`, `ci`, `chore` | **sin bump** | No afecta el runtime |

**Dónde vive la versión:**

```
services/auth/VERSION          ← "1.3.0"
services/chat/VERSION          ← "1.1.2"
services/rag/VERSION           ← "1.0.5"
services/feedback/VERSION      ← "0.2.0"
apps/web/package.json          ← "version": "1.4.1"
```

El archivo `VERSION` es un plaintext con el número. GitHub Actions lo lee,
bumpa según el commit type, y commitea el bump. El tag de Git es
`{servicio}/v{version}` (ej: `auth/v1.3.0`).

**Docker image tags:**

**Cómo se genera:**

```yaml
# cliff.toml (configuración git-cliff)
[changelog]
header = "# Changelog — {{ service_name }}\n"
body = """
{% for group, commits in commits | group_by(attribute="group") %}
### {{ group | upper_first }}
{% for commit in commits %}
- {{ commit.message | upper_first }} ({{ commit.id | truncate(length=7) }})
{% endfor %}
{% endfor %}
"""

[git]
conventional_commits = true
filter_unconventional = true
commit_parsers = [
    { message = "^feat", group = "Added" },
    { message = "^fix", group = "Fixed" },
    { message = "^refactor", group = "Changed" },
    { message = "^perf", group = "Performance" },
    { message = "^doc", skip = true },
    { message = "^test", skip = true },
    { message = "^ci", skip = true },
    { message = "^chore", skip = true },
]
```

### Zero-downtime deploy — cómo funciona

```
sda deploy auth
  │
  ├── 1. Pull ghcr.io/camionerou/sda-auth:1.3.0
  │
  ├── 2. Health check pre-deploy
  │      GET http://auth:8080/health → 200 OK
  │
  ├── 3. Start new container (start-first strategy)
  │      docker compose up -d --no-deps auth
  │      (nueva instancia arranca ANTES de bajar la vieja)
  │
  ├── 4. Wait for new container healthy
  │      GET http://auth-new:8080/health → 200 OK (retry 10x, 2s interval)
  │
  ├── 5. Drain old container
  │      Traefik redirige tráfico al nuevo
  │      Viejo recibe SIGTERM → deja de aceptar, termina requests en curso (30s)
  │
  ├── 6. Remove old container
  │
  ├── 7. Smoke test
  │      POST /v1/auth/login con credenciales de test → 200 OK
  │      Verify JWT contains expected claims
  │
  ├── 8. Grafana annotation
  │      "auth deployed v1.3.0" marcado en timeline
  │
  └── 9. Notify
       NATS: platform.deploy.completed {service: "auth", version: "1.3.0"}
       → Notification Service → platform admins
```

**Si el nuevo falla, el viejo nunca se tocó:**

- Paso 4 falla (nuevo no arranca) → se borra el nuevo, el viejo sigue sirviendo como si nada pasó. Zero downtime, zero impacto.
- Paso 7 falla (smoke test no pasa) → Traefik vuelve a rutear al viejo, se borra el nuevo. Los usuarios nunca vieron el nuevo.
- En ningún momento el viejo se baja hasta que el nuevo esté healthy y verificado. Esto es idéntico al modelo de Railway/Fly.io.

### Deploy de servicios GPU — drain-then-swap

Los servicios que usan GPU (RAG pipeline, nv-ingest, Vision, Speech, Optimization)
**no pueden correr dos instancias en paralelo** — no hay VRAM para duplicar los
modelos. Estos servicios usan una estrategia diferente: **drain-then-swap**.

**Distinción clave:**

| | Servicios CPU (auth, chat, ws, etc.) | Servicios GPU (rag, vision, speech, etc.) |
|---|---|---|
| **Estrategia** | Start-first (Railway-style) | Drain-then-swap |
| **Downtime** | Zero | Segundos (5-15s típico) |
| **Dos instancias** | Sí, simultáneas | No, secuenciales |
| **Rollback** | Borrar el nuevo, viejo intacto | Re-levantar el viejo |

**Flujo drain-then-swap para servicios GPU:**

```
sda deploy rag
  │
  ├── 1. Pull nueva imagen
  │
  ├── 2. Notify WebSocket Hub: "rag entering maintenance"
  │      → Hub pone requests RAG en cola (buffer de 30s)
  │      → Frontend muestra "Procesando..." (no error, no spinner de carga)
  │      → Requests que NO son RAG siguen funcionando normal
  │
  ├── 3. Drain: esperar que requests RAG en curso terminen (max 30s)
  │
  ├── 4. Stop old container
  │      → GPU VRAM liberada
  │
  ├── 5. Start new container
  │      → Modelos se cargan en GPU (10-30s para embeddings + reranking)
  │
  ├── 6. Health check
  │      GET http://rag:8080/health → 200 OK (retry 15x, 2s interval)
  │      Verify: modelo cargado, puede hacer inference
  │
  ├── 7. Notify Hub: "rag back online"
  │      → Hub dequeue: requests que estaban esperando se procesan
  │      → Frontend: "Procesando..." desaparece, respuestas llegan
  │
  └── 8. Si paso 5 o 6 falla:
         → Start OLD container (imagen anterior sigue en cache local)
         → Health check del viejo
         → Hub dequeue con el viejo
         → Alerta: "deploy de rag falló, rollback automático"
```

**Qué ve el usuario durante un deploy GPU:**

```
Antes:     "¿Qué dice el contrato?" → respuesta en 2s ✓
Durante:   "¿Cuántos vehículos hay?" → "Procesando..." (5-15s) → respuesta ✓
Después:   "¿Cuál es la ruta?" → respuesta en 2s ✓
```

El usuario **nunca ve un error**. Ve un delay breve. El chat, la navegación,
los módulos — todo sigue funcionando. Solo las queries que necesitan GPU se
encolan brevemente.

**El WebSocket Hub es la clave.** Como todo pasa por el Hub, él puede buffearear
requests de un servicio específico sin afectar al resto del sistema. Esto no
sería posible si el frontend hablara directo con los servicios.

### Ventana de mantenimiento para updates grandes

Algunos updates no se pueden hacer con drain-then-swap:
- **Cambio de modelo** (ej: embeddings v1 → v2): requiere re-indexar colecciones
- **Migration de Milvus**: requiere downtime de la DB vectorial
- **Upgrade del Blueprint**: puede cambiar el schema de la API

Para estos casos, se programa una **ventana de mantenimiento**:

```
sda maintenance start --duration 30m --message "Actualizando modelos de IA"
  │
  ├── Notify todos los tenants via WebSocket:
  │   "Mantenimiento programado: los servicios de IA estarán pausados ~30 min"
  │
  ├── Frontend muestra banner: "Mantenimiento en curso. Chat disponible sin IA."
  │   (el chat puede funcionar sin RAG — solo pierde la parte de búsqueda)
  │
  ├── ── ejecutar el update ──
  │
  └── sda maintenance end
      → Banner desaparece
      → Servicios verificados
      → Notify tenants: "Mantenimiento completado"
```

Estas ventanas se hacen fuera de horario laboral (23:00-06:00 Argentina)
y se notifican con 24hs de anticipación via Notification Service.

### Environments

| Environment | Dónde | Para qué | Deploy |
|---|---|---|---|
| **dev** | Local (Docker Compose) | Desarrollo. `make dev` levanta todo. `RAG_MOCK=true`. | Automático al guardar (hot reload) |
| **test** | CI (GitHub Actions) | Tests automáticos en cada PR. Stack completo en containers efímeros. | Automático en cada push |
| **prod** | Workstation inhouse | Producción. Tenants reales. | Manual via `sda deploy` (nunca automático) |

No hay staging. El stack de test en CI es suficiente. Si un cambio necesita
verificación manual antes de prod, se testea en dev con datos de prueba.
Deploy a prod es siempre explícito — nunca automático en merge.

## Riesgos

| Riesgo | Mitigación |
|---|---|
| Go learning curve | chi y sqlc son simples. Auth Service como primer ejercicio. |
| Complejidad de microservicios | Docker Compose simplifica. Servicios pequeños y enfocados. OpenTelemetry para debugging. |
| Scope creep general | El sistema es ambicioso. Priorizar módulos por demanda real, no por completitud. |
| NVIDIA Blueprint + Go | El proxy HTTP es simple. Go maneja SSE/streaming nativamente. |
| Migraciones multi-tenant | golang-migrate + script que itera sobre todas las instancias de tenant. |
| Latencia cloud ↔ inhouse | Cloudflare Tunnel optimiza. WebSocket mantiene conexión persistente. Budget de <50ms. |
| VRAM contention | Mode switching + LLM en API externa + MIG partitions. Monitoreo en Grafana. |
| Scope creep con módulos | Priorizar por stickiness. Un módulo sólido > tres a medias. |
| NeMo Agent Toolkit maturity | v1.5 es usable. Fallback a custom agent logic en Go si no cumple. |
| Adopción PyME en LATAM | Onboarding frictionless, pricing transparente, WhatsApp integration, soporte en español. |
| Un solo servidor, single point of failure | Backups en cloud, monitoring 24/7, graceful degradation, runbook de recovery. |
| Licencias NVIDIA AI Enterprise | Tier 1-2 servicios son Apache/MIT. NIMs de producción pueden requerir licencia ($4,500/GPU/año). |
| n8n licencia "fair-code" | Sustainable Use License prohíbe ofrecer n8n como servicio. Si se embebe per-tenant, evaluar licencia Enterprise de n8n o usar Temporal + builder visual custom. |

## Operaciones

### File storage

Documentos ingresados se almacenan en el filesystem local de la workstation, organizado por tenant:

```
/data/storage/{tenant_slug}/documents/     ← archivos originales
/data/storage/{tenant_slug}/processed/     ← archivos procesados para Milvus
```

Backups de storage van a S3/R2 junto con los backups de DB. A futuro, si el storage crece, se puede migrar a MinIO (S3-compatible, self-hosted) sin cambiar la API.

### API versioning

Todas las APIs REST externas llevan versión en el path: `/v1/auth/login`, `/v1/chat/sessions`. Cuando un endpoint cambie de forma breaking, se crea `/v2/` y el viejo se mantiene hasta que todos los frontends migren. gRPC interno usa versioning en el package del proto (`auth.v1`, `auth.v2`).

### Zero-downtime deploys

Cada servicio Go implementa graceful shutdown: al recibir SIGTERM, deja de aceptar requests nuevos pero termina los que están en curso (timeout 30s). Docker Compose con `deploy.update_config.order: start-first` levanta la nueva instancia antes de bajar la vieja. WebSocket Hub transfiere conexiones al nuevo pod via Redis pub/sub.

### Rollback

Cada Docker image se taggea con el commit SHA. Rollback = cambiar el tag al SHA anterior y `docker compose up -d`. Las migrations de DB son reversibles (cada migration tiene `up` y `down`). Si un rollback requiere down-migration, se corre con `sda db migrate-down --tenant all --steps 1`.

## Decisiones resueltas

| Pregunta | Decisión | Razón |
|---|---|---|
| Subdomain vs path routing | **Subdomain** (`saldivia.app.com`) | Aislamiento limpio, cookies por subdomain, Traefik lo soporta nativamente, escala mejor |
| gRPC desde el día 1 | **HTTP interno primero, gRPC cuando los contratos se estabilicen** | Empezar simple, migrar después. HTTP es más fácil de debuggear al principio |
| OpenAPI spec primero | **Sí** | Specs se escriben ANTES de implementar. Son el contrato frontend-backend |
| NemoClaw vs NeMo Agent Toolkit | **NeMo Agent Toolkit** | NemoClaw es alpha/single-player. Agent Toolkit v1.5 es maduro, config-driven, MCP-compatible |
| LLM local vs API externa | **API externa (Nemotron-Super-49B via NVIDIA API)** | Libera GPU para servicios especializados. Cuando el volumen justifique, se trae a local con vLLM |
| Agent framework custom vs NeMo | **NeMo Agent Toolkit** | Config-driven, evaluadores incluidos, MCP server/client, multi-RAG. Evita reinventar la rueda |
| Pricing model | **Hybrid (suscripción + créditos IA)** | PyMEs necesitan precio predecible (base). Enterprises quieren escalar (créditos). GPU propia = margen 90%+ |
| Document AI engine | **Docling (MIT) + nv-ingest** | Docling: mejor ratio calidad/performance, MIT license. nv-ingest: GPU-acelerado para volumen alto |
| Workflow engine | **n8n (embebido per-tenant)** | 400+ integraciones, visual builder, CPU-only, fair-code license (free self-hosted) |
| SSO implementation | **Cuando haya demanda enterprise** | Importante pero no urgente para PyMEs |
| WhatsApp integration | **Prioridad alta** | Canal #1 de comunicación en LATAM. No es opcional para adopción |
| Guardrails | **NeMo Guardrails (Apache 2.0)** | Per-tenant policies en YAML, content filtering, anti-jailbreak. Open source |
| OCR para planos técnicos | **Surya (GPL 3.0)** | Superior para layouts complejos y planos. 90+ idiomas. GPU-acelerado |
| Route optimization | **NVIDIA cuOpt (Apache 2.0)** | 100x más rápido que CPU solvers. GPU-nativo. Directo para vertical transporte |
| Speech ASR | **Whisper large-v3 (MIT)** | Gold standard, 99 idiomas, 1.5GB VRAM. Sin dependencia de API externa |
| Forecasting | **TimesFM (Google, Apache 2.0)** | Foundation model, 307B time points de training. Mejor generalización out-of-the-box |
