# Plan 15 — BigBrother: Network Intelligence Service

> **Prerequisito:** Plan 14 implementado (deploy estable, observabilidad completa)
> **Fecha:** 2026-04-09
> **Estado:** pendiente
> **Estimacion:** 3-4 semanas
> **Servicios afectados:** nuevo `services/bigbrother/`, `deploy/`, `modules/bigbrother/`
> **Scope:** Servicio de inteligencia de red que descubre, clasifica, documenta,
>   y permite interactuar con todos los dispositivos de la red empresarial.

---

## Contexto

500+ dispositivos en la red de Saldivia Buses (PLCs, computadoras, servidores,
cámaras, impresoras, switches). Nadie lleva inventario. Cuando algo se cae,
no hay registro de qué era ni qué hacía.

BigBrother resuelve esto: **enchufás algo a la red → el sistema ya sabe qué es**.

### Capacidades target

1. **Descubrimiento automático** — escanea la red cada 5 min, detecta todo
2. **Fingerprinting** — MAC vendor + port scan + SNMP + OS detection → clasificación
3. **Control PLC** — Modbus TCP read/write, OPC-UA browse/subscribe, full control
4. **Control PC** — SSH root (Linux), WinRM admin (Windows), SFTP file transfer
5. **Auto-documentación** — genera ficha técnica por dispositivo
6. **Integración SDA** — el agente puede responder "qué PLCs hay en la línea 3"

### Decisión de arquitectura: Opción A (microservicio SDA)

BigBrother vive en `services/bigbrother/` como un microservicio más de SDA.
Comparte auth, PostgreSQL, NATS, Redis. UI via el frontend de SDA.

## Librerías Go

| Función | Librería | Notas |
|---|---|---|
| ARP scan | `github.com/mdlayher/arp` | Layer 2, MIT, requiere raw socket (root) |
| Port scan + OS | `github.com/Ullaakut/nmap/v3` | Wrapper nmap, requiere binary `nmap` |
| SNMP | `github.com/gosnmp/gosnmp` | v2c/v3, Walk/BulkWalk |

| MAC vendor | `github.com/klauspost/oui` | Offline OUI database, IEEE source |
| Modbus TCP | `github.com/simonvetter/modbus` | Client + server, read/write registers |
| OPC-UA | `github.com/gopcua/opcua` | Browse/subscribe, used in prod by Telegraf |
| SSH | `golang.org/x/crypto/ssh` | Stdlib extended |
| SFTP | `github.com/pkg/sftp` | File transfer over SSH |
| WinRM | `github.com/masterzen/winrm` | Windows remote exec, no Kerberos |

**System dependencies:** `nmap` binary en el container BigBrother.

## Database schema

```sql
-- NNN_bigbrother.up.sql

CREATE TABLE bb_devices (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL,
    ip          INET NOT NULL,
    mac         MACADDR,
    hostname    TEXT,
    vendor      TEXT,
    device_type TEXT NOT NULL DEFAULT 'unknown',
        -- 'plc', 'workstation', 'server', 'switch', 'printer',
        -- 'camera', 'ap', 'phone', 'iot', 'unknown'
    os          TEXT,
    model       TEXT,
    location    TEXT,
    status      TEXT NOT NULL DEFAULT 'online',
        -- 'online', 'offline', 'degraded', 'new'
    first_seen  TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen   TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata    JSONB NOT NULL DEFAULT '{}',
    UNIQUE (tenant_id, mac)
);
CREATE INDEX idx_bb_devices_status ON bb_devices(tenant_id, status);
CREATE INDEX idx_bb_devices_type ON bb_devices(tenant_id, device_type);

CREATE TABLE bb_ports (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    port        INT NOT NULL,
    protocol    TEXT NOT NULL DEFAULT 'tcp',
    service     TEXT,
    version     TEXT,
    state       TEXT NOT NULL DEFAULT 'open',
    UNIQUE (device_id, port, protocol)
);

CREATE TABLE bb_capabilities (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    capability  TEXT NOT NULL,
    details     JSONB NOT NULL DEFAULT '{}',
    verified_at TIMESTAMPTZ
);

CREATE TABLE bb_plc_registers (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    protocol    TEXT NOT NULL,
    address     TEXT NOT NULL,
    name        TEXT,
    data_type   TEXT,
    last_value  TEXT,
    last_read   TIMESTAMPTZ,
    writable    BOOLEAN NOT NULL DEFAULT false,
    UNIQUE (device_id, protocol, address)
);

CREATE TABLE bb_computer_info (
    device_id   UUID PRIMARY KEY REFERENCES bb_devices(id) ON DELETE CASCADE,
    os_version  TEXT,
    username    TEXT,
    cpu         TEXT,
    ram_gb      FLOAT,
    disk_total_gb FLOAT,
    disk_free_gb  FLOAT,
    software    JSONB NOT NULL DEFAULT '[]',
    last_scan   TIMESTAMPTZ
);

CREATE TABLE bb_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL,
    device_id   UUID REFERENCES bb_devices(id) ON DELETE SET NULL,
    event_type  TEXT NOT NULL,
    details     JSONB NOT NULL DEFAULT '{}',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_bb_events_time ON bb_events(tenant_id, created_at DESC);

-- Credential storage — encrypted, separate from device data
CREATE TABLE bb_credentials (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL,
    device_id   UUID REFERENCES bb_devices(id) ON DELETE CASCADE,
    cred_type   TEXT NOT NULL, -- 'ssh_key', 'ssh_password', 'winrm', 'snmp_community'
    encrypted   BYTEA NOT NULL, -- encrypted with tenant-specific key
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    rotated_at  TIMESTAMPTZ
);
```

## Fases de implementacion

### Fase 1: Scaffold + ARP discovery loop (semana 1)

Crear servicio base SDA standard. Implementar discovery loop:
1. ARP scan cada 5 min → lista de IPs + MACs
2. Comparar con DB → nuevos = `discovered`, desaparecidos = `went_offline`
3. MAC vendor lookup offline
4. Persistir a bb_devices + emitir NATS events

**Entregable:** servicio que detecta dispositivos nuevos en la red.

---

### Fase 2: Fingerprinting + clasificación (semana 1-2)

Pipeline de fingerprinting para dispositivos nuevos/cambiados:
1. nmap `-sV -O -T4` (service + OS detection, aggressive timing)
2. SNMP GET sysDescr si puerto 161 abierto
3. Clasificación por reglas determinísticas

**Optimización (fix M3):** ARP scan cada 5 min (rapido). Full nmap fingerprint
solo para dispositivos nuevos o cada 24h para existentes. Evita 30+ min de scan.

**DHCP handling (fix M2):** bb_devices unique por MAC, no IP. Si misma MAC
aparece con IP distinta → UPDATE ip, no INSERT.

**Entregable:** cada dispositivo tiene tipo, vendor, OS, puertos, servicios.

---

### Fase 3: PLC communication module (semana 2)

Modbus TCP + OPC-UA clients.

**Modbus:**
- Connect a PLCs detectados con puerto 502
- Scan holding registers 40001-40100 para auto-discover
- Read/write individual registers via API

**OPC-UA:**
- Connect a PLCs con puerto 4840
- Browse tree completo de nodes
- Subscribe a cambios de valor

**Seguridad PLC (fix C4):**
- Write operations requieren confirmación (POST con `confirm: true`)
- Todas las operaciones auditadas via `pkg/audit`
- Credentials encriptadas en bb_credentials, no en JSONB plano

**Entregable:** leer y escribir registros PLC desde la API.

---

### Fase 4: Computer remote access (semana 2-3)

SSH + WinRM + SFTP.

**Info recopilada:**
- OS version, hostname, logged-in user
- CPU, RAM total/free, disk total/free
- Software instalado (dpkg/rpm, wmic)
- Red: IP, MAC, gateway, DNS
- Uptime, procesos top 20

**Seguridad remota (fix C4):**
- Command allowlist — solo comandos de recopilacion de info:
  `uname`, `df`, `free`, `top`, `dpkg -l`, `systeminfo`, `wmic`, `Get-Process`
- `/exec` endpoint acepta SOLO comandos de la allowlist
- Comandos custom requieren permiso `bigbrother.admin` + audit log
- Credentials encriptadas en DB, nunca en texto plano
- Key rotation: campo `rotated_at` en bb_credentials

**Entregable:** info completa de cada computadora, file sync a MinIO.

## Seguridad (restricciones, no features)

| Aspecto | Medida |
|---|---|
| Credentials | Encriptadas en DB (bb_credentials), nunca plaintext |
| PLC writes | Requieren `confirm: true` + audit log |
| Remote exec | Command allowlist por defecto, admin para custom |
| Docker network | BigBrother en red `backend`, NO en red publica |
| Audit trail | Toda operación remota auditada (quién, qué, cuándo, resultado) |
| ARP scan | Frecuencia configurable, CrowdSec allowlist para BigBrother IP |
| RBAC | 3 niveles: `read` (ver), `write` (exec allowlist + PLC read), `admin` (exec custom + scan + PLC write) |

---

## Riesgos

| Riesgo | Mitigacion |
|---|---|
| ARP scan trigger IDS/firewall | CrowdSec allowlist + configurable frequency |
| nmap 500 devices = 30+ min | ARP fast (5min) + nmap solo nuevos/diario |
| DHCP cambia IPs | Unique por MAC, UPDATE ip on change |
| PLC write = safety concern | Confirm + audit + admin-only |
| SSH key compromised | Encrypted storage + rotation + audit |
| Dispositivos behind NAT/VPN | Documentar como limitación, probe remoto futuro |
| WinRM no soporta Kerberos | Solo local accounts, documentar |

## Verificacion

1. Enchufar dispositivo nuevo → aparece en `/v1/bigbrother/devices` en < 10 min
2. PLC Modbus → leer registro via API matchea valor real
3. SSH exec `uname -a` en server Linux → output correcto
4. WinRM exec `systeminfo` en PC Windows → output correcto
5. Desenchufar dispositivo → status cambia a `offline` + NATS event
6. `GET /v1/bigbrother/stats` → conteo correcto por tipo
7. Agent: "listame los PLCs" → tool llama BigBrother API → respuesta correcta
