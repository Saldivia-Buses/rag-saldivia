# Plan 15 — BigBrother: Network Intelligence Service

> **Prerequisito:** Plan 14 implementado (deploy estable, observabilidad completa)
> **Fecha:** 2026-04-09
> **Estado:** pendiente
> **Estimacion:** 3-4 semanas
> **Servicios afectados:** nuevo `services/bigbrother/`, `deploy/`, `modules/bigbrother/`
> **Scope:** Servicio de inteligencia de red que descubre, clasifica, documenta,
>   y permite interactuar con todos los dispositivos de la red empresarial.

---

## Contexto

500+ dispositivos en la red de Saldivia Buses (PLCs, computadoras, servidores,
cámaras, impresoras, switches). Nadie lleva inventario. Cuando algo se cae,
no hay registro de qué era ni qué hacía.

BigBrother resuelve esto: **enchufás algo a la red → el sistema ya sabe qué es**.

### Capacidades target

1. **Descubrimiento automático** — escanea la red cada 5 min, detecta todo
2. **Fingerprinting** — MAC vendor + port scan + SNMP + OS detection → clasificación
3. **Control PLC** — Modbus TCP read/write, OPC-UA browse/subscribe, full control
4. **Control PC** — SSH root (Linux), WinRM admin (Windows), SFTP file transfer
5. **Auto-documentación** — genera ficha técnica por dispositivo
6. **Integración SDA** — el agente puede responder "qué PLCs hay en la línea 3"

### Decisión de arquitectura: Opción A (microservicio SDA)

BigBrother vive en `services/bigbrother/` como un microservicio más de SDA.
Comparte auth, PostgreSQL, NATS, Redis. UI via el frontend de SDA.

## Modelo de seguridad (security-auditor review applied)

### Principio: BigBrother tiene un perfil de seguridad DIFERENTE a otros servicios SDA

BigBrother necesita: raw sockets (ARP), nmap binary, host network access, root
SSH/WinRM. Esto viola las convenciones SDA (distroless, non-root, network isolation).
Estas excepciones están documentadas y acotadas:

### Docker security profile (fix C4)

```yaml
bigbrother:
  image: sda-bigbrother:latest
  cap_drop: [ALL]
  cap_add: [NET_RAW]              # ARP scan only, NOT --privileged
  read_only: true                  # immutable root filesystem
  tmpfs:
    - /tmp:size=100M               # nmap temp files
  security_opt:
    - no-new-privileges:true
  networks:
    - lan                          # macvlan for physical LAN access
    - backend                      # bridge for Postgres/NATS/Redis
  # NOT on frontend network — BigBrother never talks to Traefik
```

### Credential management — envelope encryption (fix C1)

```
┌─ Docker Secret ─┐     ┌─ bb_credentials table ─────────────┐
│  KEK (master)    │────►│  encrypted_dek  BYTEA              │
│  (per-tenant)    │     │  encrypted_data BYTEA              │
└──────────────────┘     │  key_version    INT                │
                         │  key_fingerprint TEXT (SSH only)   │
                         └────────────────────────────────────┘
```

- **KEK (Key Encryption Key):** Docker secret, uno por tenant. Nunca en DB.
- **DEK (Data Encryption Key):** Random AES-256 key por credential, encrypted by KEK.
- **Data:** SSH key/password/SNMP community encrypted by DEK.
- **Rotation:** Cambiar KEK → re-encrypt solo DEKs (rápido). Cambiar credential
  → new DEK, encrypted by current KEK.
- **Memory zeroing:** `defer clear(plaintext)` post-uso. Documentar que Go GC
  no garantiza zeroing inmediato.

### Command execution — enum-based, no user strings (fix C2)

```go
// NUNCA aceptar strings del usuario como comandos
type CommandType string
const (
    CmdSystemInfo   CommandType = "system_info"    // uname -a (Linux) / systeminfo (Windows)
    CmdDiskUsage    CommandType = "disk_usage"      // df -h / wmic logicaldisk
    CmdMemory       CommandType = "memory"          // free -m / wmic memorychip
    CmdProcesses    CommandType = "processes"        // ps aux --sort=-%mem | head -20
    CmdSoftware     CommandType = "software"         // dpkg -l / wmic product
    CmdNetwork      CommandType = "network"          // ip addr / ipconfig /all
    CmdUptime       CommandType = "uptime"           // uptime / systeminfo | find "Boot"
)

// Cada enum mapea a un comando fijo server-side. El usuario elige de un dropdown, no escribe.
var commandMap = map[CommandType]struct{ linux, windows string }{
    CmdSystemInfo: {"uname -a", "systeminfo"},
    CmdDiskUsage:  {"df -h", "wmic logicaldisk get size,freespace,caption"},
    // ...
}
```

**NO hay endpoint de comando custom.** Ni siquiera para admin. Si necesitan un
comando nuevo, se agrega al enum en código y se deploya. Esto elimina
shell injection, argument injection, y escalation via prompt injection.

### PLC safety tiers (fix C3)

```sql
ALTER TABLE bb_plc_registers ADD COLUMN safety_tier TEXT NOT NULL DEFAULT 'unclassified';
-- 'unclassified' (default, auto-discovered, READ ONLY)
-- 'safe' (informational: display text, counters)
-- 'controlled' (operational: temperature setpoint, speed)
-- 'critical' (safety: emergency stop, pressure relief)

ALTER TABLE bb_plc_registers ADD COLUMN min_value FLOAT;
ALTER TABLE bb_plc_registers ADD COLUMN max_value FLOAT;
ALTER TABLE bb_plc_registers ADD COLUMN max_writes_per_min INT DEFAULT 1;
```

**Reglas de escritura:**
- `unclassified` → **READ ONLY.** No se puede escribir hasta que un humano clasifique.
- `safe` → write permitido con `bigbrother.plc.write` permission.
- `controlled` → write permitido + valor dentro de [min, max] + rate limit.
- `critical` → write requiere 2 usuarios autenticados (two-person rule) + audit.
- Auto-discovery NUNCA clasifica. Solo detecta y marca como `unclassified`.

**Write-back verification:** Después de escribir, leer inmediatamente y verificar.
Discrepancia → alerta + rollback attempt.

### RBAC — 5 niveles (fix H1)

| Permiso | Capacidad |
|---|---|
| `bigbrother.read` | Ver dispositivos, eventos, topología, registros PLC (valores) |
| `bigbrother.exec` | Ejecutar comandos enum en computadoras (allowlist only) |
| `bigbrother.plc.read` | Leer registros PLC (ya incluido en .read, separado para granularidad) |
| `bigbrother.plc.write` | Escribir registros safe/controlled (con safety checks) |
| `bigbrother.admin` | Trigger scans, gestionar credentials, clasificar registros, critical writes |

Device-group scoping futuro: users con permisos por grupo de dispositivos
(ej: "office PCs" pero no "factory PLCs"). V1 es global per-tenant.

### Tenant isolation model (fix H3 — audit)

**Decisión: single-tenant para V1.** BigBrother opera como servicio de plataforma.
Todos los dispositivos pertenecen al tenant configurado en `SDA_TENANT_SLUG`.
`tenant_id` en las tablas es defense-in-depth (preparación multi-tenant futuro).

Razón: la red física es compartida. Un PLC no "pertenece" a un tenant — pertenece
a la red. Multi-tenant BigBrother requiere subnet-to-tenant mapping que es scope
de una versión futura.

### Agent tool safety (fix H4)

`ejecutar_comando` en el agent tool manifest requiere:
1. **UI confirmation step** — el agente propone la acción, el usuario clickea "Aprobar"
2. Rate limit: 1 exec per conversation turn
3. Log completo: prompt context + tool input + tool output + user who approved

## Database schema

```sql
-- NNN_bigbrother.up.sql

CREATE TABLE bb_devices (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL,
    ip          INET NOT NULL,
    mac         MACADDR,
    hostname    TEXT,
    vendor      TEXT,
    device_type TEXT NOT NULL DEFAULT 'unknown',
        -- 'plc', 'workstation', 'server', 'switch', 'printer',
        -- 'camera', 'ap', 'phone', 'iot', 'unknown'
    os          TEXT,
    model       TEXT,
    location    TEXT,
    status      TEXT NOT NULL DEFAULT 'online',
        -- 'online', 'offline', 'degraded', 'new'
    first_seen  TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen   TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata    JSONB NOT NULL DEFAULT '{}',
    UNIQUE (tenant_id, mac)
);
CREATE INDEX idx_bb_devices_status ON bb_devices(tenant_id, status);
CREATE INDEX idx_bb_devices_type ON bb_devices(tenant_id, device_type);

CREATE TABLE bb_ports (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    port        INT NOT NULL,
    protocol    TEXT NOT NULL DEFAULT 'tcp',
    service     TEXT,
    version     TEXT,
    state       TEXT NOT NULL DEFAULT 'open',
    UNIQUE (device_id, port, protocol)
);

CREATE TABLE bb_capabilities (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id   UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    capability  TEXT NOT NULL,
    details     JSONB NOT NULL DEFAULT '{}',
    verified_at TIMESTAMPTZ
);

CREATE TABLE bb_plc_registers (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id       UUID NOT NULL REFERENCES bb_devices(id) ON DELETE CASCADE,
    protocol        TEXT NOT NULL,
    address         TEXT NOT NULL,
    name            TEXT,
    data_type       TEXT,
    last_value      TEXT,
    last_read       TIMESTAMPTZ,
    writable        BOOLEAN NOT NULL DEFAULT false,
    safety_tier     TEXT NOT NULL DEFAULT 'unclassified',
        -- 'unclassified' (read only until human classifies)
        -- 'safe', 'controlled', 'critical'
    min_value       FLOAT,          -- NULL = no range check
    max_value       FLOAT,
    max_writes_per_min INT DEFAULT 1,
    UNIQUE (device_id, protocol, address)
);

CREATE TABLE bb_computer_info (
    device_id   UUID PRIMARY KEY REFERENCES bb_devices(id) ON DELETE CASCADE,
    os_version  TEXT,
    username    TEXT,
    cpu         TEXT,
    ram_gb      FLOAT,
    disk_total_gb FLOAT,
    disk_free_gb  FLOAT,
    software    JSONB NOT NULL DEFAULT '[]',
    last_scan   TIMESTAMPTZ
);

CREATE TABLE bb_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID NOT NULL,
    device_id   UUID REFERENCES bb_devices(id) ON DELETE SET NULL,
    event_type  TEXT NOT NULL,
    details     JSONB NOT NULL DEFAULT '{}',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_bb_events_time ON bb_events(tenant_id, created_at DESC);

-- Credential storage — envelope encryption (KEK in Docker secret, DEK per credential)
CREATE TABLE bb_credentials (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL,
    device_id       UUID REFERENCES bb_devices(id) ON DELETE CASCADE,
    cred_type       TEXT NOT NULL, -- 'ssh_key', 'ssh_password', 'winrm', 'snmp_community'
    encrypted_dek   BYTEA NOT NULL, -- DEK encrypted by tenant KEK
    encrypted_data  BYTEA NOT NULL, -- credential encrypted by DEK (AES-256-GCM)
    key_version     INT NOT NULL DEFAULT 1, -- KEK version for rotation
    key_fingerprint TEXT,           -- SSH key fingerprint (SHA-256, non-secret)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    rotated_at      TIMESTAMPTZ
);
```

## Fases de implementacion

### Fase 1: Scaffold + ARP discovery loop (semana 1)

Crear servicio base SDA standard. Implementar discovery loop:
1. ARP scan cada 5 min → lista de IPs + MACs
2. Comparar con DB → nuevos = `discovered`, desaparecidos = `went_offline`
3. MAC vendor lookup offline
4. Persistir a bb_devices + emitir NATS events

**Entregable:** servicio que detecta dispositivos nuevos en la red.

---

### Fase 2: Fingerprinting + clasificación (semana 1-2)

Pipeline de fingerprinting para dispositivos nuevos/cambiados:
1. nmap `-sV -O -T4` (service + OS detection, aggressive timing)
2. SNMP GET sysDescr si puerto 161 abierto
3. Clasificación por reglas determinísticas

**Optimización (fix M3):** ARP scan cada 5 min (rapido). Full nmap fingerprint
solo para dispositivos nuevos o cada 24h para existentes. Evita 30+ min de scan.

**DHCP handling (fix M2):** bb_devices unique por MAC, no IP. Si misma MAC
aparece con IP distinta → UPDATE ip, no INSERT.

**Entregable:** cada dispositivo tiene tipo, vendor, OS, puertos, servicios.

---

### Fase 3: PLC communication module (semana 2)

Modbus TCP + OPC-UA clients.

**Modbus:**
- Connect a PLCs detectados con puerto 502
- Scan holding registers 40001-40100 para auto-discover
- Read/write individual registers via API

**OPC-UA:**
- Connect a PLCs con puerto 4840
- Browse tree completo de nodes
- Subscribe a cambios de valor

**Seguridad PLC (fix C4):**
- Write operations requieren confirmación (POST con `confirm: true`)
- Todas las operaciones auditadas via `pkg/audit`
- Credentials encriptadas en bb_credentials, no en JSONB plano

**Entregable:** leer y escribir registros PLC desde la API.

---

### Fase 4: Computer remote access (semana 2-3)

SSH + WinRM + SFTP.

**Info recopilada:**
- OS version, hostname, logged-in user
- CPU, RAM total/free, disk total/free
- Software instalado (dpkg/rpm, wmic)
- Red: IP, MAC, gateway, DNS
- Uptime, procesos top 20

**Seguridad remota (fix C4):**
- Command allowlist — solo comandos de recopilacion de info:
  `uname`, `df`, `free`, `top`, `dpkg -l`, `systeminfo`, `wmic`, `Get-Process`
- `/exec` endpoint acepta SOLO comandos de la allowlist
- Comandos custom requieren permiso `bigbrother.admin` + audit log
- Credentials encriptadas en DB, nunca en texto plano
- Key rotation: campo `rotated_at` en bb_credentials

**Entregable:** info completa de cada computadora, file sync a MinIO.

## Seguridad (restricciones, no features)

| Aspecto | Medida |
|---|---|
| Credentials | Envelope encryption (KEK Docker secret + DEK per credential) |
| PLC writes | Safety tiers + value range + rate limit + two-person for critical |
| Remote exec | Enum-based commands only (no user strings, no shell, no custom) |
| Docker | CAP_NET_RAW only, read-only FS, no-new-privileges, seccomp |
| Network | macvlan (LAN scan) + backend bridge (DB/NATS). NO frontend |
| Audit | Fail-closed: PLC write/exec aborted if audit write fails |
| Credential access | Log every decrypt (not just use) for forensic analysis |
| ARP scan | Configurable frequency per subnet, incremental scan, IDS-documented |
| RBAC | 5 niveles: read, exec, plc.read, plc.write, admin |
| SSH keys | Validated on insert (Ed25519 preferred, RSA >= 4096), fingerprint stored |
| SNMP | Prefer v3 in prod. v2c community strings documented as plaintext limitation |
| Modbus/OPC-UA | No TLS (protocol limitation). Documented. OPC-UA default SignAndEncrypt |
| Agent tool | UI confirmation required, rate limited, full context logged |
| Tenant isolation | Single-tenant V1, all devices belong to SDA_TENANT_SLUG |
| NATS auth | Dedicated user with publish to `tenant.*.bigbrother.*` only |

### Items de seguridad adicionales (del audit)

- **Health check endpoint:** `GET /health` como todos los servicios SDA
- **Rate limiting:** 10 req/min para `/exec` y PLC write endpoints
- **Input validation:** `http.MaxBytesReader` en todos los handlers, UUID validation
- **Graceful shutdown:** Cerrar SSH sessions y Modbus connections en SIGTERM
- **Credential backup:** Encrypted backup de bb_credentials separado del backup general
- **SFTP scope:** Solo system files predefinidos (/etc/os-release, hostfile). NO user documents.
  Max 10MB por archivo, 100MB por dispositivo. Bucket separado, 30 días retención.
- **Auto-discovery scan mode:** `passive` (detect only) → `active` (ports) → `full` (read registers).
  Default: passive. Active/full requieren opt-in per device.
- **Bible exceptions documentadas:**
  - "Containers distroless/scratch" → BigBrother usa alpine (necesita nmap)
  - "Docker networks per tenant" → BigBrother usa host/macvlan (necesita L2 access)

## Riesgos

| Riesgo | Mitigacion |
|---|---|
| ARP scan trigger IDS | Frecuencia configurable, incremental scan, documentar con network team |
| nmap 500 devices = 30+ min | ARP fast (5min) + nmap solo nuevos o diario |
| DHCP cambia IPs | Unique por MAC, UPDATE ip on change |
| PLC write = daño físico | Safety tiers + value range + rate limit + two-person critical |
| SSH key compromised | Envelope encryption + rotation + decrypt logging + zeroing |
| Dispositivos behind NAT/VPN | Limitación documentada, probe remoto en versión futura |
| WinRM sin Kerberos | Solo local accounts, unique password per device, documentar |
| BigBrother comprometido = game over | CAP_NET_RAW only, read-only FS, seccomp, network isolation |
| Prompt injection → exec via agent | UI confirmation step, rate limit, full context logging |
| MAC spoofing → table bloat | Rate limit: max 50 new devices per scan cycle |
| Cross-tenant credential leak | Single-tenant V1, queries siempre filtran por tenant_id |

## Verificacion

1. Enchufar dispositivo nuevo → aparece en `/v1/bigbrother/devices` en < 10 min
2. PLC Modbus → leer registro via API matchea valor real
3. SSH exec `uname -a` en server Linux → output correcto
4. WinRM exec `systeminfo` en PC Windows → output correcto
5. Desenchufar dispositivo → status cambia a `offline` + NATS event
6. `GET /v1/bigbrother/stats` → conteo correcto por tipo
7. Agent: "listame los PLCs" → tool llama BigBrother API → respuesta correcta
