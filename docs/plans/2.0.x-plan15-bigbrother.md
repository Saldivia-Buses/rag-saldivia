## Contexto

500+ dispositivos en la red de Saldivia Buses (PLCs, computadoras, servidores,
camaras, impresoras, switches). Nadie lleva inventario. Cuando algo se cae,
no hay registro de que era ni que hacia.

BigBrother resuelve esto: **enchufas algo a la red → el sistema ya sabe que es**.

### Capacidades target

1. **Descubrimiento automatico** — escanea la red, detecta todo
2. **Fingerprinting** — MAC vendor + port scan + SNMP + OS detection → clasificacion
3. **Control PLC** — Modbus TCP read/write, OPC-UA browse/subscribe, full control
4. **Control PC** — SSH (Linux), WinRM over HTTPS (Windows), SFTP file transfer
5. **Auto-documentacion** — genera ficha tecnica por dispositivo
6. **Integracion SDA** — el agente puede responder "que PLCs hay en la linea 3"
7. **Frontend page** — pagina dedicada en el frontend para visualizar la red
8. **IPv4 only en V1** — IPv6 NDP support es trabajo futuro

### Decision de arquitectura: hibrido (pkg/ building blocks + servicio centralizado)

BigBrother vive en `services/bigbrother/` como un microservicio mas de SDA.
Comparte auth, PostgreSQL, NATS, Redis. UI via pagina dedicada en el frontend
y via agent tools en el chat.

**Principio: cada plan debe dejar cosas reutilizables para todo el sistema.**

Las capacidades puras (protocolos, crypto, patrones) van a `pkg/` para que
cualquier servicio futuro las use. BigBrother es el **policy enforcer** que
usa esos building blocks y agrega: credentials vault, safety tiers, audit
centralizado, scanner/discovery.

**Otros servicios que necesiten interactuar con devices llaman a BigBrother
via HTTP** — nunca importan `pkg/plc` directo para devices de produccion.
BigBrother centraliza permisos, safety checks, rate limits, y audit trail.

## Modelo de seguridad

### Principio: BigBrother tiene un perfil de seguridad DIFERENTE a otros servicios SDA

BigBrother necesita: raw sockets (ARP), nmap binary, host network access,
SSH/WinRM remoto. Esto viola convenciones SDA (distroless, non-root, network isolation).
Estas excepciones estan documentadas y acotadas.

### Docker security profile

```yaml
bigbrother:
  image: sda-bigbrother:latest
  cap_drop: [ALL]
  cap_add: [NET_RAW]              # ARP scan only, NOT --privileged
  read_only: true                  # immutable root filesystem
  tmpfs:
    - /tmp:size=100M,noexec,nosuid,nodev  # nmap temp files (fix H6: no exec in tmpfs)
  security_opt:
    - "seccomp:deploy/seccomp/bigbrother.json"  # allowlist profile
    - no-new-privileges:true
  networks:
    - frontend                     # Traefik routing (REQUIRED — Traefik only routes to frontend) (fix C5)
    - backend                      # NATS
    - data                         # PostgreSQL, Redis
    - lan                          # macvlan for physical LAN access
  # BigBrother needs all 4 networks. This is the largest surface of any
  # SDA container. Mitigated by: CAP_NET_RAW only, read-only FS, seccomp,
  # no-new-privileges, noexec tmpfs. Monitored via access logs.
```

**Seccomp profile** (`deploy/seccomp/bigbrother.json`): basado en el default
profile de Docker, con estos syscalls **explicitamente denegados**:
`mount`, `ptrace`, `setns`, `clone3` (con namespace flags), `keyctl`,
`pivot_root`, `reboot`, `kexec_load`, `init_module`, `delete_module`.

**Approach:** Empezar con Docker default seccomp y restar syscalls peligrosos,
en vez de construir allowlist from scratch. Validar empiricamente con
`strace nmap <target>` para capturar el set completo de syscalls necesarios
(incluye `poll`, `epoll_wait`, `mmap`, `brk`, `futex`, `openat`, `getpid`,
`gettid`, `clock_gettime` que nmap requiere). (fix L4)

### Credential management — envelope encryption con AAD

```
┌─ Docker Secret ─┐     ┌─ bb_credentials table ─────────────┐
│  KEK (master)    │────►│  encrypted_dek  BYTEA              │
│  (32 bytes)      │     │  encrypted_data BYTEA              │
└──────────────────┘     │  key_version    INT                │
                         │  key_fingerprint TEXT (SSH only)   │
                         └────────────────────────────────────┘
```

**Implementacion en `pkg/crypto/envelope.go`** (reutilizable por cualquier servicio):

Usa Go stdlib (`crypto/aes` + `crypto/cipher`):
1. Load KEK (32 bytes raw) from file path (BigBrother: `/run/secrets/bb_kek`)
2. **KEK format:** Generar con `openssl rand 32 > bb-kek` (raw bytes, NO hex).
   Startup check: `len(kek) != 32 → fatal("KEK must be exactly 32 bytes")`. (fix L7)
3. Generate random DEK (32 bytes) per credential via `crypto/rand`
4. **Nonce**: 12 bytes from `crypto/rand.Read()` — NUNCA reusar. Con random nonces
   y 96-bit nonce size, el birthday bound permite ~2^32 encryptions per key.
   Cada credential tiene su propio DEK, asi que no es concern practico.
5. **AAD (Additional Authenticated Data)**: caller provides (BigBrother usa
   `credential_id || device_id || tenant_id`). Esto vincula el ciphertext a su
   contexto. Si alguien swapea blobs, `aead.Open()` falla.
6. Encrypt data with DEK using AES-256-GCM (nonce prepended to ciphertext)
7. Encrypt DEK with KEK using AES-256-GCM (AAD = caller-provided context ID)
8. Return `encrypted_dek` + `encrypted_data` for storage
9. `defer clear(plaintext)` post-uso. Nota: Go GC no garantiza zeroing inmediato.
   `clear()` solo zeroes el backing array actual. Documentado como best-effort.

**API de `pkg/crypto/envelope`:**

```go
// Package envelope provides KEK/DEK envelope encryption with AAD.
// Any service that needs to store reversible secrets can use this.
package envelope

type Encryptor struct { kek []byte }

func NewEncryptor(kekPath string) (*Encryptor, error)  // loads + validates KEK
func (e *Encryptor) Encrypt(plaintext, aad []byte) (encDEK, encData []byte, err error)
func (e *Encryptor) Decrypt(encDEK, encData, aad []byte) ([]byte, error)
func (e *Encryptor) RotateKEK(newKEKPath string) error  // re-encrypt DEKs only
```

**Rotation:** Cambiar KEK → re-encrypt solo DEKs (rapido). Cambiar credential
→ new DEK, encrypted by current KEK.

**Rate limit decryption:** Max 10 decrypts/minuto **en Redis** (no in-memory).
Key pattern: `sda:bb:decrypt_rate:{user_id}` con INCR + EXPIRE 60s.
Survives restarts. Alerta si se supera threshold. Previene uso de BigBrother
como decryption oracle. (fix H9)

**KEK lifecycle:**
1. Generar con `openssl rand 32 > deploy/secrets/bb-kek` (gitignored, raw bytes)
2. **Backup:** copia encriptada en ubicacion offline separada (USB en caja fuerte)
3. `key_version` en `bb_credentials` permite rotation sin perder acceso
4. **Emergency:** si KEK se pierde, credentials deben re-ingresarse manualmente.
   Documentar runbook de recovery.

### WinRM: HTTPS obligatorio + cert validation (fix C1 + H7)

WinRM Basic Auth sobre HTTP transmite credentials en base64 (NO encriptado).
**BigBrother REQUIERE WinRM over HTTPS (port 5986).**

```go
// WinRM siempre via HTTPS — nunca HTTP
// insecure=false: validates server certificate (fix H7)
endpoint := winrm.NewEndpoint(host, 5986, true, false, caCert, nil, nil, 0)
client, err := winrm.NewClient(endpoint, username, password)
```

**Certificate validation (fix H7):** `insecure` parameter is `false` (validates
TLS cert). BigBrother loads a CA cert for internal PKI. Si no hay PKI, implementar
TOFU (Trust On First Use): almacenar fingerprint del cert del server en primer
connection, rechazar cambios posteriores. Windows targets necesitan certs firmados
por una CA conocida por BigBrother.

Si el target PC no soporta HTTPS WinRM, es un prerequisito de setup:
```powershell
# Setup script para Windows targets
winrm quickconfig -transport:https
```

Fallback: usar SSH para Windows donde sea posible (Win10+ soporta OpenSSH).

### SFTP path allowlist + symlink protection (fix H8)

```go
var allowedSFTPPaths = map[string]bool{
    "/etc/os-release": true,
    "/etc/hostname":   true,
    "/etc/hosts":      true,
}
```

Cualquier path fuera del mapa es rechazado en la capa de servicio.

**Symlink protection (fix H8):** Despues de validar el path contra el allowlist,
usar `sftp.Lstat()` para verificar que el target NO es un symlink:
```go
info, err := sftpClient.Lstat(path)
if info.Mode()&os.ModeSymlink != 0 {
    return fmt.Errorf("symlink detected at %s, rejected for security", path)
}
```
Esto previene que un target comprometido use symlinks para exfiltrar archivos
fuera del allowlist (ej: `/etc/hostname → /etc/shadow`).

Max 10MB por archivo, 100MB por dispositivo. Bucket MinIO separado, 30 dias retencion.

### Two-person rule — generic pattern en `pkg/approval/` (fix C7)

**`pkg/approval/approval.go`** — patron generico reutilizable:

```go
// Package approval provides a generic two-person approval pattern.
// Any operation that requires a second person to approve before execution.
package approval

type Status string
const (
    StatusPending  Status = "pending"
    StatusApproved Status = "approved"
    StatusExpired  Status = "expired"
    StatusRejected Status = "rejected"
)

type PendingAction struct {
    ID          uuid.UUID
    TenantID    uuid.UUID
    ResourceID  uuid.UUID  // what is being acted upon
    Action      string     // "plc_write", "tenant_delete", etc.
    Payload     []byte     // action-specific data (JSON)
    RequestorID uuid.UUID
    ApprovedBy  *uuid.UUID // NULL until approved (fix H-GW4/H-GW5)
    ApprovedAt  *time.Time
    Status      Status     // pending/approved/expired/rejected (fix C7)
    ExpiresAt   time.Time
    CreatedAt   time.Time
}

// Store is the interface for persisting pending actions.
type Store interface {
    Create(ctx context.Context, a PendingAction) error
    Approve(ctx context.Context, id, approverID uuid.UUID) (*PendingAction, error)
    // Approve uses: UPDATE ... SET approved_by=$1, status='approved'
    //   WHERE id=$2 AND status='pending' AND approved_by IS NULL RETURNING *
    // Atomic — concurrent approvals: only one succeeds. (fix H-GW5)
    CleanExpired(ctx context.Context) (int64, error)
}
```

BigBrother implementa `Store` contra `bb_pending_writes`. Otros servicios
futuros (platform tenant delete, compliance data purge) implementan contra
sus propias tablas usando el mismo patron.

**Flujo completo para PLC critical writes:**

1. **User A** (con `bigbrother.admin`) llama `POST /devices/{id}/registers/{addr}`
   con `safety_tier=critical` → servidor crea fila en `bb_pending_writes`:
   `{requestor_id, device_id, address, value, status: "pending", expires_at (5 min)}`.
   Retorna `202 Accepted` con `{request_id, status: "pending_approval"}`.
   **Publica NATS event** `tenant.{slug}.bigbrother.plc.approval_requested`
   para que notification pueda alertar a otros admins. (fix L6)

2. **User B** (diferente `X-User-ID`, debe tener `bigbrother.admin`) llama
   `POST /devices/{id}/registers/{addr}/approve?request_id=X`.

3. Servidor verifica:
   - `requestor_id != approver_id` (no self-approve)
   - `status = 'pending'` (no re-approve)
   - `UPDATE ... SET approved_by=$1, status='approved' WHERE id=$2 AND status='pending' AND approved_by IS NULL RETURNING *`
     → si affected rows = 0, ya fue aprobado/expirado (atómico, sin race condition) (fix H-GW5)
   - Dentro del time window (5 min)
   - Fail-closed audit de ambos usuarios
   - Safety tier checks (min/max, rate limit)
   - Ejecuta el write

4. **Timeout** (5 min sin aprobacion) → cleanup goroutine marca `status='expired'`,
   evento auditado `pending_write_expired`.

**Concurrent pending writes guard (fix C7):** Partial unique index:
```sql
CREATE UNIQUE INDEX idx_bb_pending_active
    ON bb_pending_writes(tenant_id, device_id, register_addr)
    WHERE status = 'pending';
```
Solo puede existir UN pending write activo por registro por tenant. Intentar
crear otro retorna 409 Conflict.

### PLC writes NO expuestos como agent tools (fix C4)

**PLC writes NO son accesibles via el agent runtime en V1.** PLC writes
requieren el endpoint API dedicado con safety tier enforcement, fail-closed
audit, y (para critical) two-person approval. No se pueden mediar via LLM
tool calling porque:

1. Riesgo de prompt injection es demasiado alto para actuadores fisicos
2. Two-person rule no se puede enforcar via tool confirmation
3. Validacion de value range necesita UI que muestre contexto min/max

Si PLC write tools se agregan en version futura, DEBEN tener:
`requires_confirmation: true`, server-side permission check (`bigbrother.plc.write`),
safety tier enforcement a nivel del tool (no solo la API), y flujo de
aprobacion separado para critical tier.

### Tenant isolation

**Single-tenant para V1.** Todos los dispositivos pertenecen al tenant de `SDA_TENANT_SLUG`.
`tenant_id` en TODAS las tablas (incluyendo hijas) es defense-in-depth.
**Todas las queries DEBEN filtrar por tenant_id** — nunca confiar solo en device_id FK.
**Todos los UNIQUE constraints incluyen tenant_id** — defense-in-depth. (fix H-GW3)

No hay RLS en V1. Para multi-tenant V2: agregar RLS + audit de queries.

**NATS tenant scoping:** NATS permissions usan `tenant.*.bigbrother.>` con wildcard `*`.
En V1 single-tenant esto es aceptable. Para V2, hardcodear el slug especifico
en la NATS user config: `tenant.{SPECIFIC_SLUG}.bigbrother.>`.

### Agent tool safety

`execute_command` en el agent tool manifest requiere:
1. **UI confirmation step** — `requires_confirmation: true` en tools.yaml
2. Rate limit: 1 exec per conversation turn
3. Log completo: prompt context + tool input + tool output + user who approved

**PREREQUISITO (fix C6):** El bug en `services/agent/internal/tools/executor.go`
donde `ExecuteConfirmed()` no re-verifica `RequiresConfirmation` DEBE ser
corregido ANTES de que BigBrother shipee. Fix: `ExecuteConfirmed()` debe verificar
que `def.RequiresConfirmation == true` y que existe un valid confirmation token/nonce
emitido por un `Execute()` previo. Sin esto, `requires_confirmation` es security theater.
**Esto afecta TODA tool con requires_confirmation, no solo BigBrother.**

### Scan modes (fix M7 + M14)

| Modo | Comportamiento | IDS risk |
|---|---|---|
| `passive` | Monitoreo ARP table only (no paquetes enviados). Deteccion lenta. | Zero |
| `active` | ARP sweep cada 30 min + fingerprint solo devices nuevos. | Bajo |
| `full` | ARP sweep cada 5 min + full nmap + SNMP walk. Lab/deployment inicial. | Alto |

Default: `passive`. `active`/`full` requieren opt-in via `SCAN_MODE` env var (startup default).

**Runtime mode change (fix M14):** Endpoint `POST /v1/bigbrother/scan/mode`
(requires `bigbrother.admin`) permite cambiar el scan mode sin reiniciar el
container. Util para: switch rapido a passive si se detecta actividad sospechosa.
El env var define el default al startup; el endpoint lo overridea en runtime.
Cambio auditado.

## Audit fail-closed

**Hallazgo de investigacion:** `pkg/audit.Writer.Write()` es non-failing (errores se
loguean pero no se retornan). BigBrother necesita fail-closed para PLC writes y exec.

**Solucion:** Agregar `StrictLogger` interface + `WriteStrict()` a `pkg/audit` (fix D4).
Ademas, agregar `tenant_id` a ambos metodos (fix H10):

```go
// StrictLogger is the interface for fail-closed audit logging.
// Use for operations that MUST be audited (PLC writes, remote exec).
type StrictLogger interface {
    WriteStrict(ctx context.Context, e Entry) error
}

// Entry now includes TenantID for multi-tenant isolation (fix H10).
type Entry struct {
    TenantID  string         // tenant context (nullable for backward compat)
    UserID    string
    Action    string
    Resource  string
    Details   map[string]any
    IP        string
    UserAgent string
}

// Ensure Writer implements both interfaces at compile time.
var _ Logger = (*Writer)(nil)
var _ StrictLogger = (*Writer)(nil)

// WriteStrict inserts an audit entry and returns the error.
// If audit fails, the caller MUST abort the operation.
func (w *Writer) WriteStrict(ctx context.Context, e Entry) error {
    details, err := json.Marshal(e.Details)
    if err != nil {
        details = []byte("{}")
    }
    _, err = w.db.Exec(ctx,
        `INSERT INTO audit_log (tenant_id, user_id, action, resource, details, ip_address, user_agent)
         VALUES ($1, $2, $3, $4, $5, $6, $7)`,
        nilIfEmpty(e.TenantID), nilIfEmpty(e.UserID), e.Action, nilIfEmpty(e.Resource),
        details, nilIfEmpty(e.IP), nilIfEmpty(e.UserAgent),
    )
    if err != nil {
        slog.Error("STRICT audit write failed — operation will be aborted", "error", err, "action", e.Action)
    }
    return err
}
```

**Nota:** Agregar columna `tenant_id` a `audit_log` table requiere migration en
`db/tenant/migrations/`. Nullable para backward compatibility con entries existentes.

Backward-compatible: `Write()` sigue siendo non-failing. BigBrother usa `StrictLogger`
para PLC writes y exec, `Logger` para operaciones no-criticas.

## Riesgos

| Riesgo | Mitigacion |
|---|---|
| ARP scan trigger IDS | Scan modes: passive (default), active, full. Configurable runtime. |
| nmap 500 devices = 30+ min | ARP fast + nmap solo nuevos o diario |
| DHCP cambia IPs | Upsert strategy: (1) ON CONFLICT (tenant_id, mac) WHERE mac IS NOT NULL → update IP + last_seen, (2) ON CONFLICT (tenant_id, ip) → update MAC if learned. MAC es el primary identifier, IP es secondary. |
| PLC write = dano fisico | Safety tiers + value range + rate limit + two-person critical |
| SSH key compromised | Envelope encryption + AAD + rotation + decrypt rate limit (Redis) + zeroing |
| macvlan no funciona en WSL2 | Dev con mock/stub, testing real en workstation fisica |
| WinRM credential sniffing | HTTPS obligatorio (port 5986) + cert validation / TOFU. HTTP rechazado. |
| WinRM MITM en LAN | Cert validation (insecure=false) + TOFU o PKI interna (fix H7) |
| BigBrother comprometido | CAP_NET_RAW only, read-only FS, seccomp, no-new-privileges, noexec tmpfs |
| Prompt injection → exec via agent | UI confirmation + confirmation token verification (fix C6), rate limit, full context logging |
| Prompt injection → PLC write | PLC writes NO expuestos como agent tools en V1 |
| MAC spoofing → table bloat | Rate limit: max 50 new devices per scan cycle |
| KEK perdido | Backup offline, key_version para rotation, recovery runbook |
| Credential swap via DB access | AAD en encryption vincula ciphertext a fila DB |
| Scanner goroutine muerta | Health check incluye scanner alive check |
| VMs/containers con MAC duplicado | Partial unique index + IP fallback unique |
| Modbus TCP sin autenticacion | Limitacion del protocolo. Recomendar VLAN separada. (fix M12) |
| Symlink traversal via SFTP | Lstat() check pre-read, reject symlinks (fix H8) |
| Decrypt rate limit reset on restart | Rate limit en Redis, survives restarts (fix H9) |
| Event retention DELETE locks table | Batched deletes de 1000 con sleep (fix M13) |
| Two concurrent pending writes same register | Partial unique index WHERE status='pending' (fix C7) |
| Two concurrent approvals | Atomic UPDATE ... WHERE approved_by IS NULL (fix H-GW5) |
