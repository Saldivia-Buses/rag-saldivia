## Problema

El repo tiene coverage muy desparejo. Varios servicios tienen 1-2 test files que cubren solo el camino feliz del handler. Cuando alguien rompe una invariante de tenant isolation, una rotacion de refresh token, o un consumer NATS, los tests no se enteran. El objetivo no es coverage por porcentaje — es que los paths criticos de negocio y seguridad sean verificables en CI sin tocar infra real.

## Principios de implementacion

### Patron unit (handler layer)

```go
// mock implementa la interfaz del servicio (definida en el handler)
type mockXService struct { ... }

func TestHandlerX_Escenario(t *testing.T) {
    mock := &mockXService{...}
    h := NewX(mock)
    req := httptest.NewRequest(...)
    rec := httptest.NewRecorder()
    h.Method(rec, req)
    assert.Equal(t, http.StatusXXX, rec.Code)
    // verificar body JSON cuando el error no debe filtrar internals
}
```

Siempre cubrir:
- Camino feliz (200/201/204)
- Input invalido (400)
- Missing identity header (401)
- Not found (404) — cuando aplica
- Internal error → respuesta generica "internal error", no filtrar detalles

### Patron integration (service layer)

```go
//go:build integration

func TestX_Escenario(t *testing.T) {
    pool, cleanup := setupTestDB(t)  // testcontainers postgres
    defer cleanup()
    // seed data
    svc := NewX(pool, ...)
    result, err := svc.Method(ctx, ...)
    // assert
}
```

- Siempre `defer cleanup()` en la primera linea despues de setup
- Cada test crea su propia data — no comparte estado con otros tests
- Verificar invariante de tenant isolation: data de un tenant no aparece en queries de otro

### TDD anchors — flags para investigar (no corregir en este plan)

Si al escribir un test se descubre comportamiento inesperado, documentarlo con:
```go
// TDD-ANCHOR: [descripcion del comportamiento observado]
// Si esto falla en prod → abrir issue + plan separado
```

## Infraestructura de tests compartida

### testcontainers: ya esta configurado

El patron `setupTestDB(t)` de `services/auth/internal/service/auth_integration_test.go` es el modelo. Cada servicio que necesite integration tests define su propio `setupTestDB(t)` local con las tablas que necesita.

No hay un helper compartido en `pkg/testutil/` — cada servicio es independiente para evitar coupling entre tests. Si en el futuro se repite mucho codigo, se puede extraer a `pkg/testutil/` en un plan separado.

### NATS embebido para integration tests de consumers

Para testear consumers NATS sin levantar NATS real, usar el servidor embebido `nats-server/v2/server` en modo `InProcessServer`. Patron:

```go
//go:build integration

import (
    "github.com/nats-io/nats-server/v2/server"
    "github.com/nats-io/nats.go"
)

func setupTestNATS(t *testing.T) (*nats.Conn, func()) {
    t.Helper()
    opts := &server.Options{JetStream: true, Port: -1, NoLog: true, NoSigs: true}
    ns, err := server.NewServer(opts)
    if err != nil {
        t.Fatalf("nats server: %v", err)
    }
    go ns.Start()
    if !ns.ReadyForConnections(5 * time.Second) {
        t.Fatal("nats not ready")
    }
    nc, err := nats.Connect(ns.ClientURL())
    if err != nil {
        t.Fatalf("nats connect: %v", err)
    }
    return nc, func() { nc.Close(); ns.Shutdown() }
}
```

Usar SOLO en tests de consumers donde el comportamiento depende de JetStream real. Para unit tests de logica de parsing y routing, usar mocks directamente.

### Build tags

- Tests sin dependencias externas: sin build tag (corren con `go test ./...`)
- Tests con testcontainers o NATS embebido: `//go:build integration` + ejecutar con `-tags=integration`
- CI gate 2 corre `go test ./...` sin tag de integration — los integration tests no bloquean el gate base
- Los integration tests se ejecutan en un job separado (o localmente) con `make test-integration` (ya existe en `Makefile:169`)

## Risks

### Flakiness potencial

| Escenario | Riesgo | Mitigacion |
|-----------|--------|-----------|
| testcontainers timeout | Alto en CI lento | `WithStartupTimeout(30s)`, retry logic en setup |
| NATS embebido puerto aleatorio | Bajo — usa `Port: -1` | Ya mitigado con `port: -1` |
| Tests paralelos con testcontainers | Medio — cada test crea su DB | `t.Parallel()` solo en unit tests, no en integration |
| `time.Now()` en assertions | Medio | Usar rangos (`within 1s`) en vez de equality |
| Goroutine leaks en hub tests | Bajo | Siempre llamar `hub.Run()` con context cancelable |

### testcontainers en CI

`ubuntu-latest` ya tiene Docker disponible — los runners de Gate 2 (`go-test` en `ci.yml:110`) pueden levantar testcontainers sin configuracion adicional.

**Limitacion conocida:** CI Gate 2 actualmente corre `go test -race -count=1 ./services/...` SIN `-tags=integration` (ver `ci.yml:122-135`). Los integration tests escritos en este plan **no se ejecutaran en CI** hasta que se agregue un job separado `integration-test`. Esto queda como **follow-up explicito** — fuera del scope de este plan (que es solo `_test.go`). Mientras tanto, los integration tests se validan localmente con `make test-integration` y por el agente que ejecute cada fase.

### Coverage no es suficiente

Un test que siempre pasa porque el mock no verifica argumentos no prueba nada. Regla: si el mock no captura los argumentos pasados y los assertions no los verifican, el test no aporta valor. Cada mock debe tener campos de captura para verificar que el servicio fue llamado con los parametros correctos.
