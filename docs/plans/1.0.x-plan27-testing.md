## Contexto

El analisis midio 38% de cobertura. La industria esta en ~88%. Los modulos
mas criticos del sistema no tienen tests:

- `rbac.ts` — 302 lineas de permission resolution (seguridad)
- `channels.ts` — 255 lineas de CRUD canales (messaging)
- `messaging.ts` — 243 lineas de mensajes + reactions + search (messaging)
- `ai-stream.ts` — 117 lineas del adapter NVIDIA -> AI SDK (core RAG pipeline)
- `client.ts` — 262 lineas del cliente RAG con mock mode + error handling
- Auth routes — login, logout, refresh recien cambiados en Plan 26

**Principio clave:** cuando un test falle, arreglar el codigo — no el test.
Los tests se escriben contra lo que el codigo *deberia* hacer. Si el
comportamiento real difiere, el bug es el codigo.

### Fase 4: Tests para ai-stream.ts

**Archivo nuevo:** `apps/web/src/lib/rag/__tests__/ai-stream.test.ts`

**Que testear:**

El adapter es el core del pipeline RAG. Transforma SSE de NVIDIA al protocolo AI SDK.

- Input con tokens simples → emite text-start + text-deltas + text-end
- Input con citations en delta.sources → emite data-sources con CitationSchema
- Input con `[DONE]` → no crashea, cierra correctamente
- Input con lineas malformadas → las ignora sin crashear
- Input vacio → emite text-end sin text-start (verify behavior)
- Buffer parcial entre chunks → se procesa correctamente
- Multiple tokens en un solo chunk → todos emitidos

**Como testear:** Crear ReadableStream con chunks SSE simulados,
llamar `createRagStreamResponse()`, leer la response y verificar el output.

**Estimacion:** ~8-10 tests

**Verificacion:**
- [ ] `bun test apps/web/src/lib/rag/__tests__/ai-stream.test.ts` → green
- [ ] Fix de bugs encontrados

**Commit:** `test(rag): add ai-stream adapter tests — plan27 f4`

### Fase 6: Tests para auth routes (Plan 26 access+refresh)

**Archivos nuevos:**
- `apps/web/src/__tests__/auth-login.test.ts`
- `apps/web/src/__tests__/auth-refresh.test.ts`
- `apps/web/src/__tests__/auth-logout.test.ts`

**Que testear:**

**Login:**
- Credenciales validas → 200, Set-Cookie con auth_token + refresh_token
- Email incorrecto → 401
- Password incorrecto → 401 (con delay anti-brute-force)
- Cuenta desactivada → 403
- Body invalido → 400

**Refresh:**
- Refresh token valido → 200, nuevos tokens, viejo revocado en Redis
- Sin refresh cookie → 401
- Refresh token expirado → 401
- Refresh token revocado → 401 (previene reuso)
- Usuario desactivado entre login y refresh → 403

**Logout:**
- Con tokens validos → 200, ambos JTIs revocados, cookies cleared
- Sin token → 200 (no falla, es idempotente)

**Estimacion:** ~12-15 tests

**Commit:** `test(auth): add login, refresh, logout route tests — plan27 f6`
