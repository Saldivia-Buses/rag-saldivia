## Contexto

El backend esta funcional, hardened (Plan 08), y con gRPC wiring (Plan 09).
Antes de entrar al frontend, hay 4 areas que necesitan cierre:

1. **Performance** — batch inserts, WS preloading, pool tuning
2. **Developer Experience** — interfaces para mocking, Routes() pattern, error handling
3. **gRPC expansion** — WS mutations via gRPC, health checks, blacklist en gRPC
4. **Security residual** — 3 items del post-audit que quedaron sin resolver (H1, H2, M6)

### Correcciones del review (v2)

- **B1:** interface `llm.ChatClient` debe matchear la firma real de `Client.Chat()` (5 params, no request struct)
- **B3:** WS Hub JWT expiry — usar mensaje `authenticate` para refresh token, o re-extraer JWT de cada mutation message
- **M2:** Platform `requirePlatformAdmin` tambien debe chequear blacklist
- **M3:** Auth `resolveService()` en multi-tenant debe pasar blacklist al service
- **M4:** Batch insert DELETE+COPY es all-or-nothing — documentar
- **SC1:** OpenAPI (Fase 8) reducida a 1 servicio como prueba de concepto

Este plan es la ultima ronda de trabajo en backend antes de ir al frontend (Plan 11+).
No agrega features nuevas. Cierra gaps y mejora la experiencia de desarrollo.

### Fase 3: Performance — batch insert de document pages

**Servicios:** ingest
**Esfuerzo:** chico (2-3h)

**Por que:** un PDF de 200 paginas genera 200 INSERT round-trips. Con
`pgx.CopyFrom` se reduce a 1 round-trip. El ON CONFLICT de la query actual
se maneja con una TX: DELETE viejas + COPY nuevas.

**Cambios:**

1. Crear funcion `BatchInsertPages` en `services/ingest/internal/service/extractor_consumer.go`:
   ```go
   func (c *ExtractorConsumer) batchInsertPages(ctx context.Context, docID string, pages []ExtractionPage) (int, error) {
       tx, err := c.pool.Begin(ctx)
       if err != nil {
           return 0, fmt.Errorf("begin tx: %w", err)
       }
       defer tx.Rollback(ctx)

       // Delete existing pages for idempotent re-processing
       _, err = tx.Exec(ctx, "DELETE FROM document_pages WHERE document_id = $1", docID)
       if err != nil {
           return 0, fmt.Errorf("delete old pages: %w", err)
       }

       rows := make([][]any, len(pages))
       for i, p := range pages {
           tables := p.Tables
           if tables == nil { tables = json.RawMessage("[]") }
           images := p.Images
           if images == nil { images = json.RawMessage("[]") }
           rows[i] = []any{docID, int32(p.PageNumber), p.Text, tables, images}
       }

       count, err := tx.CopyFrom(ctx,
           pgx.Identifier{"document_pages"},
           []string{"document_id", "page_number", "text", "tables", "images"},
           pgx.CopyFromRows(rows),
       )
       if err != nil {
           return 0, fmt.Errorf("copy pages: %w", err)
       }

       if err := tx.Commit(ctx); err != nil {
           return 0, fmt.Errorf("commit: %w", err)
       }
       return int(count), nil
   }
   ```

2. Reemplazar el loop en `handleResult` (lineas 130-152) por:
   ```go
   storedPages, err := c.batchInsertPages(ctx, result.DocumentID, result.Pages)
   if err != nil {
       slog.Error("batch insert pages failed", "doc_id", result.DocumentID, "error", err)
       setDocError(ctx, c.repo, result.DocumentID, "page insert failed: "+err.Error())
       msg.Nak()
       return
   }
   ```

3. Agregar import `"github.com/jackc/pgx/v5"` (para `pgx.CopyFrom`, `pgx.Identifier`)

**Archivos modificados:**
- `services/ingest/internal/service/extractor_consumer.go`

### Fase 9: DX — hot reload + pool tuning evaluacion

**Servicios:** todos (dev tooling)
**Esfuerzo:** chico (1-2h)

**Por que:** rebuild manual cada cambio es friccion. `air` da hot reload para Go.
Pool tuning se evalua pero NO se cambia prematuramente.

**Cambios hot reload:**

1. Crear `.air.toml` en la raiz del repo:
   ```toml
   [build]
   cmd = "go build -o ./tmp/main ./cmd/main.go"
   bin = "./tmp/main"
   include_ext = ["go"]
   exclude_dir = ["tmp", "vendor", "node_modules", "_archive", "apps"]
   kill_delay = "500ms"
   ```

2. Crear `services/.scaffold/.air.toml` (template para scaffold):
   ```toml
   [build]
   cmd = "go build -o ./tmp/main ./cmd/main.go"
   bin = "./tmp/main"
   include_ext = ["go"]
   exclude_dir = ["tmp"]
   ```

3. Agregar `make dev-auth` y patron general al Makefile:
   ```makefile
   dev-%:
   	cd services/$* && air
   ```

4. Agregar `air` al `.gitignore`:
   ```
   tmp/
   ```

**Pool tuning evaluacion:**

`PoolMaxConns = 4` en `pkg/tenant/resolver.go:43`.

Con 10 servicios y 3 usuarios, el max total de conns por tenant es 40 (10*4).
PostgreSQL default `max_connections = 100`. Esto deja margen para herramientas
(pgAdmin, backups, migrations).

**Decision: no cambiar.** 4 conns/pool es correcto para el MVP.
Si se necesita mas, agregar env var `POOL_MAX_CONNS` en el resolver.
Documentar la decision con un comentario en el codigo.

**Cambio unico:** agregar env var override en resolver:
```go
// In NewResolver or wherever PoolMaxConns is set:
if v := config.Env("POOL_MAX_CONNS", ""); v != "" {
    if n, err := strconv.ParseInt(v, 10, 32); err == nil && n > 0 {
        r.PoolMaxConns = int32(n)
    }
}
```

**Archivos nuevos:**
- `.air.toml`
- `services/.scaffold/.air.toml`

**Archivos modificados:**
- `Makefile` (dev-% target)
- `.gitignore` (tmp/)
- `pkg/tenant/resolver.go` (env var override para PoolMaxConns + comentario de decision)

**Verificacion:**
- [ ] `make dev-auth` arranca auth con hot reload
- [ ] Editar un handler -> servicio se re-compila automaticamente
- [ ] `make build` no afectado

**Commit:** `chore(dev): add air hot reload + pool tuning docs — plan 10 phase 9`

## Riesgos

| # | Riesgo | Mitigacion |
|---|--------|-----------|
| R1 | Redis como dependencia nueva en 7 servicios | Graceful fallback: `FailOpen: true`, Redis caido = sin blacklist pero funcional |
| R2 | Auth Routes() rompe tests existentes | `auth_test.go` ya testea handlers directamente, no rutas. Verificar. |
| R3 | WS mutations complican el protocolo | Mantener simple: action + data, response es el mismo Message struct |
| R4 | swaggo annotations son verbose y bitrot-prone | Solo 3 servicios como baseline. Evaluar antes de expandir. |
