# Plan 22 — Frontend Final: TanStack Query, Error Handling, UI Polish

> **Prerequisito:** Plan 17 + 18 completados. Plan 19 + 20 opcionales (fases 0+1+5 son validas sin ellos).
> Las fases que dependen de Plan 19 (AFIP tab) o Plan 20 (analytics/export) se ejecutan cuando esos planes se completen.
> **Fecha:** 2026-04-10
> **Estado:** Draft — pendiente aprobacion
> **Version target:** 2.0.11
> **Sprint:** think > plan > execute > review > test > ship (por fase)
> **Servicios afectados:** `apps/web/`
> **Estimacion:** 6 fases, ~6-8 dias
> **Migrations:** ninguna (frontend only)

## Contexto

Plan 17 implemento 16+ paginas ERP en el frontend. Todas funcionan, pero
tienen deuda tecnica sistematica:

1. **useState/useEffect manual** en todas las paginas — no hay cache, no hay
   retry, no hay deduplication, no hay optimistic updates
2. **Zero error handling en mutations** — si un POST falla, el usuario no ve
   nada. `console.error` no cuenta.
3. **10+ tabs del spec sin implementar** — las paginas muestran los tabs
   principales pero faltan los secundarios
4. **Create flows faltantes** — contable, ventas, facturacion son read-only,
   no se puede crear desde el frontend
5. **Search sin debounce** — cada keystroke hace un API call
6. **Helpers duplicados** — `fmtMoney` esta definido en 8+ archivos
7. **Tildes inconsistentes** — mezcla de "Contabilización" y "Contabilizacion"
8. **Pagination incompleta** — muchas paginas cargan todo sin paginar

**Este plan convierte el frontend ERP de "funciona" a "produccion-ready".**

> **Nota review 2.0.5:** Las paginas financieras son mas pequenas de lo estimado
> originalmente. Tamanos reales: contable ~217 lineas, tesoreria ~160, facturacion
> ~121 (no ~900-1000 como se asumia). Esto reduce el esfuerzo de migracion a TQ.
> Path real de las paginas: `apps/web/src/app/(modules)/administracion/{pagina}/page.tsx`

## Principios

1. **TanStack Query es el unico patron de data fetching.** Cero useState para server state.
2. **Toda mutation tiene feedback.** Toast de exito/error. Loading state en botones.
3. **Helpers se importan, no se copian.** Un archivo, un import.
4. **Mobile-first no, pero responsive si.** Tablas deben ser usables en tablet.
5. **Codigo viejo se BORRA, no se comenta.** Cuando una pagina se migra de
   `useState + useEffect + fetch manual` a `useQuery/useMutation`, el codigo
   viejo se elimina en el mismo commit. Nunca dejar codigo comentado "por las
   dudas" — git tiene el historial. Las PRs de Plan 22 no aceptan lineas
   `// const [data, setData] = useState(...)` o similares.

```typescript
const moneyFormatter = new Intl.NumberFormat("es-AR", {
  style: "currency", currency: "ARS", maximumFractionDigits: 0,
});
const numberFormatter = new Intl.NumberFormat("es-AR", {
  maximumFractionDigits: 2,
});
const dateFormatter = new Intl.DateTimeFormat("es-AR", {
  day: "2-digit", month: "2-digit", year: "numeric",
});

export function fmtMoney(n: number | null | undefined): string {
  if (n == null || n === 0) return "\u2014";
  return moneyFormatter.format(n);
}

export function fmtDate(d: string | null | undefined): string {
  if (!d) return "\u2014";
  return dateFormatter.format(new Date(d));
}

export function fmtNumber(n: number | null | undefined, decimals = 2): string {
  if (n == null) return "\u2014";
  return numberFormatter.format(n);
}

export function fmtPercent(n: number | null | undefined): string {
  if (n == null) return "\u2014";
  return `${(n * 100).toFixed(1)}%`;
}

export function parseDecimal(s: string): number {
  // Handles "1.234,56" (es-AR format) → 1234.56
  return parseFloat(s.replace(/\./g, "").replace(",", "."));
}
```

### Search con `useDeferredValue` (React 19) + fallback debounce

**Antes:** API call on every keystroke.

**Despues:** `useDeferredValue` de React 19 como default, con fallback a
`useDebounce` clasico si hay problemas de SSR o transitions.

```typescript
// apps/web/src/lib/erp/use-erp-search.ts
import { useDeferredValue, useState } from "react";

// Primary: React 19 built-in
export function useERPSearch(minLength = 2) {
  const [search, setSearch] = useState("");
  const deferredSearch = useDeferredValue(search);
  const enabled = deferredSearch.length >= minLength;
  return { search, setSearch, deferredSearch, enabled };
}
```

```typescript
// Fallback si useDeferredValue causa problemas (SSR, transitions, etc.):
// apps/web/src/lib/erp/use-debounce.ts
import { useEffect, useState } from "react";

export function useDebounce<T>(value: T, ms = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const t = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(t);
  }, [value, ms]);
  return debounced;
}
```

Uso (swap entre ambos sin cambiar el resto del codigo):

```typescript
function ArticleSearch() {
  const { search, setSearch, deferredSearch, enabled } = useERPSearch(2);
  // Fallback: const [search, setSearch] = useState("");
  //          const deferredSearch = useDebounce(search, 300);
  //          const enabled = deferredSearch.length >= 2;

  const { data } = useQuery({
    queryKey: erpKeys.stockArticles({ search: deferredSearch }),
    queryFn: () => api.get(`/v1/erp/stock/articles?search=${deferredSearch}`),
    enabled,
  });

  return <Input value={search} onChange={e => setSearch(e.target.value)} placeholder="Buscar artículos..." />;
}
```

**Decision gate:** empezar con `useDeferredValue`. Si aparece cualquier problema
(SSR hydration mismatch, input que lagea, tests flakey), cambiar a `useDebounce`
sin re-escribir las paginas — solo el hook cambia.

### Toast mapping de permisos (403 amigable)

El frontend **no tiene framework de permisos** — los botones siempre son
visibles y confian en que el backend rechaza con 403. Plan 22 no construye
framework de permisos (es otro plan aparte), pero **si** hace el toast del
403 legible traduciendo el permiso faltante a un mensaje humano.

```typescript
// apps/web/src/lib/erp/permission-messages.ts
export const permissionMessages: Record<string, string> = {
  "erp.accounting.close": "No tenes permiso para cerrar el ejercicio",
  "erp.treasury.reconcile": "No tenes permiso para reconciliar la cuenta bancaria",
  "erp.invoicing.void": "No tenes permiso para anular el comprobante",
  "erp.purchasing.inspect": "No tenes permiso para inspeccionar la recepcion",
  "erp.treasury.receipt": "No tenes permiso para registrar el recibo",
  "erp.afip.emit": "No tenes permiso para emitir comprobantes AFIP",
  "erp.afip.write": "No tenes permiso para modificar la configuracion AFIP",
  // ... uno por cada permiso nuevo de planes 18+19
};

export function permissionErrorToast(err: unknown): void {
  if (err instanceof ApiError && err.status === 403) {
    const perm = err.body?.missing_permission;
    const msg = perm ? permissionMessages[perm] : null;
    toast.error(msg ?? "No tenes permiso para esta accion", {
      description: perm ? `Permiso requerido: ${perm}` : undefined,
    });
    return;
  }
  toast.error("Error inesperado", {
    description: err instanceof Error ? err.message : undefined,
  });
}
```

**Contrato con backend:** cuando un handler rechaza por falta de permiso,
el body JSON incluye `{"missing_permission": "erp.accounting.close"}`. El
middleware `sdamw.RequirePermission` en **`pkg/middleware/rbac.go`** ya rechaza
con 403 pero hoy emite solo `{"error": "insufficient permissions"}` — sin
el nombre del permiso faltante.

**NO se crea:** `provider.tsx` ni `mutations.ts` — ya existen providers en
`lib/api/query-provider.tsx` y `lib/ws/provider.tsx`. Los mutation patterns
son inline con `useMutation` + `toast` en cada componente.

> **Regla providers vs helpers:** "NO crear providers nuevos" significa que
> `QueryProvider`, `WsProvider`, y `<Toaster />` ya existen y no se duplican.
> Crear archivos helper en `lib/erp/` **no es** crear un provider — son
> utilities importables. La regla bloquea duplicar contexts de React, no
> bloquea organizar codigo compartido.

## Post-plan: que queda reutilizable

- **lib/erp/** — todo el sistema de data fetching del ERP, reutilizable
  por cualquier modulo futuro
- **Componentes de UI** — pagination, empty/error states, export buttons,
  usables en cualquier pagina del sistema
- **`useDeferredValue` pattern** — React 19 built-in para search, sin hooks custom
- **WS → TanStack Query bridge** — patron de invalidation por dominio en
  WsProvider existente, extensible a cualquier modulo futuro
