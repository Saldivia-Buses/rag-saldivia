## Qué es este documento

El **roadmap de la serie 1.0.x** — qué se va a hacer, en qué orden, y cómo.
Para las reglas permanentes del proyecto (workflow, protocolos, naming, stack),
ver `docs/bible.md`.

## Principios (inspirados en nanostack ZEN + experiencia propia)

1. **Cuestioná el requerimiento antes de escribir código.** Si Enzo pide algo,
  Opus tiene el deber de preguntar "¿es realmente necesario?" antes de planificar.
2. **Si el plan es difícil de explicar, el plan está mal.**
3. **Borrá lo que no debería existir. No optimices lo que queda hasta hacerlo.**
4. **Si nadie usaría una v1 rota, el scope está mal.** Achicar el scope, no la ambición.
5. **Cada paso debe saber lo que el paso anterior decidió.** (artifact persistence)
6. **Evidencia le gana a convicción. Convicción le gana a consenso.**
7. **Arreglalo o preguntá. Nunca lo ignores.**
8. **La seguridad no es un tradeoff. Es una restricción.**
9. **El output debería verse mejor de lo que se pidió.**

## Stack definitivo (decidido 2026-03-29)


| Componente         | Tecnología                        | Nota                                        |
| ------------------ | --------------------------------- | ------------------------------------------- |
| Frontend + Backend | Next.js 16 App Router (unificado) | Extraíble a futuro via packages/            |
| Base de datos      | SQLite (libsql)                   | Migrar a Postgres con Drizzle cuando escale |
| ORM                | Drizzle                           | Soporta SQLite y Postgres                   |
| Runtime            | Bun                               | TypeScript nativo, rápido                   |
| Auth               | JWT (jose) + Redis blacklist      | Sin cambios                                 |
| Queue              | BullMQ + Redis                    | Sin cambios                                 |
| AI/Streaming       | **Vercel AI SDK** (`ai`)          | Reemplaza SSE manual (Plan 14)              |
| Generative UI      | **json-render**                   | Respuestas ricas del RAG (Plan 17)          |
| Validación         | Zod                               | Compartido entre todos los paquetes         |
| CSS                | Tailwind v4 + shadcn/ui + Radix   | Tokens claude-azure                         |
| Monorepo           | Turborepo + Bun workspaces        | Sin cambios                                 |
| Testing            | bun:test + happy-dom + Playwright | Sin cambios                                 |


**Modelo de deploy:** single-tenant by deployment. Cada empresa = su propio
servidor con el sistema completo. No es multi-tenant con DB compartida.

**Lo que cambia respecto a hoy:** Vercel AI SDK y json-render. El resto se mantiene.
SQLite es suficiente para single-tenant. Si un cliente necesita >50 usuarios
simultáneos, Drizzle facilita la migración a Postgres en 1 día.

**Por qué Next.js unificado:** la lógica de negocio ya está en `packages/`
(db, shared, config, logger). Si algún día se necesita un API backend separado,
se importa desde packages/ sin duplicar código.

## Herramientas disponibles — cuándo usar cada una

### MCPs configurados (`.mcp.json`)

| MCP | Estado | Para qué usarlo | Cuándo |
|---|---|---|---|
| **CodeGraphContext** | ✅ activo | Complejidad ciclomática, dead code, dependencias entre archivos, quién llama a qué | Antes de refactors. Para entender impacto de un cambio. Para encontrar código no usado. |
| **Playwright** | ✅ activo | Navegación browser, screenshots, testing visual, verificar UI en localhost | Verificación visual después de cambios de UI. E2E testing. Screenshots para comparar. |
| **GitHub** | ✅ activo | Buscar repos, leer archivos de repos remotos, buscar issues | Cuando Enzo pasa un repo para evaluar. Para ver código de librerías. |
| **context7** | ✅ activo | Docs de librerías (Next.js, Drizzle, Zod, Tailwind, shadcn) | **Antes de usar una API de librería.** Mejor que WebSearch para docs técnicos. |
| **repomix** | ⚠️ intermitente | Empaquetar múltiples archivos, grep sobre el codebase entero | **Workaround:** usar Grep + Glob + Read nativas. No se reparó en Plan 13 — funciona a veces. |
| **firecrawl** | ⚠️ intermitente | Scraping de SPAs (React/Next.js), screenshots de sitios | **Workaround:** usar WebFetch. No se reparó en Plan 13 — depende de API key/red. |

### Tools nativas de Claude Code

| Tool | Para qué | Cuándo |
|---|---|---|
| **Read** | Leer archivos del repo | Siempre — para entender código antes de modificar |
| **Grep** | Buscar patrones en el código | Para encontrar usos de una función, imports, strings |
| **Glob** | Buscar archivos por nombre/patrón | Para encontrar archivos (`**/*.test.tsx`, `**/route.ts`) |
| **Edit** | Modificar archivos existentes | Para cambios quirúrgicos en archivos existentes |
| **Write** | Crear archivos nuevos | Para archivos nuevos (componentes, tests, docs) |
| **Bash** | Comandos de terminal | Para git, bun, tsc, y cualquier comando del sistema |
| **WebFetch** | Leer contenido de URLs | Para investigar repos, docs, artículos |
| **WebSearch** | Buscar en la web | Para encontrar soluciones, docs, comparaciones |
| **Agent** | Invocar agents especializados | Para code review, security audit, debugging, etc. |

### Reglas de uso de herramientas

1. **context7 ANTES que WebSearch** para docs de librerías. context7 tiene docs
   estructuradas y actualizadas de Next.js, Drizzle, Zod, Tailwind, shadcn, etc.

2. **CodeGraphContext ANTES de refactors.** Antes de mover o renombrar algo, verificar
   quién lo usa con `analyze_code_relationships` → `find_callers`.

3. **Grep/Glob > repomix** para búsquedas locales. Las tools nativas son más rápidas
   y no dependen de un MCP.

4. **WebFetch > firecrawl** para la mayoría de URLs. firecrawl solo es necesario para
   SPAs que requieren JavaScript rendering.

5. **Playwright para verificación visual.** Después de cambios de UI, abrir localhost
   y tomar screenshot para comparar.

6. **Nunca leer 10 archivos uno por uno.** Si se necesitan muchos archivos, usar Grep
   para encontrar lo relevante primero.

## Workflow — Cómo se trabaja de ahora en más

**Centro de operaciones: Claude Code CLI.**
Todo pasa por acá. Opus hace todo — planifica, ejecuta, revisa. Cursor es backup.

### Modelo de ejecución

```
Enzo ←→ Opus (Claude Code CLI)
              ↓
         Opus planifica → Enzo aprueba
              ↓
         Opus ejecuta → quality gates → commit
              ↓
         Repeat por fase
```

**Opus (claude-opus-4-6)** — hace TODO. Planifica, ejecuta código, revisa, testea,
documenta. Un solo modelo con contexto completo. Sin delegación, sin teléfono
descompuesto.

**Cursor** — backup. Solo si Claude Code se cae o no está disponible.

### Situaciones de trabajo


| Situación | Quién | Cuándo |
|---|---|---|
| Planificación | Opus + Enzo | Antes de cualquier cambio no trivial |
| Ejecución (cualquier tamaño) | Opus | Plan aprobado, fase por fase |
| Code review | Opus | Después de cada fase (auto-review) |
| Debugging | Opus | Algo no funciona |
| Tests | Opus | Después de cada fase |
| Documentación | Opus | Después de cambios estructurales |
| Exploración | Opus | Antes de planificar |
| Deploy/ops | Opus | Cuando se pide |
| Security audit | Opus | Antes de releases |


### Reglas generales

1. **Enzo aprueba todo.** Ningún cambio se commitea sin su OK.
2. **Una fase, un commit.** Atómico, rollbackeable.
3. **Tests después de cada fase.** Si algo se rompe, se arregla antes de avanzar.
4. **Opus hace todo.** Sin delegación a otros modelos. Contexto completo siempre.
5. **Commits granulares:** `tipo(scope): descripción — planN fX`
6. **Docs siempre sincronizados.** Si un cambio afecta la doc, se actualiza en la misma fase.

### Sprint sequence (inspirado en nanostack)

Cada release sigue esta secuencia. No se salta ningún paso.

```
THINK   → Opus cuestiona el scope con Enzo. ¿Es necesario? ¿Es lo mínimo
           viable? ¿Existe algo que resuelva el 80%? (anti-sycophancy: Opus
           debe pushback si el scope es incorrecto, no solo decir que sí)

PLAN    → Opus escribe plan detallado en docs/plans/. Enzo aprueba.
           El plan es un contrato: si dice 4 archivos, se tocan 4 archivos.

EXECUTE → Opus ejecuta fase por fase. Contexto completo, sin delegación.

REVIEW  → Opus revisa cada fase en dos pasadas:
           Pass 1: ¿es correcto? ¿compila? ¿tests pasan?
           Pass 2: ¿hay scope drift? ¿cambios no planeados? ¿security issues?

TEST    → Verificación estándar después de cada fase:
           bunx tsc --noEmit → bun run test → bun run test:components → lint

SHIP    → Opus actualiza docs + CHANGELOG. Commit final. Tag si es release.
```

**Scope drift detection (de nanostack /review):**
Después de ejecutar una fase, Opus compara los archivos que realmente
cambió contra los que el plan decía. Si hay archivos extra o faltantes → warning
que se reporta a Enzo antes de commitear.

### Intensity modes

Cada tarea tiene un nivel de intensidad que determina cuánta verificación se hace:


| Modo         | Cuándo                                            | Qué se hace                                          |
| ------------ | ------------------------------------------------- | ---------------------------------------------------- |
| **Quick**    | Cambios de 1-3 archivos, docs, config             | Opus revisa directo, `tsc` + tests                   |
| **Standard** | Fases normales de un plan                         | Review 2 pasadas, tests completos, scope drift check |
| **Thorough** | Releases, cambios de seguridad, refactors grandes | Todo standard + security audit + review exhaustivo   |


### Artifact persistence

Los resultados de reviews y audits se guardan para referencia futura:

```
docs/artifacts/
  plan13-f3-review.md      ← resultado del review de la fase 3
  plan13-security-audit.md  ← auditoría de seguridad pre-release
  plan14-scope-drift.md     ← reporte de drift si lo hubo
```

Esto permite que en conversaciones futuras, un agente pueda leer qué se revisó
y qué issues se encontraron en cada fase.

### WTF likelihood (de nanostack /qa)

Si durante la ejecución de un plan se acumulan muchos fixes inesperados:

- **> 3 fixes no planeados en una fase** → parar y reportar a Enzo
- **Tests que se rompen en cascada** → parar, diagnosticar raíz antes de seguir
- **Archivos no planeados que necesitan cambio** → scope drift, parar y rediseñar

La idea: si algo sale consistentemente mal, el plan tiene un problema fundamental.
Seguir parcheando crea regresiones. Mejor parar, entender, y ajustar.

### Conflict resolution (de nanostack conflict-precedents)

Cuando dos fases del sprint se contradicen, hay reglas claras:

**Plan dice "3 PRs chicos" vs Review dice "es atómico, no separar":**
→ Agrupar tiene prioridad si hay acoplamiento real. Test: ¿cada subset funciona
  solo y se puede rollbackear sin romper los otros? Si sí → separar. Si no → junto.

**Se descubre algo nuevo durante ejecución que expande el scope:**
→ Es scope creep por defecto. Se frena y se reporta a Enzo.
→ Excepción: solo si es un prerequisito real que rompe el trabajo planeado sin él.
  "Nice to have" no es excepción.

**Security vs cualquier otra fase:**
→ Security siempre gana. Si el security audit dice que hay un problema, se arregla
  aunque el plan no lo contemplara.

### Guard rules (operaciones prohibidas)

Estas operaciones están **prohibidas** sin aprobación explícita de Enzo:


| Categoría          | Operaciones bloqueadas                                             |
| ------------------ | ------------------------------------------------------------------ |
| Destrucción masiva | `rm -rf /`, wildcard removes, recursive deletes fuera del proyecto |
| Historia de Git    | `--force` push, `reset --hard`, borrar branches remotas            |
| Base de datos      | `DROP TABLE`, `DROP DATABASE`, `TRUNCATE`                          |
| Producción         | Deploy sin verificación, acceso a env de producción sin pedir      |
| Seguridad          | `chmod 777`, `--no-verify`, `--skip-checks`                        |
| Código remoto      | Pipe de curl/wget a shell (`curl ... | sh`)                        |


Claude Code ya tiene permisos built-in, pero estas reglas son explícitas del
proyecto. Si Opus necesita hacer algo de esta lista, paran y preguntan.

## Protocolos de trabajo

### 1. Convenciones de naming

Todo agente debe seguir estas convenciones sin excepción:

**Archivos:**

| Tipo | Convención | Ejemplo |
|---|---|---|
| Componentes React | `PascalCase.tsx` | `ChatInterface.tsx` |
| Pages/routes | `page.tsx`, `layout.tsx` (convención Next.js) | `app/(app)/chat/page.tsx` |
| Hooks | `camelCase.ts` con prefijo `use` | `useRagStream.ts` |
| Lib/utils | `kebab-case.ts` | `lib/rag/client.ts` |
| Tests | mismo nombre + `.test.ts(x)` | `ChatInterface.test.tsx` |
| CSS | `kebab-case.css` | `globals.css` |
| Plans | `1.0.x-plan{N}-{slug}.md` | `1.0.x-plan14-ai-sdk.md` |
| ADRs | `{NNN}-{slug}.md` | `012-stack-definitivo-saas.md` |

**Código:**

| Tipo | Convención | Ejemplo |
|---|---|---|
| Componentes React | `PascalCase` | `function ChatInterface()` |
| Funciones | `camelCase` | `function getUserById()` |
| Variables | `camelCase` | `const sessionList = ...` |
| Constantes | `UPPER_SNAKE_CASE` | `const MAX_RETRIES = 3` |
| Tipos/Interfaces | `PascalCase` | `type UserRole = ...` |
| Zod schemas | `PascalCase` + `Schema` | `const RagParamsSchema = z.object(...)` |
| CSS classes | Tailwind utilities | `className="bg-surface text-fg"` |
| Env vars | `UPPER_SNAKE_CASE` | `JWT_SECRET` |

**Git:**

| Tipo | Convención | Ejemplo |
|---|---|---|
| Branches | `plan{N}-{slug}` o directo en `1.0.x` | `plan14-ai-sdk` |
| Commits | `tipo(scope): desc — planN fN` | `feat(chat): integrate ai sdk — plan14 f1` |
| Tags | `v{MAJOR}.{MINOR}.{PATCH}` | `v1.1.0` |

### 2. Git workflow

**Para la 1.0.x (Enzo solo):**
- Se trabaja **directo en `1.0.x`** — no hay feature branches por fase
- Un commit por fase completada
- No hay PRs (no hay reviewer humano aparte de Opus)
- Cada commit es atómico: una fase = un commit

**Cuando haya más gente:**
- Feature branch por plan: `plan{N}-{slug}`
- PR al terminar el plan completo
- Opus hace code review en el PR
- Enzo aprueba y mergea

### 3. Inicio de sesión — cómo arranca cada conversación

```
1. Opus lee MEMORY.md + CLAUDE.md (automático)
2. Enzo dice qué quiere hacer:
   - "seguimos con el plan 14" → Opus lee el plan, resume dónde quedamos
   - "quiero hacer X" → Opus evalúa si hay un plan, si no → think
   - "qué hay pendiente?" → Opus resume el estado del roadmap
3. Opus confirma con Enzo qué se va a hacer
4. Se arranca
```

**Sesión nueva sin contexto:** Opus lee la biblia
(`docs/plans/1.0.x-plan13-series-master.md`) para entender el roadmap.

### 4. Agents especializados (todos Opus)

Los agents en `.claude/agents/` son **perfiles especializados de Opus**, no modelos
distintos. Cuando Opus necesita hacer un code review, invoca el agent `frontend-reviewer`
que es Opus mismo pero con un checklist especializado.

**Todos los agents corren como Opus.** La diferencia es el contexto y la checklist,
no el modelo.

| Agent | Cuándo se usa |
|---|---|
| `frontend-reviewer` | Después de cambios en componentes/UI |
| `gateway-reviewer` | Después de cambios en API routes/auth |
| `security-auditor` | Antes de releases |
| `test-writer` | Cuando se necesitan tests nuevos |
| `debugger` | Cuando algo no funciona |
| `doc-writer` | Cuando hay que actualizar docs |
| `deploy` | Cuando se deploya |
| `status` | Para ver estado de servicios |
| `plan-writer` | Para escribir planes nuevos |
| `ingest` | Para ingestar documentos |

### 5. Quality gates

**Después de CADA fase (obligatorio):**
```bash
bunx tsc --noEmit          # tipos
bun run test               # unit tests
bun run lint               # lint
```

**Después de CADA plan completado:**
```bash
bun run test:components    # component tests
bun run test:visual        # visual regression (si hay cambios UI)
bun run test:a11y          # accesibilidad (si hay cambios UI)
```

**Antes de un release:**
```bash
# Todo lo anterior +
bun run test:e2e           # E2E
# Security audit (agent security-auditor)
# Review completo de CLAUDE.md
```

**Si un quality gate falla:** la fase NO se commitea. Se arregla primero.

### 6. Protocolo de escalación

**Opus escala a Enzo cuando:**
- Scope drift (archivos fuera del plan)
- > 3 fixes no planeados (WTF likelihood)
- Decisión de producto (naming, UX, features)
- Algo podría romper funcionalidad existente
- Cambio de dependencia (agregar/quitar paquete)
- Cualquier operación de guard rules

**Opus NO escala para:**
- Cambios rutinarios dentro del scope
- Fixes de tests del mismo cambio
- Ajustes de imports
- Formatting/lint

### 7. Recovery de errores

| Situación | Protocolo |
|---|---|
| Tests fallan post-commit | Diagnosticar raíz. Culpa del commit → fix forward. Preexistente → reportar a Enzo. **Nunca revert.** |
| Build no compila | Diagnosticar, arreglar, nuevo commit. Si no se resuelve → escalar. |
| Scope drift | **SIEMPRE parar.** Tolerancia cero. Reportar a Enzo. |
| Resultado inesperado de una edición | Diagnosticar, intentar de nuevo con otro approach. Máx 2 reintentos → escalar a Enzo. |
| Conflicto de merge | Conservar cambios recientes. Si hay ambigüedad → escalar. |

**Regla de oro:** fix forward, nunca revert. Un commit de fix es preferible
a borrar historia.

### 8. Supervisión — cómo Enzo monitorea

**Modelo: checkpoint por fase.**

Después de cada fase, Opus reporta:

```
✅ Plan 14, Fase 2 completada

Cambios:
- apps/web/src/hooks/useChat.ts (nuevo)
- apps/web/src/app/api/rag/generate/route.ts (modificado)

Tests: 311 passed, 0 failed
Types: 0 errors
Lint: clean

¿Sigo con la Fase 3 o querés revisar algo?
```

**Respuestas de Enzo:**
- **"dale"** → Opus continúa
- **"mostrame el diff"** → Opus muestra cambios detallados
- **"pará"** → Opus se detiene
- **"volvé atrás"** → Opus revierte la fase (excepción a "nunca revert")

**Al final de cada plan:** resumen completo con todos los cambios, archivos
tocados, tests agregados, y estado del repo.

## Cómo crece el proyecto — protocolo de expansión

Este es el sistema que permite que en 1 día de conversación con Opus,
el proyecto mejore x10. La clave: todo entra por conversación, todo se
formaliza en planes, todo se ejecuta con el sprint sequence.

### Flujo para agregar algo nuevo

```
Enzo: "mirá este repo" / "quiero implementar X" / "encontré esta herramienta"
                          ↓
Opus: THINK — ¿nos sirve? ¿qué problema resuelve? ¿ya tenemos algo que
       cubre el 80%? ¿encaja con el stack? ¿cuánto esfuerzo?
                          ↓
Si sirve → Opus actualiza docs/toolbox.md (si es herramienta externa)
         → Opus escribe plan usando docs/templates/plan-template.md
         → Opus agrega el plan al roadmap de la biblia
         → Enzo aprueba
         → Se ejecuta con sprint sequence
                          ↓
Si no sirve → Opus explica por qué, se registra en toolbox como "descartado"
```

### Tipos de expansión

**Nueva herramienta/librería:**
1. Opus investiga (WebFetch, firecrawl, GitHub MCP)
2. Evalúa contra nuestro stack y use case
3. Agrega a `docs/toolbox.md` con evaluación
4. Si se decide integrar → plan nuevo

**Nueva feature:**
1. Opus corre THINK (¿es necesaria? ¿scope mínimo?)
2. Escribe plan con el template
3. Agrega al roadmap
4. Se ejecuta cuando tenga prerequisitos cumplidos

**Mejora de workflow:**
1. Se discute con Enzo
2. Se actualiza la biblia directamente (protocolos, convenciones, etc.)
3. Si requiere cambios de código → plan nuevo

**Bug o problema:**
1. Agent debugger diagnostica
2. Si es chico → Opus fix directo (modo quick)
3. Si es grande → plan nuevo

### Reglas de expansión

1. **Todo pasa por la biblia.** Si un plan nuevo no está registrado en el
   roadmap de este documento, no existe oficialmente.
2. **El toolbox es el registro de herramientas.** Cuando se menciona un repo
   o herramienta, Opus lo agrega automáticamente.
3. **Los planes se pueden reordenar.** Si aparece algo urgente, se mueve arriba.
   Los prerequisitos se ajustan.
4. **Planes paralelos son posibles** si no comparten archivos. Ejemplo: Plan 14
   (AI SDK, toca hooks/api) y Plan 15 (tokens, toca CSS) pueden correr en paralelo.
5. **Un plan nunca modifica la biblia sin discutir.** La biblia se actualiza
   por conversación con Enzo, no como side-effect de un plan.

### El día de 10x

Un día productivo se ve así:

```
Mañana:
- Enzo: "arrancá el plan 14"
- Opus ejecuta fase por fase
- 4-5 fases completadas con checkpoints
- Plan 14 terminado

Mediodía:
- Enzo: "encontré este repo, miralo"
- Opus investiga, evalúa, agrega al toolbox
- Si es bueno → plan nuevo escrito en 15 minutos
- Enzo aprueba → se arranca ejecución

Tarde:
- Segundo plan en ejecución
- O: UI polish, fixes, mejoras menores en modo quick
- O: explorar más herramientas para el roadmap

Resultado: 2 planes completados + 3 herramientas evaluadas + roadmap actualizado
```

La velocidad viene de:
- **Templates** → planes se escriben en minutos, no horas
- **Agents actualizados** → Sonnet ejecuta sin confusión
- **CLAUDE.md preciso** → agentes se orientan al instante
- **Sprint sequence** → sin pasos perdidos ni regresiones
- **Quality gates** → errores se atrapan inmediatamente
- **Scope drift detection** → nunca se pierde tiempo en trabajo no planeado

## Mantenimiento de la biblia

### El problema

Toda documentación muere si nadie la mantiene. CLAUDE.md tenía 591 líneas
describiendo 24 páginas que nadie usaba. Las ADRs referencian Python.
Los agents apuntan a SvelteKit. Todo por falta de mantenimiento.

### La solución: trigger events

La biblia NO se actualiza "cuando haya tiempo". Se actualiza cuando pasa
algo específico. Estos son los triggers:

| Evento | Qué se actualiza en la biblia | Quién |
|---|---|---|
| Plan completado | Roadmap: marcar como completado | Opus, en el último commit del plan |
| Plan nuevo creado | Roadmap: agregar a la tabla | Opus, al crear el plan |
| Decisión tomada | Sección "Decisiones": agregar | Opus, en la conversación |
| Herramienta nueva encontrada | `docs/toolbox.md` (no la biblia) | Opus, automáticamente |
| Stack cambia | Sección "Stack definitivo" | Opus, requiere OK de Enzo |
| Protocolo cambia | Sección "Protocolos" | Opus, requiere OK de Enzo |
| Convención nueva | Sección "Naming/Git" | Opus, requiere OK de Enzo |
| Agente actualizado | Sección "Herramientas" si cambia un MCP | Opus |

### Qué secciones son "vivas" vs "estables"

**Vivas (cambian seguido):**
- Roadmap de planes (cada plan nuevo/completado)
- Decisiones (cada decisión nueva)
- Diagnóstico (se actualiza post Plan 13, después rara vez)

**Estables (rara vez cambian):**
- Principios
- Stack definitivo (solo cambia si hay migración)
- Organización del código
- Protocolos de trabajo
- Naming conventions
- Sprint sequence

**Si una sección viva crece demasiado → se extrae.** Por ejemplo, si el
roadmap tiene 30 planes, se mueve a `docs/roadmap.md` y la biblia solo
tiene un link. La biblia es el **índice autoritativo**, no la enciclopedia.

### Regla de oro

**Si Opus lee algo en la biblia, tiene que ser verdad HOY.**

Si no es verdad → se actualiza inmediatamente, no "después". Un dato falso
en la biblia es peor que ningún dato, porque Opus toma decisiones basándose
en lo que lee acá.

### Verificación periódica

Cada 5 planes completados (o cada ~2 semanas de trabajo), Opus hace una
revisión rápida de la biblia:

1. ¿El roadmap refleja la realidad?
2. ¿Las decisiones están actualizadas?
3. ¿Alguna sección creció demasiado y hay que extraerla?
4. ¿Las herramientas/MCPs siguen funcionando?
5. ¿Hay información contradictoria?

Esto se hace en modo quick (5-10 minutos), no es un plan entero.

### CLAUDE.md como puerta de entrada

CLAUDE.md es lo primero que cualquier agente lee. Debe contener una
referencia clara a la biblia:

```markdown
## Biblia del proyecto
Para el roadmap completo, workflow, protocolos, y decisiones:
→ `docs/plans/1.0.x-plan13-series-master.md`
Leer ANTES de cualquier trabajo significativo.
```

Así, incluso un agente que no conoce el proyecto sabe dónde ir.
