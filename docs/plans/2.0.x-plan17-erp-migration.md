# Plan 17 — ERP Migration: Histrix → SDA

> **Prerequisito:** Plan 16 (backend polish) mergeado, ERP service con sugerencias funcionando
> **Fecha:** 2026-04-09
> **Estado:** Draft — pendiente aprobacion (auditado: security + gateway, fixes aplicados)
> **Auditorias aplicadas:** security-auditor + gateway-reviewer (2026-04-09)
> **Version target:** 2.0.6+
> **Sprint:** think > plan > execute > review > test > ship (por fase)
> **Servicios afectados:** `services/erp/`, `apps/web/`, `db/tenant/migrations/`
> **Estimacion:** 13 fases + 1 fase data migration

## Contexto

Saldivia Buses opera con **Histrix**, un ERP legacy PHP/MySQL con 15+ anos de
vida. Es un framework de forms XML que genera CRUD automatico contra una base
MySQL monolitica. Funciona, pero:

- UI de 2010 (jQuery 1.6, forms XML, page reloads)
- Sin API (todo es form submission directo a DB)
- Sin audit trail real
- Sin real-time
- Sin mobile
- Sin multi-tenant
- 676 tablas, muchas duplicadas, abandonadas, o de sistema
- Business logic embebida en SQL inline dentro de XMLs
- Auth via LDAP + PHP sessions

**Este plan reemplaza Histrix con modulos ERP dentro de SDA Framework.**

Histrix sigue funcionando durante toda la migracion. Se apaga cuando SDA tiene
paridad funcional completa y la data historica esta migrada.

## Principios

1. **Reinventar, no replicar.** Misma funcionalidad de negocio, 1/5 de las tablas.
2. **Un servicio, muchos modulos.** Todo vive en `services/erp/`, cada dominio
   es un sub-modulo con handler + service + repository.
3. **Catalogos genericos.** 79 tablas lookup → 1 tabla `erp_catalogs`.
4. **Real-time desde dia 1.** NATS → WebSocket para cada operacion.
5. **Data historica despues.** Primero el sistema nuevo funciona. Despues se migra
   la data vieja con scripts dedicados.
6. **Cada fase es deployable.** No hay "fase incompleta". Cada fase termina con
   endpoints funcionando, frontend usable, y tests.

### Permisos RBAC

Cada dominio tiene su par de permisos: `erp.{dominio}.read` / `erp.{dominio}.write`.
El admin del tenant asigna permisos por usuario. Ejemplo:

| Rol | Permisos |
|---|---|
| Contadora | `erp.catalogs.read`, `erp.accounting.*`, `erp.treasury.*`, `erp.invoicing.*` |
| Jefe produccion | `erp.catalogs.read`, `erp.stock.*`, `erp.production.*`, `erp.quality.*` |
| RRHH | `erp.catalogs.read`, `erp.entities.read`, `erp.hr.*` |
| Gerencia | `erp.*` (todo) |

### NATS events (patron comun)

Cada operacion de escritura emite:

1. **Notify** → `tenant.{slug}.notify.erp_{dominio}_{accion}` — para notification service
2. **Broadcast** → `tenant.{slug}.erp_{dominio}` — para WS Hub → frontend real-time

**REGLA: NATS broadcasts solo contienen IDs, nunca montos ni PII.** El frontend
obtiene el dato completo via HTTP autenticado. Los Notify incluyen texto
descriptivo minimo (sin montos).

## Patrones obligatorios de seguridad e infraestructura

Cada fase DEBE implementar estos patrones. No son opcionales.

### P1. Tenant isolation: RLS + JWT (no env vars)

**Problema resuelto:** El handler actual de sugerencias lee tenant de `TENANT_SLUG`
env var. Esto no escala a multi-tenant y no tiene defense-in-depth.

**Regla:**
1. Fase 0 migra sugerencias para leer tenant de `tenant.FromContext(r.Context())`
   (inyectado por auth middleware desde JWT claim), no de env var.
2. Cada migration agrega RLS a sus tablas:

```sql
ALTER TABLE erp_{tabla} ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON erp_{tabla}
    USING (tenant_id = current_setting('app.tenant_id', true));
```

3. El pool wrapper (`pkg/database`) setea `SET LOCAL app.tenant_id = $1` al
   checkout de cada conexion, usando el tenant del JWT.
4. RLS es defense-in-depth. Los queries TAMBIEN deben incluir `WHERE tenant_id`.

### P2. sqlc obligatorio para todos los modulos nuevos

**Regla:** Todos los modulos nuevos (fase 0 en adelante) usan sqlc-generated code.
Fase 0 tambien migra el repositorio de sugerencias a sqlc.

`services/erp/db/queries/{modulo}.sql` con anotaciones sqlc.
`services/erp/db/sqlc.yaml` configurado.

Elimina: queries SQL hand-written con riesgo de parameter ordering bugs y
tenant_id faltante.

### P3. Inmutabilidad de registros financieros

**Regla:** Tablas financieras con status `'posted'` o `'confirmed'` son inmutables
a nivel de DB. Se implementa con triggers:

```sql
CREATE FUNCTION erp_prevent_financial_mutation() RETURNS trigger AS $$
BEGIN
    IF OLD.status IN ('posted', 'confirmed', 'paid') THEN
        RAISE EXCEPTION 'Cannot modify financial record with status %', OLD.status;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Se aplica a: `erp_journal_entries`, `erp_treasury_movements`, `erp_invoices`,
`erp_tax_entries`, `erp_stock_movements` (siempre inmutable, sin trigger — no
tiene UPDATE/DELETE en el service layer).

**Reversal:** Crear un contra-asiento/contra-movimiento nuevo, nunca UPDATE.

### P4. ON DELETE RESTRICT en documentos financieros

**Regla:** Todas las tablas hijas de documentos financieros usan
`ON DELETE RESTRICT`, nunca CASCADE. Documentos financieros no se pueden borrar
(soft-delete via status = 'cancelled' cuando estan en draft).

Aplica a: `erp_journal_lines`, `erp_invoice_lines`, `erp_tax_entries`,
`erp_treasury_movements` (hijos de bank_accounts), `erp_withholdings`.

Excepcion: tablas no-financieras en draft (cotizaciones, OC en draft) pueden
usar CASCADE.

### P5. Permission seeding en cada migration

**Regla:** Cada migration de fase incluye INSERT de permisos:

```sql
INSERT INTO permissions (id, name, description, category) VALUES
    ('erp.{dominio}.read', 'Ver {dominio}', '...', 'erp'),
    ('erp.{dominio}.write', 'Gestionar {dominio}', '...', 'erp')
ON CONFLICT (id) DO NOTHING;
```

Permisos especiales para operaciones criticas:
- `erp.accounting.reverse` — reversar asientos
- `erp.invoicing.post` — contabilizar facturas
- `erp.purchasing.approve` — aprobar OC
- `erp.treasury.confirm` — confirmar movimientos

### P6. RBAC prefix matching

**Prerequisito de fase 0:** Implementar prefix matching en
`pkg/middleware/RequirePermission()`. Si un usuario tiene `erp.accounting.*`,
matchea contra cualquier permiso que empiece con `erp.accounting.`.
`erp.*` matchea todo el ERP.

### P7. StrictLogger vs Logger

| Operacion | Audit type |
|---|---|
| Crear/modificar asiento contable | **StrictLogger** (fail-closed) |
| Crear/confirmar movimiento tesoreria | **StrictLogger** |
| Contabilizar/cancelar factura | **StrictLogger** |
| Reversar cualquier documento | **StrictLogger** |
| Cambiar salary/PII empleado | **StrictLogger** |
| CRUD catalogos, entidades, stock, etc. | Logger (best-effort) |
| Reads de PII (salary, tax_id) | Logger (access audit) |

### P8. PII encryption

Campos sensibles encriptados con `pkg/crypto/Encryptor` (envelope encryption,
AAD bound a tenant_id + entity_id):

- `erp_entities.tax_id` → `encrypted_tax_id BYTEA` + `tax_id_hash TEXT` (para busqueda)
- `erp_employee_details.salary` → `encrypted_salary BYTEA`
- `erp_employee_details.metadata` cuando contiene datos medicos/legales

El service layer encripta al escribir, desencripta al leer. El hash permite
buscar por CUIT sin desencriptar.

### P9. Body size limits

Todos los POST/PUT/PATCH usan `http.MaxBytesReader`:
- Endpoints normales: 64KB
- Document upload: 10MB
- Bulk import: 1MB

### P10. Sequence numbers atomicos

```sql
CREATE FUNCTION next_erp_sequence(p_tenant TEXT, p_domain TEXT, p_prefix TEXT DEFAULT '')
RETURNS BIGINT AS $$
    UPDATE erp_sequences SET next_value = next_value + 1
    WHERE tenant_id = p_tenant AND domain = p_domain AND prefix = p_prefix
    RETURNING next_value - 1;
$$ LANGUAGE sql;
```

Lock-free, concurrency-safe. Usado por todos los modulos para generar numeros
de factura, remito, OC, asiento, etc.

### P11. RequireModule middleware

Todos los endpoints ERP estan detras de `RequireModule("erp")`. Si el tenant
no tiene habilitado el modulo ERP, recibe 403. Implementar en
`pkg/middleware/module.go` como parte de fase 0.

### P12. Stock levels sync

`erp_stock_levels` se actualiza dentro de la MISMA transaccion que el INSERT
en `erp_stock_movements`. El service layer usa `pool.BeginTx()` para ambas
operaciones. Si falla cualquiera, rollback completo.

### P13. Balance validation en DB

```sql
CREATE FUNCTION erp_validate_journal_balance() RETURNS trigger AS $$
DECLARE
    total_debit NUMERIC;
    total_credit NUMERIC;
BEGIN
    SELECT COALESCE(SUM(debit), 0), COALESCE(SUM(credit), 0)
    INTO total_debit, total_credit
    FROM erp_journal_lines WHERE entry_id = NEW.id AND tenant_id = NEW.tenant_id;

    IF total_debit != total_credit THEN
        RAISE EXCEPTION 'Journal entry % is unbalanced: debit=% credit=%',
            NEW.number, total_debit, total_credit;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Se valida al cambiar status a 'posted' (deferred check)
CREATE CONSTRAINT TRIGGER trg_journal_balance
    AFTER UPDATE ON erp_journal_entries
    DEFERRABLE INITIALLY DEFERRED
    FOR EACH ROW
    WHEN (NEW.status = 'posted' AND OLD.status != 'posted')
    EXECUTE FUNCTION erp_validate_journal_balance();
```

## Fase 13 — Data Migration (MySQL → PostgreSQL)

**Se ejecuta cuando las fases 0-12 estan completas y el sistema nuevo funciona.**

### Approach

Un CLI command: `sda migrate-legacy` que:

1. Se conecta a MySQL viejo (read-only)
2. Para cada dominio, lee las tablas legacy
3. Transforma: agrega tenant_id, convierte IDs a UUID (con tabla de mapping),
   limpia encoding, normaliza datos
4. Escribe en PostgreSQL nuevo
5. Genera reporte de migracion (registros migrados, errores, warnings)

### Tabla de mapping

```sql
-- En la DB de migracion (temporal, no en tenant)
CREATE TABLE migration_id_map (
    legacy_table TEXT NOT NULL,
    legacy_id    TEXT NOT NULL,      -- ID viejo (puede ser INT o VARCHAR)
    new_id       UUID NOT NULL,
    PRIMARY KEY (legacy_table, legacy_id)
);
```

### Orden de migracion (respetar FKs)

1. erp_catalogs ← todas las tablas TIPO*/GEN_*
2. erp_entities ← PERSONAL + REG_CUENTA (clientes/proveedores)
3. erp_articles ← STK_ARTICULOS
4. erp_warehouses ← DEPOSITO, STK_DEPOSITOS
5. erp_accounts ← CTB_CUENTAS
6. erp_cost_centers ← CTB_CENTROS
7. erp_departments ← SECCION, RH_SECCIONES
8. ... (resto en orden de dependencia)
9. Movimientos transaccionales al final (stock, contabilidad, tesoreria)

### Lo que NO se migra

- Tablas HTX_* (sistema Histrix)
- Tablas _OLD, _BORRAR, _BAK
- oauth_* (auth es otro sistema)
- replicasql, tmp_*, calendar* DAV

### Seguridad del migration tool

- Credenciales MySQL via Docker secrets file, nunca command-line args
- User MySQL: read-only (`SELECT` grant only)
- User PostgreSQL: write-only en schema ERP
- `migration_id_map` en schema temporal, DROP despues de validacion
- CLI no loguea valores PII — solo counts e IDs
- Ejecutar en red privada sin acceso externo

### Validacion post-migracion

Para cada dominio:
- Count de registros legacy vs nuevo
- Sumas de control (montos totales de movimientos)
- Sample check manual (10 registros random por tabla)
