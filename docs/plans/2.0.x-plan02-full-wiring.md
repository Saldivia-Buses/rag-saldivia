---

## Contexto

UI funcional pero con datos mock. Son dos islas que no se hablan. Este plan

Cada frontend sabe su tenant por subdomain (`saldivia.sda.app`).

## Fase 1 — API client + TanStack Query + auth guard

> El frontend puede hablar con el backend y proteger rutas.

### 1.1 Instalar dependencias

```bash
cd apps/web
bun add @tanstack/react-query zod
```

### 1.2 API client (`apps/web/src/lib/api/client.ts`)

```typescript
// Resuelve tenant del hostname: saldivia.sda.app → "saldivia"
// En dev: lee de env NEXT_PUBLIC_TENANT_SLUG o usa "dev"
// Manda Authorization: Bearer <token> en cada request
// Auto-refresh del token cuando expira (401 → POST /v1/auth/refresh)
// Retry logic: 1 retry en 5xx, 0 en 4xx
```

Funciones:
- `api.get<T>(path)` — GET con auth headers
- `api.post<T>(path, body)` — POST con auth headers
- `api.patch<T>(path, body)` — PATCH
- `api.delete(path)` — DELETE
- `api.stream(path, body)` — SSE streaming (para RAG)
- `getApiBaseUrl()` — resuelve URL del backend (en prod: via Cloudflare Tunnel, en dev: `http://localhost` via Traefik)

Token storage:
- Access token: en memoria (variable, no localStorage)
- Refresh token: HttpOnly cookie (set por el backend) — **requiere cambio en auth service**

### 1.3 Auth store (`apps/web/src/lib/auth/store.ts`)

Usar Zustand (ya instalado pero no usado):
- `user`: datos del usuario (id, email, name, role, tenantSlug)
- `accessToken`: en memoria
- `isAuthenticated`: derived
- `login(email, password)`: llama `/v1/auth/login`, guarda tokens
- `logout()`: revoca refresh, limpia state, redirect a login app
- `refresh()`: llama `/v1/auth/refresh`, actualiza access token

### 1.4 TanStack Query provider (`apps/web/src/lib/api/query-provider.tsx`)

- Wraps app con `QueryClientProvider`
- Default staleTime: 30s para queries normales
- `Infinity` para datos que se invalidan por WebSocket (módulos, notificaciones)
- Retry: 1 para queries, 0 para mutations
- Error handler global: 401 → redirect a login

### 1.5 Auth guard middleware (`apps/web/src/middleware.ts`)

Next.js middleware que:
- Rutas públicas (`/login`, `/forgot-password`): pass through
- Rutas protegidas (`/chat`, `/dashboard`, etc.): verificar que hay token
  - Sin token → redirect a login app
  - Con token expirado → intentar refresh
  - Con token válido → continuar

### 1.6 Cambio en Auth Service (backend)

Agregar endpoint `POST /v1/auth/refresh`:
- Recibe refresh token (de cookie HttpOnly)
- Verifica validez + no revocado
- Genera nuevo access + refresh token (rotation)
- Responde con access token en body + nuevo refresh en Set-Cookie

Agregar endpoint `POST /v1/auth/logout`:
- Revoca refresh token
- Limpia cookie

Agregar endpoint `GET /v1/auth/me`:
- Devuelve datos del usuario autenticado (id, email, name, role, tenant)
- Usado por el frontend al cargar la app

## Fase 2 — Wire login real + login app separada

> Login funcional conectado al backend. App aislada.

### 2.1 Crear `apps/login/`

Mini app Next.js aislada:
- Su propio `package.json`, `next.config.ts`, `tailwind.config`
- Comparte el theme system (importa de `apps/web` o paquete compartido)
- Solo tiene: login page, forgot-password page, logo
- Zero layout/sidebar/nav del sistema principal
- Redirige a `apps/web` después de login exitoso

Estructura:
```
apps/login/
  package.json
  next.config.ts
  src/
    app/
      layout.tsx          ← minimal, solo logo + theme
      page.tsx            ← redirect a /login
      login/page.tsx      ← formulario email/password
      forgot-password/page.tsx
    components/
      login-form.tsx      ← extraído de login5.tsx actual
    lib/
      utils.ts
```

### 2.2 Wire login form al backend

- `POST /v1/auth/login` con email + password
- Recibe access_token + refresh_token
- Redirige a `https://{tenant}.sda.app/` (la app principal)
- Maneja errores: 401 (credenciales), 429 (rate limit), 423 (locked)

### 2.3 Limpiar auth routes de `apps/web`

- Eliminar `apps/web/src/app/(auth)/` completamente
- El middleware redirige a login app si no hay sesión

## Fase 3 — WebSocket client

> Frontend conectado al WS Hub. Datos en tiempo real.

### 3.1 WebSocket manager (`apps/web/src/lib/ws/manager.ts`)

```typescript
// Conecta a ws://{backend}/ws con Authorization header
// Auto-reconnect con exponential backoff (1s, 2s, 4s, 8s, max 30s)
// Heartbeat cada 30s
// Parsea mensajes del Hub y los rutea a handlers
```

### 3.2 WebSocket provider (`apps/web/src/lib/ws/provider.tsx`)

React context que:
- Inicializa conexión al montar la app (después de auth)
- Expone: `subscribe(channel, handler)`, `send(message)`
- Invalida TanStack Query cache cuando llegan eventos:
  - `modules` → invalidar `['modules', 'enabled']`
  - `notifications` → invalidar `['notifications']`
  - `chat.messages:{id}` → invalidar `['messages', id]`
  - `ingest.jobs` → invalidar `['ingest', 'jobs']`

### 3.3 Preload data

Cuando el WS conecta, el Hub envía preload:
- Módulos habilitados → hydrate query cache `['modules', 'enabled']`
- Count de notificaciones → hydrate `['notifications', 'count']`
- User preferences → hydrate `['user', 'preferences']`

## Fase 4 — Wire todas las páginas

> Cada página conectada a datos reales.

### 4.1 Chat (`/chat`)

- Listar sesiones: `GET /v1/chat/sessions` via TanStack Query
- Crear sesión: `POST /v1/chat/sessions` mutation
- Mensajes: `GET /v1/chat/sessions/{id}/messages`
- Enviar mensaje: `POST /v1/chat/sessions/{id}/messages`
- RAG streaming: SSE via `api.stream('/v1/rag/generate', { messages, collection })`
- Parsear SSE chunks OpenAI-compatible → renderizar progresivamente
- Sources: mostrar documentos citados debajo de la respuesta

### 4.2 Dashboard (`/dashboard`)

Datos reales:
- Sesiones recientes (desde chat API)
- Notificaciones sin leer (count)
- Módulos activos del tenant
- Quick actions (nueva sesión, upload documento)

### 4.3 Notifications (`/notifications`)

- Listar: `GET /v1/notifications` con TanStack Query
- Mark read: `PATCH /v1/notifications/{id}/read` mutation
- Mark all read: `POST /v1/notifications/read-all`
- Unread count: `GET /v1/notifications/count` (cacheado, invalidado por WS)
- Preferences: `GET/PUT /v1/notifications/preferences`

### 4.4 Settings (`/settings`)

Wire formulario actual a backend:
- `GET /v1/auth/me` → cargar datos del usuario
- `PATCH /v1/auth/me` → guardar cambios (nombre, avatar, etc.)
- Requiere agregar endpoint PATCH en Auth Service

### 4.5 System settings (`/system-settings`)

- Language/timezone: guardar en user preferences (Auth Service o localStorage)
- Theme: ya funciona con localStorage (mantener)

### 4.6 Sidebar dinámico

Reemplazar sidebar hardcodeado:
- Items core fijos (chat, documentos, colecciones)
- Items de módulos desde `useEnabledModules()` + MODULE_REGISTRY
- Admin items solo si role === 'admin'
- Actualización en tiempo real via WebSocket

### 4.7 Search palette

Reemplazar pages hardcodeadas:
- Generar lista dinámica desde core items + módulos habilitados
- Agregar búsqueda de sesiones de chat (query al backend)

El spec dice: "HTTP interno primero, gRPC cuando los contratos se estabilicen."
Los servicios ya se comunican via HTTP + NATS. Los protos son el **contrato
futuro** — los definimos ahora como documentación, no como implementación.

```
proto/
  auth/v1/auth.proto
  chat/v1/chat.proto
  rag/v1/rag.proto
  notification/v1/notification.proto
  platform/v1/platform.proto
  ingest/v1/ingest.proto
```

Cada `.proto` define los mensajes y servicios que mapean a los endpoints REST
actuales. No se genera código Go todavía — se generará cuando se migre a gRPC.

## Fase 8 — CLI (`sda`)

> Herramienta de control del sistema.

### 8.1 Estructura

```
tools/cli/
  main.go
  cmd/
    root.go           ← cobra root command
    tenant.go         ← sda tenant create/list/get/disable/status
    user.go           ← sda user create/delete/reset-password/list
    service.go        ← sda service health/logs/restart
    db.go             ← sda db migrate/backup/restore/seed
    deploy.go         ← sda deploy {service}
    rollback.go       ← sda rollback {service}
    rag.go            ← sda rag collections/ingest/query
    version.go        ← sda version
  go.mod
```

### 8.2 Shared logic (`tools/pkg/`)

```
tools/pkg/
  admin/
    tenant.go         ← crear tenant (DB + Redis + migrations + DNS)
    user.go           ← gestión de usuarios
    deploy.go         ← deploy logic (pull image, health check, swap)
    service.go        ← health, logs, restart
  config/
    config.go         ← lee config del CLI (Platform DB URL, etc.)
```

CLI y MCP comparten esta lógica. El CLI es el wrapper Cobra, el MCP es el
wrapper MCP protocol. Ambos llaman a `tools/pkg/admin/`.

### 8.3 Comandos prioritarios (v1)

Primera versión solo necesita:
- `sda service health` — health check de todos los servicios
- `sda service logs {service}` — tail logs de un container
- `sda tenant list` — listar tenants
- `sda tenant create` — provisionar tenant nuevo
- `sda deploy {service}` — deploy un servicio
- `sda version` — versiones running vs available

## Fase 10 — Observability

> OpenTelemetry en cada servicio Go + Grafana stack.

### 10.1 Instrumentar servicios Go

Agregar a cada servicio:
- `go.opentelemetry.io/otel` — SDK
- `go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp` — HTTP middleware
- `go.opentelemetry.io/contrib/instrumentation/github.com/jackc/pgx/v5/otelpgx` — DB tracing

Middleware chain: `otelhttp → auth → tenant → handler`

Cada request genera un trace con:
- Tenant ID/slug
- User ID
- Service name
- HTTP method + path
- Duration
- Error (si hay)

### 10.2 Log correlation

Configurar slog para incluir trace ID en cada log line:
```
{"time":"...","level":"INFO","msg":"login success","trace_id":"abc123","tenant":"saldivia","user":"enzo"}
```

### 10.3 Grafana stack (Docker Compose)

Agregar a `deploy/docker-compose.dev.yml`:
- **Grafana** (:3001) — dashboards
- **Tempo** — trace storage
- **Prometheus** — metrics
- **Loki** — log aggregation
- **OTel Collector** — recibe telemetry de todos los servicios, rutea a Tempo/Prometheus/Loki

### 10.4 Dashboards pre-configurados

Provisionar via Grafana provisioning:
- Service overview (requests/sec, latency p50/p95/p99, errors by service)
- Per-tenant dashboard (filter by tenant slug)
- RAG quality (latency, token usage)
