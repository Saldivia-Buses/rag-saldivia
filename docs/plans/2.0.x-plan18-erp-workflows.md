# Plan 18 — ERP Workflows: Operaciones Cross-Domain

> **Prerequisito:** Plan 17 (fases 0-12) completado + fixes de auditoría
> **Fecha:** 2026-04-10
> **Estado:** Draft — pendiente aprobacion
> **Version target:** 2.0.7
> **Sprint:** think > plan > execute > review > test > ship (por fase)
> **Servicios afectados:** `services/erp/`, `apps/web/`, `db/tenant/migrations/`
> **Estimacion:** 5 fases, ~6-8 dias
> **Migrations:** 030-034

## Contexto

Plan 17 implementó las entidades base del ERP: catalogos, entidades, stock,
contabilidad, tesoreria, compras, ventas, facturacion, cuentas corrientes,
produccion, RRHH, calidad, mantenimiento, admin. Pero los **workflows
cross-domain** que hacen que un ERP funcione en el dia a dia quedaron afuera:

- No se puede cerrar un ejercicio contable
- No se puede conciliar extractos bancarios
- No se puede anular una factura posteada (y que eso revierta stock, IVA, cta corriente)
- No se puede inspeccionar mercaderia al recibirla
- No se puede registrar un recibo de cobro que genere movimientos en treasury + cta corriente + contabilidad

**Sin estos workflows, Histrix no se puede apagar.** Son operaciones que la
contadora y el tesorero ejecutan todos los dias.

## Principios

1. **Cada workflow es una transaccion atomica.** Si falla cualquier paso, rollback completo.
2. **Los contra-documentos son la reversa.** Nunca UPDATE, siempre INSERT de documento inverso.
3. **Audit trail strict en todo.** StrictLogger para toda operacion financiera.
4. **Los endpoints devuelven todos los IDs generados.** El frontend sabe exactamente que se creo.

## Fase 0 — Cierre de ejercicio contable

**Objetivo:** Cerrar un ejercicio fiscal, generar asiento de cierre (resultado
a patrimonio), y abrir el ejercicio siguiente con asientos de apertura.

### Logica de negocio

1. Validar que el ejercicio existe y esta `open`
2. Validar que no hay asientos en `draft` dentro del periodo
3. Calcular saldos de todas las cuentas de resultado (income + expense)
4. Generar **asiento de cierre**: debita cuentas income, acredita cuentas expense → resultado del ejercicio a cuenta de patrimonio configurable
5. Generar **asiento de apertura** del nuevo ejercicio: copia saldos patrimoniales (asset, liability, equity) al primer dia del nuevo periodo
6. Cambiar status del ejercicio a `closed`
7. Crear el nuevo ejercicio con status `open`

```sql
    COALESCE(SUM(jl.debit), 0) AS total_debit,
    COALESCE(SUM(jl.credit), 0) AS total_credit,
    COALESCE(SUM(jl.debit), 0) - COALESCE(SUM(jl.credit), 0) AS balance
FROM erp_accounts a
LEFT JOIN erp_journal_lines jl ON jl.account_id = a.id AND jl.tenant_id = a.tenant_id
LEFT JOIN erp_journal_entries je ON je.id = jl.entry_id AND je.tenant_id = jl.tenant_id
WHERE a.tenant_id = @tenant_id
    AND a.is_detail = true
    AND je.fiscal_year_id = @fiscal_year_id
    AND je.status = 'posted'
GROUP BY a.id, a.code, a.name, a.account_type
HAVING COALESCE(SUM(jl.debit), 0) - COALESCE(SUM(jl.credit), 0) != 0;
```

```go
func (s *Accounting) CloseFiscalYear(ctx context.Context, slug string, yearID uuid.UUID) (*CloseResult, error) {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    // 1. Validar estado
    year := s.repo.GetFiscalYear(ctx, tx, slug, yearID)  // must be 'open'
    drafts := s.repo.ListDraftEntriesInPeriod(ctx, tx, slug, yearID)
    if len(drafts) > 0 { return error("hay asientos en draft") }

    // 2. Calcular saldos de resultado
    balances := s.repo.GetAccountBalancesForClose(ctx, tx, slug, yearID)

    // 3. Generar asiento de cierre
    closingEntry := s.createClosingEntry(ctx, tx, slug, year, balances)

    // 4. Generar asiento de apertura (nuevo ejercicio)
    patrimonialBalances := filterPatrimonial(balances)  // asset, liability, equity
    newYear := s.repo.CreateFiscalYear(ctx, tx, slug, year.Year+1, ...)
    openingEntry := s.createOpeningEntry(ctx, tx, slug, newYear, patrimonialBalances)

    // 5. Cerrar el ejercicio
    s.repo.CloseFiscalYear(ctx, tx, slug, yearID, closingEntry.ID, openingEntry.ID, userID)

    tx.Commit(ctx)
    s.audit.Strict(ctx, "fiscal_year_close", ...)
    s.nats.Publish("tenant."+slug+".erp_accounting", ...)
    return &CloseResult{ClosingEntryID, OpeningEntryID, NewYearID}
}
```

## Fase 1 — Reconciliacion bancaria

**Objetivo:** Comparar movimientos bancarios (del extracto) contra movimientos
registrados en tesoreria. Marcar matcheados. Detectar diferencias.

### Logica de negocio

1. El usuario sube/carga los movimientos del extracto bancario
2. El sistema compara contra `erp_treasury_movements` de esa cuenta bancaria
3. Auto-match por monto + fecha (tolerancia +-2 dias)
4. El usuario confirma, ajusta, o matchea manualmente lo que no auto-matched
5. Al confirmar la reconciliacion, se marca cada movimiento como conciliado

### Schema

```sql
-- 031_erp_reconciliation.up.sql

-- Treasury-specific trigger: allows reconciliation flag updates on confirmed movements
-- The generic erp_prevent_financial_mutation() blocks ALL updates on confirmed records.
-- Treasury needs to allow setting reconciled/reconciliation_id without changing core fields.
CREATE OR REPLACE FUNCTION erp_prevent_treasury_mutation() RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'DELETE' THEN
        RAISE EXCEPTION 'cannot delete treasury movement with status %', OLD.status;
    END IF;
    -- Allow voiding: confirmed → cancelled/reversed
    IF NEW.status IN ('cancelled', 'reversed') AND OLD.status IN ('confirmed', 'pending') THEN
        RETURN NEW;
    END IF;
    -- Allow reconciliation updates (only reconciled + reconciliation_id may change)
    IF OLD.amount = NEW.amount AND OLD.movement_type = NEW.movement_type
       AND OLD.status = NEW.status AND OLD.date = NEW.date AND OLD.number = NEW.number THEN
        RETURN NEW;
    END IF;
    RAISE EXCEPTION 'cannot modify treasury movement with status %', OLD.status;
END;
$$ LANGUAGE plpgsql;
```

### Auto-match algorithm

```go
func (s *Treasury) AutoMatch(ctx context.Context, slug string, reconID uuid.UUID) (*AutoMatchResult, error) {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    lines := s.repo.ListUnmatchedStatementLines(ctx, tx, slug, reconID)
    movements := s.repo.ListUnreconciledMovements(ctx, tx, slug, recon.BankAccountID, recon.Period)

    matched := 0
    for _, line := range lines {
        for _, mov := range movements {
            if mov.Matched { continue }
            if amountsMatch(line.Amount, mov) && datesClose(line.Date, mov.Date, 2) {
                s.repo.MatchLine(ctx, tx, line.ID, mov.ID)
                mov.Matched = true
                matched++
                break
            }
        }
    }
    tx.Commit(ctx)
    return &AutoMatchResult{Matched: matched, Unmatched: len(lines) - matched}
}
```

## Fase 2 — Anulacion en cascada

**Objetivo:** Anular una factura posteada revirtiendo TODOS los efectos en una
sola transaccion: contra-asiento contable, reversa IVA, reversa stock (si
aplica), reversa cuenta corriente.

### Logica de negocio

La anulacion en cascada es el workflow mas critico del ERP. Una factura posteada
genera:

1. Asiento contable (`erp_journal_entries` + `erp_journal_lines`)
2. Entradas IVA (`erp_tax_entries`)
3. Movimiento cuenta corriente (`erp_account_movements`)
4. Opcionalmente: movimiento stock (`erp_stock_movements` + `erp_stock_levels`)

**Anular = crear contra-documentos para cada uno.** Nunca UPDATE ni DELETE.

### Cascade reversal order (en una TX)

```
1. Crear nota de credito/debito inversa (erp_invoices status='cancelled', reference a original)
2. Crear contra-asiento contable (erp_journal_entries entry_type='reversal')
3. Crear contra-entradas IVA (erp_tax_entries con montos negativos)
4. Crear contra-movimiento cuenta corriente (erp_account_movements inverso)
5. Si la factura tenia stock asociado: crear contra-movimiento stock
6. Marcar factura original como status='cancelled'
7. Marcar asiento original como status='reversed'
```

### Schema

```sql
-- 032_erp_cascade_void.up.sql

-- C1 FIX: Replace generic immutability function to allow controlled state transitions.
-- The original blocks ALL updates on posted/confirmed/paid/reversed.
-- Voiding and reversing are legitimate financial operations that change status
-- but don't modify the financial data itself.
CREATE OR REPLACE FUNCTION erp_prevent_financial_mutation() RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'DELETE' THEN
        RAISE EXCEPTION 'cannot delete financial record with status %', OLD.status;
    END IF;
    -- Allow voiding: posted/paid/confirmed → cancelled
    IF NEW.status = 'cancelled' AND OLD.status IN ('posted', 'paid', 'confirmed') THEN
        RETURN NEW;
    END IF;
    -- Allow reversing: posted/confirmed → reversed
    IF NEW.status = 'reversed' AND OLD.status IN ('posted', 'confirmed') THEN
        RETURN NEW;
    END IF;
    -- Block all other modifications
    RAISE EXCEPTION 'cannot modify financial record with status %', OLD.status;
END;
$$ LANGUAGE plpgsql;

-- C2 FIX: Add 'reversal' to entry_type CHECK constraint.
-- Reversal entries are created by the cascade void workflow.
ALTER TABLE erp_journal_entries DROP CONSTRAINT erp_journal_entries_entry_type_check;
```

```go
func (s *Invoicing) VoidInvoice(ctx context.Context, slug string, invoiceID uuid.UUID, reason string) (*VoidResult, error) {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    inv := s.repo.GetInvoice(ctx, tx, slug, invoiceID)
    if inv.Status != "posted" && inv.Status != "paid" {
        return nil, errors.New("solo se pueden anular facturas posted/paid")
    }

    result := &VoidResult{}

    // 1. Contra-asiento contable
    if inv.JournalEntryID != nil {
        entry := s.repo.GetJournalEntry(ctx, tx, slug, *inv.JournalEntryID)
        reversalEntry := s.accounting.CreateReversalEntry(ctx, tx, slug, entry)
        s.repo.MarkEntryReversed(ctx, tx, entry.ID, reversalEntry.ID)
        result.ReversalEntryID = reversalEntry.ID
    }

    // 2. Contra-entries IVA
    taxEntries := s.repo.ListTaxEntriesByInvoice(ctx, tx, slug, invoiceID)
    for _, te := range taxEntries {
        s.repo.CreateTaxEntry(ctx, tx, negateTaxEntry(te))
    }
    result.TaxEntriesReversed = len(taxEntries)

    // 3. Contra-movimiento cuenta corriente
    acctMovs := s.repo.ListAccountMovementsByInvoice(ctx, tx, slug, invoiceID)
    for _, am := range acctMovs {
        s.accounts.CreateReversalMovement(ctx, tx, slug, am)
    }

    // 4. Contra-movimiento stock (si aplica)
    stockMovs := s.repo.ListStockMovementsByRef(ctx, tx, slug, "invoice", invoiceID)
    for _, sm := range stockMovs {
        s.stock.CreateReversalMovement(ctx, tx, slug, sm)
    }
    result.StockMovementsReversed = len(stockMovs)

    // 5. Marcar factura como cancelled
    s.repo.VoidInvoice(ctx, tx, slug, invoiceID, reason, userID)

    tx.Commit(ctx)
    s.audit.Strict(ctx, "invoice_void_cascade", ...)
    s.nats.Publish("tenant."+slug+".erp_invoicing", ...)
    return result, nil
}
```

## Fase 3 — Recepcion con control de calidad

**Objetivo:** Extender la recepcion de compras (plan 17, fase 5) con inspeccion
de calidad: accept/reject por linea, demerits al proveedor, quarantine.

### Logica de negocio

1. Al recibir mercaderia, el inspector puede marcar cada linea como:
   - `accepted` — ingresa a stock normalmente
   - `rejected` — no ingresa, genera debit note al proveedor
   - `quarantine` — ingresa a deposito de cuarentena, pendiente decision
2. Cada rechazo acumula **demerits** en la entidad proveedor
3. Si se supera un threshold de demerits, el proveedor se marca como `flagged`
4. Items en quarantine se resuelven con accept (mueve a deposito final) o reject

### Schema

```go
func (s *Purchasing) InspectReceipt(ctx context.Context, slug string, receiptID uuid.UUID, inspections []InspectionInput) error {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    // Resolve supplier through receipt → order → supplier_id
    // (erp_purchase_receipts has order_id, not supplier_id)
    receipt := s.repo.GetReceipt(ctx, tx, slug, receiptID)
    order := s.repo.GetPurchaseOrder(ctx, tx, slug, receipt.OrderID)
    supplierID := order.SupplierID

    for _, inp := range inspections {
        // Crear inspeccion
        qc := s.repo.CreateInspection(ctx, tx, ...)

        // Accepted → stock movement al deposito destino
        if inp.AcceptedQty > 0 {
            s.stock.CreateMovement(ctx, tx, slug, StockMovement{
                Type: "in", Quantity: inp.AcceptedQty,
                ReferenceType: "qc_inspection", ReferenceID: qc.ID,
            })
        }

        // Rejected → demerit al proveedor (resolved via order)
        if inp.RejectedQty > 0 {
            s.repo.CreateDemerit(ctx, tx, slug, supplierID, qc.ID, inp.RejectedQty, inp.Notes)
        }

        // Quarantine → registro sin stock movement aun
        if inp.QuarantineQty > 0 {
            s.repo.CreateQuarantine(ctx, tx, slug, qc.ID, inp.ArticleID, inp.QuarantineQty)
        }
    }

    tx.Commit(ctx)
    s.audit.Log(ctx, "qc_inspection", ...)
    return nil
}
```

## Fase 4 — Recibos de cobro y pago

**Objetivo:** Workflow completo de registrar un cobro a un cliente o un pago a
un proveedor. Un recibo genera movimientos en treasury + cuenta corriente +
contabilidad en una sola transaccion.

### Logica de negocio

Un **recibo de cobro** (el cliente nos paga):
1. El usuario selecciona el cliente y las facturas a cobrar
2. Selecciona los medios de pago (efectivo, cheque, transferencia, etc.)
3. Opcionalmente registra retenciones que el cliente nos practica
4. El sistema en 1 TX:
   a. Crea movimiento(s) treasury por cada medio de pago
   b. Crea asiento contable automatico
   c. Actualiza balances de cuenta corriente (imputa pago a facturas)
   d. Registra retenciones si las hay

Un **recibo de pago** (nosotros pagamos a proveedor): misma logica invertida.

### Service layer (pseudocodigo)

```go
type ReceiptInput struct {
    ReceiptType string              // "collection" | "payment"
    EntityID    uuid.UUID
    Date        time.Time
    Payments    []PaymentInput      // medios de pago
    Invoices    []AllocationInput   // facturas a imputar
    Withholdings []WithholdingInput // retenciones
}

func (s *Treasury) CreateReceipt(ctx context.Context, slug string, inp ReceiptInput) (*Receipt, error) {
    tx, _ := s.pool.BeginTx(ctx)
    defer tx.Rollback(ctx)

    // Validar que sum(payments) + sum(withholdings) == sum(allocations)
    // ...

    // 1. Crear recibo
    receipt := s.repo.CreateReceipt(ctx, tx, ...)

    // 2. Por cada medio de pago → crear treasury movement
    for _, pay := range inp.Payments {
        mov := s.repo.CreateTreasuryMovement(ctx, tx, ...)
        s.repo.CreateReceiptPayment(ctx, tx, receipt.ID, pay, mov.ID)
    }

    // 3. Por cada factura → imputar en cuenta corriente
    for _, alloc := range inp.Invoices {
        acctMov := s.accounts.CreatePaymentMovement(ctx, tx, slug, alloc)
        s.repo.CreateReceiptAllocation(ctx, tx, receipt.ID, alloc, acctMov.ID)
        // Actualizar balance de la factura en account_movements
        s.accounts.AllocatePayment(ctx, tx, slug, acctMov.ID, alloc.InvoiceMovementID, alloc.Amount)
    }

    // 4. Retenciones
    for _, wh := range inp.Withholdings {
        whID := s.invoicing.CreateWithholding(ctx, tx, slug, wh)
        s.repo.CreateReceiptWithholding(ctx, tx, receipt.ID, whID)
    }

    // 5. Generar asiento contable automatico
    entry := s.accounting.CreateAutoEntry(ctx, tx, slug, receiptToJournalLines(receipt, inp))
    s.repo.SetReceiptJournalEntry(ctx, tx, receipt.ID, entry.ID)

    tx.Commit(ctx)
    s.audit.Strict(ctx, "receipt_create", ...)
    s.nats.Publish("tenant."+slug+".erp_treasury", ...)
    return receipt, nil
}
```

## Notas de implementacion

1. **Treasury trigger scope:** `erp_prevent_treasury_mutation()` protege los campos
   core (amount, movement_type, status, date, number) pero permite cambios en
   campos secundarios (notes, entity_id, etc.) en movimientos confirmados.
   Aceptable para V1 — si se necesita proteccion total, expandir la lista de
   campos chequeados.

2. **Fiscal year result_account_id:** La migration 030 agrega `result_account_id`
   a `erp_fiscal_years`, pero no hay endpoint para setearlo en un ejercicio
   existente. El endpoint `POST /v1/erp/accounting/fiscal-years` de plan 17
   debe extenderse para aceptar este campo, o agregar un `PATCH` endpoint.

3. **Consistencia de status:** Todos los documentos financieros usan `'cancelled'`
   para anulacion (no `'voided'`). Mantener esta convencion en features futuros.

## Post-plan: que queda reutilizable

- **Patron cascade void** — reutilizable para cualquier documento financiero futuro
- **Patron receipt** — workflow de cobro/pago extensible a nuevos medios de pago
- **Patron QC** — reutilizable para calidad en produccion (plan futuro)
- **Reconciliacion** — extensible a otros tipos de conciliacion (tarjetas, etc.)
