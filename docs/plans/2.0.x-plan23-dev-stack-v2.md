## Problema

SDA va a produccion con 14+ microservicios y un solo developer. El stack actual
tiene agujeros criticos para operar a esta escala:

| Area | Estado actual | Riesgo |
|------|---------------|--------|
| CI/CD | Jobs paralelos sin gates — si el build pasa, todo pasa | Un test roto no bloquea deploy |
| Deploy | `docker compose up --build` manual | Si falla, no hay rollback |
| Code review | Agents manuales (Enzo los invoca cuando se acuerda) | PRs sin review pasan |
| Alerting | Prometheus alertas → Alertmanager con receivers vacios | Nadie se entera de nada |
| Error triage | No existe | Errores se descubren cuando Enzo mira Grafana |
| Feature flags | Tabla y API existen pero no se usan | Todo es all-or-nothing |
| Version determinism | Imagenes defaultean a `latest`, binarios dicen `dev` | No se sabe que esta corriendo |
| Documentacion | Manual, se desactualiza con cada cambio | CLAUDE.md, bible divergen del codigo |

### Lo que ya funciona (no romper)

- Health checks en todos los servicios (`/health` + healthcheck en compose)
- `restart: unless-stopped` + autoheal para containers unhealthy
- Graceful shutdown (30s, SIGTERM, drain)
- OTel + Prometheus + Grafana + Loki (metricas + traces + logs)
- Prometheus alerts definidos (ServiceDown, HighErrorRate, HighLatency, etc.)
- Preflight script con 13 checks (`deploy/scripts/preflight.sh`)
- `deploy_log` table con tracking de versiones y status rollback
- `feature_flags` table con soporte per-tenant + `ListFeatureFlags` + `ToggleFeatureFlag`
- Segmentacion de red en Docker (frontend/backend/data/lan)
- Docker secrets para credenciales
- Docker socket proxy (Tecnativa) para Traefik — limita API surface

### Lo que se absorbe de Plan 14

- **(A) Config portable:** 5 variables maestras en `.env` → ya implementado
- **(B) Dependency ordering prod:** `depends_on: service_healthy` → absorber
- **(C) Alertmanager activation:** receivers comentados → activar aqui
- DB/NATS/Redis tracing → fuera de scope (es instrumentacion, no dev stack)

### Lo que se absorbe de Plan 17.5

- **Version determinism:** ldflags en Dockerfiles → absorber
- **`/v1/info` funcional:** ya funciona via `pkg/build` pero ldflags no se inyectan → fix
- **`make versions` funcional:** placeholder muerto → implementar
- **Image tags en compose:** `${SERVICE_VERSION:-latest}` → tag por SHA

## Decisiones de seguridad (resueltas en auditoria)

Estas decisiones aplican a todo el plan. Se documentan aqui para no repetirlas
en cada fase.

### DS1. Runner isolation (Security C2)

Los self-hosted runners ejecutan en la workstation de produccion. Un PR
malicioso podria ejecutar codigo arbitrario en prod.

**Mitigacion:**
- CI (`ci.yml`, `ai-review.yml`) corre SOLO en `ubuntu-latest` (GitHub-hosted). Nunca en self-hosted.
- Deploy (`deploy.yml`) corre en self-hosted SOLO con `workflow_dispatch` o tag push (requiere write access).
- Deploy a prod usa **GitHub Environments** con `required_reviewers: 1`.
- El runner corre como usuario dedicado `sda-runner` sin SSH keys ni GPG.
- Steps que tocan Docker usan container jobs cuando es posible.

### DS2. Script injection prevention (Security C1 + Gateway H4)

Ningun script usa `source` para cargar archivos generados. Todos los scripts
que leen archivos de version usan parsers seguros con validacion regex.

### DS3. Webhook authentication (Security H1 + Gateway C4)

El webhook de Alertmanager → notification es interno (Docker network).
- Alertmanager envia un `Authorization: Bearer ${ALERTMANAGER_WEBHOOK_TOKEN}`.
- El endpoint valida el token contra un Docker secret.
- El endpoint NO se expone via Traefik (ruta interna only).

### DS4. HealthWatch authentication (Security H4 + Gateway M2)

Todos los endpoints de HealthWatch requieren `requirePlatformAdmin` JWT.
El daily triage cron usa un service account JWT (short-lived, emitido por auth).

### DS5. Docker socket via proxy (Security M2 + Gateway C3)

HealthWatch NUNCA monta `/var/run/docker.sock`. Conecta via el
`docker-socket-proxy` existente en `http://docker-socket-proxy:2375`.

### DS6. AI prompt injection defense (Security H3 + Gateway H3)

Los AI review passes:
- NO interpolan diff content ni AI output en el prompt/shell via `${{ }}`.
- Usan `claude-code-action` que lee el PR diff nativamente.
- Requieren output estructurado JSON con severity scores.
- System prompt incluye instruccion anti-injection.
- AI output se escribe a archivo temporal antes de parsear con jq (nunca shell interpolation).
- Health data para triage se pasa via archivo, no interpolado en prompt string.
- Heredocs en GITHUB_OUTPUT usan delimitador random (no `EOF` fijo — evita injection).
- `@claude` mentions restringidas a `author_association == 'MEMBER'`.
- API key con spending limit mensual.

### DS7. Feature flags tenant isolation (Security H2 + Gateway H6)

`/v1/flags/evaluate` extrae `tenant_id` y `user_id` del JWT claims.
No acepta query params para identidad. El response solo incluye valores
booleanos, no metadata de flags.

Reescribir `.github/workflows/deploy.yml`:

```yaml
name: Deploy SDA

on:
  workflow_dispatch:
    inputs:
      environment: { type: choice, options: [dev, prod], default: dev }
  push:
    tags: ['v*']

jobs:
  # Fase 0: Discover services (dynamic matrix — [L5])
  discover:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.find.outputs.matrix }}
    steps:
      - uses: actions/checkout@v4
      - id: find
        run: |
          SERVICES=$(ls services/*/Dockerfile 2>/dev/null \
            | grep -v scaffold | grep -v extractor \
            | sed 's|services/||;s|/Dockerfile||' \
            | jq -R -s 'split("\n") | map(select(. != ""))')
          echo "matrix=$SERVICES" >> $GITHUB_OUTPUT

  preflight:
    runs-on: [self-hosted, workstation-1]
    steps:
      - uses: actions/checkout@v4
      - run: bash deploy/scripts/preflight.sh

  # Fase 2: Build + push images (GitHub-hosted — [DS1])

    needs: [preflight, discover]
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: ${{ fromJson(needs.discover.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - name: Build + push to GHCR
        run: |
          SERVICE="${{ matrix.service }}"
          VERSION=$(cat "services/${SERVICE}/VERSION")
          SHA=${{ github.sha }}
          docker build --build-arg VERSION=$VERSION \
            --build-arg GIT_SHA=$SHA \
            --build-arg BUILD_TIME=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
            -t "ghcr.io/camionerou/sda-${SERVICE}:${SHA}" \

    needs: [build, discover]

      - uses: actions/checkout@v4

          SHA=${{ github.sha }}

            echo "${KEY}=${SHA}"
          done > deploy/.env.deploy
      - name: Upload deploy env
        uses: actions/upload-artifact@v4
        with:
          name: deploy-env
          path: deploy/.env.deploy

    continue-on-error: true  # dev-pc may not be set up yet [L2]

      - uses: actions/download-artifact@v4
        with: { name: deploy-env, path: deploy/ }
      - name: Deploy with SHA-pinned images
        run: |
          docker compose --env-file deploy/.env.deploy \
            -f deploy/docker-compose.dev.yml up -d --pull always
      - name: Health check
        run: bash deploy/scripts/health-check.sh --env dev --timeout 120

  deploy-prod:
    needs: [deploy-dev, generate-deploy-env]
    if: inputs.environment == 'prod' || startsWith(github.ref, 'refs/tags/')
    environment:
      name: production
      url: https://${{ vars.SDA_DOMAIN }}
    runs-on: [self-hosted, workstation-1]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
        with: { name: deploy-env, path: deploy/ }
      - name: Save current versions (for rollback)
        run: |
          ROLLBACK_FILE="${RUNNER_TEMP}/sda-rollback-$(date +%s).env"
          bash deploy/scripts/save-versions.sh > "$ROLLBACK_FILE" || {
            echo "::error::Failed to capture current versions — aborting deploy"
            exit 1
          }
          chmod 600 "$ROLLBACK_FILE"
          echo "ROLLBACK_FILE=$ROLLBACK_FILE" >> $GITHUB_ENV
      - name: Deploy with SHA-pinned images
        run: |
          docker compose --env-file deploy/.env.deploy \
            -f deploy/docker-compose.prod.yml up -d --pull always
      # Fase 5: Smoke test (same job — needs $ROLLBACK_FILE from GITHUB_ENV)

  # Fase 5+6 merged into deploy-prod (ROLLBACK_FILE is job-scoped via GITHUB_ENV)

  # because $ROLLBACK_FILE set via GITHUB_ENV is only available within the same job.

  # Fase 7: Record + notify
  release:

    runs-on: [self-hosted, workstation-1]

            --version "${{ github.ref_name }}" \
            --sha "${{ github.sha }}" \
```

**2.5 — Scripts de circuit-breaker seguros**

`deploy/scripts/save-versions.sh`:
```bash
#!/bin/bash
set -euo pipefail
# Output: KEY=value lines, one per service, validated format
for entry in auth=8001 ws=8002 chat=8003 ...; do
  svc="${entry%%=*}"; port="${entry##*=}"
  ver=$(curl -sf "http://localhost:${port}/v1/info" 2>/dev/null | jq -re '.version' || echo "unknown")
  key="${svc^^}_VERSION"
  # Validate version format: semver or SHA
  if [[ "$ver" =~ ^[0-9]+\.[0-9]+\.[0-9]+$ ]] || [[ "$ver" =~ ^[a-f0-9]{7,40}$ ]]; then
    echo "${key}=${ver}"
  else
    echo "WARN: ${svc} returned invalid version: ${ver}" >&2
  fi
done
```

`deploy/scripts/rollback.sh` — **NO usa `source`** ([DS2]):
```bash
#!/bin/bash
set -euo pipefail
VERSIONS_FILE="$1"
[ -f "$VERSIONS_FILE" ] || { echo "ERROR: versions file not found: $VERSIONS_FILE"; exit 1; }

# Safe parser: only accept KEY=value lines matching expected format
declare -A VERSIONS
while IFS='=' read -r key value; do
  [[ "$key" =~ ^[A-Z_]+_VERSION$ ]] || continue
  [[ "$value" =~ ^([0-9]+\.[0-9]+\.[0-9]+|[a-f0-9]{7,40})$ ]] || {
    echo "ERROR: invalid version for $key: $value"; exit 1
  }
  VERSIONS[$key]="$value"
done < "$VERSIONS_FILE"

# Export validated versions and re-deploy
for key in "${!VERSIONS[@]}"; do export "$key=${VERSIONS[$key]}"; done
docker compose -f deploy/docker-compose.prod.yml up -d --pull always
echo "Rollback complete. Versions restored from $VERSIONS_FILE"
```

**Test:** rollback con archivo corrupto → falla con error, no ejecuta nada

## Invariantes

Estas reglas no pueden romperse durante la implementacion:

1. **Tenant isolation** — HealthWatch NO accede a datos de tenant. Solo infra/platform data. Feature flags evaluate extrae tenant del JWT, no de query params.
2. **Un pipeline** — features y bugs usan el MISMO deploy workflow. No hay atajos.
3. **Circuit-breaker es automatico** — no requiere intervencion humana para rollback. Scripts no usan `source`, validan input con regex.
4. **AI reviews son gates, no sugerencias** — critical/high findings bloquean merge. Output en JSON estructurado, scoring via jq.
5. **HealthWatch es read-only** — solo lee Prometheus/Docker proxy/services. Nunca modifica estado. Nunca monta Docker socket directo.
6. **Feature flags default off** — un flag nuevo siempre nace `enabled=false`. Rollout es deterministic hash, no random.
7. **Daily triage no requiere accion humana para dispararse** — cron, no manual.
8. **Runner isolation** — CI y AI review corren en GitHub-hosted. Solo deploy usa self-hosted, con environment protection.
9. **Webhook auth** — todo endpoint interno (alertmanager, healthwatch) requiere token o JWT. Nada expuesto sin auth.
10. **Scrubbed output** — HealthWatch summary y triage reports nunca incluyen raw errors, stack traces, IPs, SQL, o credentials.
