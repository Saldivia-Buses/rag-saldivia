Investigacion de 4 agentes encontro:

1. **~20 piezas reutilizables en `_archive/`** — notificaciones SSE con Redis Pub/Sub,

   pub/sub built-in, 4KB/conexion, 62k conexiones concurrentes).

4. **SQLite FTS5** para busqueda full-text sin dependencias externas.

extraidos. Las messaging actions usan `authAction` desde el principio.

## Stack del messaging

| Componente | Tecnologia | Nota |
|---|---|---|
| Real-time transport | **Bun native WebSocket** | Sidecar en :3001, pub/sub built-in |
| Fan-out | Redis Pub/Sub | Ya tenemos Redis |
| Presencia | Redis TTL keys | `presence:user:{id}` con 30s TTL |
| Typing indicators | Redis Pub/Sub only | Ephemeral, no DB |
| Persistencia | SQLite (Drizzle) | Nuevas tablas en schema/messaging.ts |
| Busqueda | SQLite FTS5 | Setup via raw SQL en init.ts |
| Notificaciones | WebSocket + BullMQ | WS para real-time, BullMQ para async |
| UI | React + shadcn/ui | Nuevos componentes en `components/messaging/` |

### Arquitectura WebSocket (sidecar)

```
Browser:
  HTTP/pages → :3000 (Next.js, sin modificar)
  WebSocket  → :3001 (Bun WS sidecar)

Desarrollo:
  Terminal 1: bun run dev          → Next.js :3000
  Terminal 2: bun run ws           → WebSocket :3001

Produccion:
  reverse proxy (nginx/caddy) → :443
    /           → :3000 (Next.js)
    /ws         → :3001 (WebSocket)

Comunicacion interna:
  WS Server ←→ Redis Pub/Sub ←→ BullMQ Workers
```

Ventajas del sidecar:
- Next.js no se toca. HMR, Turbopack, todo funciona igual.
- WS server es Bun puro — simple, rapido, sin magia.
- Si WS se cae, Next.js sigue funcionando (degradacion graceful).
- Escalable: multiples instancias WS detras del mismo Redis.

**FTS5 (setup en `fts5.ts`, ejecutado desde `init.ts`):**

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS msg_messages_fts
  USING fts5(content, content=msg_messages, content_rowid=rowid);

-- Triggers de sync
CREATE TRIGGER IF NOT EXISTS msg_fts_insert AFTER INSERT ON msg_messages BEGIN
  INSERT INTO msg_messages_fts(rowid, content) VALUES (new.rowid, new.content);
END;

CREATE TRIGGER IF NOT EXISTS msg_fts_delete AFTER DELETE ON msg_messages BEGIN
  INSERT INTO msg_messages_fts(msg_messages_fts, rowid, content)
    VALUES('delete', old.rowid, old.content);
END;

CREATE TRIGGER IF NOT EXISTS msg_fts_update AFTER UPDATE ON msg_messages BEGIN
  INSERT INTO msg_messages_fts(msg_messages_fts, rowid, content)
    VALUES('delete', old.rowid, old.content);
  INSERT INTO msg_messages_fts(rowid, content) VALUES (new.rowid, new.content);
END;
```

## Protocolo WebSocket

```typescript
// Client -> Server
type ClientMessage =
  | { type: "auth"; token: string }                      // primer mensaje obligatorio
  | { type: "subscribe"; channelId: string }
  | { type: "unsubscribe"; channelId: string }
  | { type: "typing_start"; channelId: string }
  | { type: "typing_stop"; channelId: string }
  | { type: "presence"; status: "online" | "away" }
  | { type: "sync"; channels: { id: string; lastMessageAt: number }[] } // reconexion
  | { type: "ping" }

// Server -> Client
type ServerMessage =
  | { type: "auth_ok"; userId: number }
  | { type: "auth_error"; reason: string }
  | { type: "message_new"; message: Message }
  | { type: "message_updated"; message: Message }
  | { type: "message_deleted"; messageId: string; channelId: string }
  | { type: "reaction_added"; messageId: string; userId: number; emoji: string }
  | { type: "reaction_removed"; messageId: string; userId: number; emoji: string }
  | { type: "typing"; channelId: string; userId: number; displayName: string }
  | { type: "presence_update"; userId: number; status: "online" | "away" | "offline" }
  | { type: "channel_updated"; channel: Channel }
  | { type: "member_joined"; channelId: string; userId: number }
  | { type: "member_left"; channelId: string; userId: number }
  | { type: "unread_update"; channelId: string; count: number }
  | { type: "sync_response"; channelId: string; messages: Message[] } // catch-up
  | { type: "pong" }
```

**Reconexion:** Al reconectar, el cliente envia `sync` con el timestamp del ultimo
mensaje por canal. El server responde con `sync_response` para los mensajes faltantes.
Si hay mas de 100 mensajes faltantes, responde con los ultimos 100 y marca el canal
como "jump to latest".

## Channel RBAC

Cada miembro de un canal tiene un rol (`channelMembers.role`):

| Permiso | owner | admin | member |
|---|---|---|---|
| Enviar mensajes | ✅ | ✅ | ✅ |
| Reaccionar | ✅ | ✅ | ✅ |
| Editar mensaje propio | ✅ | ✅ | ✅ |
| Borrar mensaje propio | ✅ | ✅ | ✅ |
| Borrar mensaje de otros | ✅ | ✅ | ❌ |
| Pin/unpin mensajes | ✅ | ✅ | ❌ |
| Invitar miembros | ✅ | ✅ | ❌ |
| Kickear miembros | ✅ | ✅ | ❌ |
| Editar canal (nombre, topic, desc) | ✅ | ✅ | ❌ |
| Promover a admin | ✅ | ❌ | ❌ |
| Archivar/borrar canal | ✅ | ❌ | ❌ |

**Crear canales:** cualquier usuario autenticado (publicos y privados).
**Crear DMs:** cualquier usuario autenticado.
**Admins del sistema** (`users.role === "admin"`) pueden moderar cualquier canal.

## Rate limiting

- **Mensajes:** max 30/minuto por usuario (previene spam)
- **Reacciones:** max 60/minuto por usuario
- **Creacion de canales:** max 5/hora por usuario
- Implementado via Redis counter con TTL (patron existente en `rate-limits.ts`)
