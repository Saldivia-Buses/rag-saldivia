# Plan 03 — Feedback Service: sistema nervioso central de la plataforma

> **Prerequisito:** Plan 02 (Full Wiring) completado — los servicios productores
> de eventos deben poder publicar a NATS. El Notification Service debe estar
> funcional (ya lo esta) para recibir alertas.
> **Sprint:** think > plan > execute > review > test > ship
> **Servicios afectados:** feedback (nuevo), notification (receptor de alertas),
> platform (migracion nueva en platform DB)

---

## Contexto

### Que problema resuelve

La plataforma no tiene forma de saber si esta funcionando bien. Hoy cada
servicio opera como isla: si la calidad de RAG baja, si un tenant tiene
errores recurrentes, si un modulo no se usa, nadie se entera hasta que un
usuario se queja. No hay metricas de calidad de IA, no hay deteccion de
anomalias, no hay health score por tenant.

### Por que ahora

El Feedback Service es prerequisito para el data flywheel de IA — el ciclo
donde el feedback del usuario mejora la calidad del sistema. Sin el, la IA
no mejora con el uso. Ademas, cuando se conecten tenants reales (despues del
Full Wiring), necesitamos visibilidad inmediata de que cada tenant funciona
bien.

### Que NO es

- No es un servicio CRUD — no recibe datos por REST
- No es Grafana — no reemplaza observability, la complementa
- No es un data warehouse — retiene datos granulares 90 dias, agregados 2 anos

### Filosofia

**Consumer puro + read API.** Todos los datos entran por NATS JetStream.
Los endpoints REST son solo para consultar. El servicio es SMART: detecta
patrones, anomalias, y degradacion. Alerta proactivamente via Notification
Service cuando algo esta mal.

## Arquitectura

```
                          NATS JetStream
                          stream: FEEDBACK
                          subjects: tenant.*.feedback.>
                          
Chat Service ──feedback.response_quality──►┐
Agent Service ──feedback.agent_quality────►│
DocAI Service ──feedback.extraction───────►│
Vision Service ──feedback.detection───────►├──► Feedback Service
Auth Service ──feedback.security──────────►│    (Go, port :8008)
Frontend ──feedback.usage─────────────────►│
Frontend ──feedback.nps───────────────────►│
Frontend ──feedback.error_report──────────►│
Frontend ──feedback.feature_request───────►│
Services ──feedback.performance───────────►┘
                                               │
                              ┌────────────────┼────────────────┐
                              ▼                ▼                ▼
                        Tenant DB         Platform DB      NATS (alertas)
                        (granular)        (agregados)       │
                                                            ▼
                                                     Notification Service
                                                     (alerta al admin)
```

### Flujo de datos

1. **Ingestion:** Consumer JetStream procesa eventos en batch (10 msgs/fetch)
2. **Storage:** Datos granulares en tenant DB (tabla `feedback_events`)
3. **Aggregation:** Cron cada hora agrega metricas en `feedback_metrics` (platform DB)
4. **Detection:** Despues de cada agregacion, el alerter revisa thresholds
5. **Alert:** Si un threshold se cruza, publica alerta a Notification Service via NATS

### Stream

```

```

### Extraccion de tenant

Mismo patron que Notification Service: el tenant slug se extrae del subject
NATS (`tenant.{slug}.feedback.*`), nunca del payload JSON. El payload es
datos del usuario (no confiable para routing), el subject es puesto por el
servicio productor (confiable).

## DB Schema

### Tenant DB — tabla `feedback_events` (granular, per-tenant)

```sql
-- UP (services/feedback/db/migrations/001_init.up.sql)

-- Feedback events (all types in one table, discriminated by category)
CREATE TABLE IF NOT EXISTS feedback_events (
    id          TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    category    TEXT NOT NULL,                                -- 'response_quality', 'agent_quality', 'extraction',
                                                             -- 'detection', 'error_report', 'feature_request',
                                                             -- 'nps', 'usage', 'performance', 'security'
    module      TEXT NOT NULL,                                -- 'chat', 'agent', 'docai', 'vision', 'auth', etc.
    user_id     TEXT,                                         -- NULL para performance/security sin user
    score       INTEGER,                                      -- 1-5 para quality, 0-10 para NPS, NULL si no aplica
    thumbs      TEXT,                                         -- 'up', 'down', NULL
    severity    TEXT,                                         -- 'info', 'warning', 'error', 'critical' (error_report, security)
    status      TEXT NOT NULL DEFAULT 'open',                 -- 'open', 'acknowledged', 'resolved', 'dismissed'
    context     JSONB NOT NULL DEFAULT '{}',                  -- payload completo (schema varia por category)
    comment     TEXT,                                         -- comentario libre del usuario
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Queries por category (mas comun)
CREATE INDEX IF NOT EXISTS idx_feedback_events_category
    ON feedback_events(category, created_at DESC);

-- Queries por modulo
CREATE INDEX IF NOT EXISTS idx_feedback_events_module
    ON feedback_events(module, created_at DESC);

-- Queries por usuario
CREATE INDEX IF NOT EXISTS idx_feedback_events_user
    ON feedback_events(user_id, created_at DESC)
    WHERE user_id IS NOT NULL;

-- Error reports sin resolver
CREATE INDEX IF NOT EXISTS idx_feedback_events_open_errors
    ON feedback_events(status, created_at DESC)
    WHERE category = 'error_report' AND status = 'open';

-- Feature requests pendientes
CREATE INDEX IF NOT EXISTS idx_feedback_events_open_features
    ON feedback_events(status, created_at DESC)
    WHERE category = 'feature_request' AND status = 'open';

-- Retention: datos granulares se purgan a los 90 dias.
-- La agregacion en platform DB preserva las metricas indefinidamente.
-- Purge via cron: DELETE FROM feedback_events WHERE created_at < now() - interval '90 days';
```

### Platform DB — tablas de metricas agregadas (cross-tenant)

```sql
-- UP (services/platform/db/migrations/002_feedback_metrics.up.sql)

-- Hourly aggregated metrics per tenant+module+category
CREATE TABLE IF NOT EXISTS feedback_metrics (
    tenant_id       TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    module          TEXT NOT NULL,
    category        TEXT NOT NULL,                         -- matches feedback_events.category
    period          TIMESTAMPTZ NOT NULL,                  -- truncated to hour
    total_events    INTEGER NOT NULL DEFAULT 0,
    positive        INTEGER NOT NULL DEFAULT 0,            -- thumbs up / confirmed / success
    negative        INTEGER NOT NULL DEFAULT 0,            -- thumbs down / corrected / failure
    avg_score       REAL,                                  -- average of score where not null
    p50_latency_ms  REAL,                                  -- performance only
    p95_latency_ms  REAL,                                  -- performance only
    p99_latency_ms  REAL,                                  -- performance only
    error_count     INTEGER NOT NULL DEFAULT 0,            -- error_report count
    metadata        JSONB NOT NULL DEFAULT '{}',           -- category-specific aggregates
    PRIMARY KEY (tenant_id, module, category, period)
);

-- Query by tenant (tenant health dashboard)
CREATE INDEX IF NOT EXISTS idx_feedback_metrics_tenant
    ON feedback_metrics(tenant_id, period DESC);

-- Query by category cross-tenant (platform dashboard)
CREATE INDEX IF NOT EXISTS idx_feedback_metrics_category
    ON feedback_metrics(category, period DESC);

-- Tenant health scores (composite, updated hourly)
CREATE TABLE IF NOT EXISTS tenant_health_scores (
    tenant_id           TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    period              TIMESTAMPTZ NOT NULL,              -- truncated to hour
    overall_score       REAL NOT NULL,                     -- 0-100
    ai_quality_score    REAL NOT NULL DEFAULT 100,         -- 0-100
    error_rate_score    REAL NOT NULL DEFAULT 100,         -- 0-100
    usage_score         REAL NOT NULL DEFAULT 100,         -- 0-100 (activity level)
    performance_score   REAL NOT NULL DEFAULT 100,         -- 0-100
    security_score      REAL NOT NULL DEFAULT 100,         -- 0-100
    nps_score           REAL,                              -- -100 to 100 (classic NPS)
    details             JSONB NOT NULL DEFAULT '{}',       -- breakdown por modulo
    PRIMARY KEY (tenant_id, period)
);

CREATE INDEX IF NOT EXISTS idx_tenant_health_scores_period
    ON tenant_health_scores(period DESC);

-- Active alerts (cross-tenant, managed by feedback service)
CREATE TABLE IF NOT EXISTS feedback_alerts (
    id              TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    tenant_id       TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    alert_type      TEXT NOT NULL,                         -- 'quality_drop', 'error_spike', 'latency_spike',
                                                          -- 'security_anomaly', 'nps_drop', 'inactive_tenant'
    severity        TEXT NOT NULL DEFAULT 'warning',       -- 'info', 'warning', 'critical'
    module          TEXT,                                   -- NULL if system-wide
    title           TEXT NOT NULL,
    description     TEXT NOT NULL,
    threshold       TEXT,                                   -- e.g., "negative_rate > 30%"
    current_value   TEXT,                                   -- e.g., "42%"
    status          TEXT NOT NULL DEFAULT 'active',        -- 'active', 'acknowledged', 'resolved', 'auto_resolved'
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    resolved_at     TIMESTAMPTZ,
    acknowledged_by TEXT
);

CREATE INDEX IF NOT EXISTS idx_feedback_alerts_active
    ON feedback_alerts(status, created_at DESC)
    WHERE status = 'active';

CREATE INDEX IF NOT EXISTS idx_feedback_alerts_tenant
    ON feedback_alerts(tenant_id, created_at DESC);
```

## Health Score Algorithm

### Composite score (0-100, por tenant, actualizado cada hora)

```
overall_score = (
    ai_quality_score    * 0.30 +    -- 30% peso
    error_rate_score    * 0.25 +    -- 25% peso
    performance_score   * 0.20 +    -- 20% peso
    security_score      * 0.15 +    -- 15% peso
    usage_score         * 0.10      -- 10% peso
)
```

### Calculo de cada sub-score

**AI Quality Score (0-100):**
```
Si no hay feedback de IA en la ultima hora: mantener score anterior.
Si hay:
  positive_rate = positive / (positive + negative)
  ai_quality_score = positive_rate * 100
  
  Si total feedback < 5 en la hora: atenuar cambio (moving average con ventana 24h)
```

**Error Rate Score (0-100):**
```
errors_per_hour = count(error_report WHERE severity IN ('error', 'critical'))
  0 errors    → 100
  1-2 errors  → 90
  3-5 errors  → 70
  6-10 errors → 50
  11-20       → 30
  20+         → 10
  
Si hay algun critical → cap at 30
```

**Performance Score (0-100):**
```
Si no hay datos de performance: 100 (asume OK).
Si hay:
  p95 < 200ms  → 100
  p95 < 500ms  → 85
  p95 < 1000ms → 70
  p95 < 2000ms → 50
  p95 < 5000ms → 30
  p95 >= 5000ms → 10
```

**Security Score (0-100):**
```
Ventana: ultima hora.
  0 eventos security warning+ → 100
  1-3 warnings                → 85
  4-10 warnings               → 70
  Cualquier critical          → 30
  Multiple criticals          → 10
```

**Usage Score (0-100):**
```
Mide actividad relativa al promedio historico del tenant.
  Si no hay historial (tenant nuevo): 50 (neutral).
  Si hay:
    current_usage = count(usage events en la ultima hora)
    avg_usage = average(usage events por hora, ultimos 7 dias)
    ratio = current_usage / max(avg_usage, 1)
    
    ratio >= 0.5  → 100 (actividad normal o superior)
    ratio >= 0.2  → 70  (baja pero presente)
    ratio >= 0.05 → 40  (muy baja)
    ratio < 0.05  → 20  (posible problema o inactividad)
```

### NPS Score (-100 a +100)

NPS clasico:
```
promoters = count(score >= 9) / total
passives  = count(score >= 7 AND score < 9) / total
detractors = count(score < 7) / total
nps = (promoters - detractors) * 100
```

Se calcula con ventana rolling de 30 dias (no por hora). Se actualiza cada
vez que llega un evento NPS.

## Alert Thresholds y Notification Triggers

### Umbrales por defecto

| Alert Type | Condicion | Severity | Mensaje ejemplo |
|---|---|---|---|
| `quality_drop` | AI quality score baja >15 puntos en 24h | warning | "Calidad de IA de tenant 'saldivia' bajo de 85 a 68 en modulo 'chat'" |
| `quality_critical` | AI quality score < 50 | critical | "Calidad critica en tenant 'saldivia': 42% respuestas negativas" |
| `error_spike` | >10 errores/hora (vs promedio <3) | warning | "Spike de errores en tenant 'saldivia': 15 errores en la ultima hora" |
| `error_critical` | Cualquier error con severity 'critical' | critical | "Error critico en tenant 'saldivia': [error_message]" |
| `latency_spike` | p95 > 2000ms por 2 horas consecutivas | warning | "Latencia alta sostenida en tenant 'saldivia': p95=2.3s" |
| `security_anomaly` | >5 eventos de seguridad warning+ en 1h | warning | "Actividad de seguridad inusual en tenant 'saldivia'" |
| `security_critical` | Cualquier evento severity 'critical' | critical | "Alerta de seguridad critica en tenant 'saldivia'" |
| `nps_drop` | NPS baja >20 puntos en 7 dias | warning | "NPS de tenant 'saldivia' bajo de 60 a 35" |
| `inactive_tenant` | 0 eventos usage en 48h (tenant activo) | info | "Tenant 'saldivia' sin actividad en 48 horas" |

### Mecanismo de alerta

1. El alerter corre despues de cada agregacion (cada hora)
2. Compara metricas actuales vs thresholds
3. Si se cruza un threshold:
   a. Crea registro en `feedback_alerts` (platform DB)
   b. Publica evento a NATS via `natspub.Notify()` con subject `tenant.{slug}.notify.feedback.alert`
   c. El Notification Service lo consume y crea notificacion in-app para platform admins
4. Auto-resolucion: si la condicion deja de cumplirse por 2 horas, status → `auto_resolved`

### Payload de alerta a Notification Service

```json
{
  "user_id": "platform-admin-broadcast",
  "type": "feedback.alert",
  "title": "Calidad critica en tenant 'saldivia'",
  "body": "42% de respuestas negativas en modulo 'chat' (ultima hora)",
  "data": {
    "alert_id": "uuid",
    "alert_type": "quality_critical",
    "tenant_id": "saldivia",
    "severity": "critical",
    "module": "chat",
    "current_value": "42%",
    "threshold": "< 50%"
  },
  "channel": "both"
}
```

Nota: `user_id: "platform-admin-broadcast"` es un ID especial que el
Notification Service debe resolver a todos los platform admins. Esto requiere
un cambio menor en notification consumer (Fase 2).

### Fase 2: Platform DB + aggregator

**Servicio:** `feedback`, `platform`
**Archivos:**
- `services/platform/db/migrations/002_feedback_metrics.up.sql`
- `services/platform/db/migrations/002_feedback_metrics.down.sql`

**Cambios:**

1. Migracion platform DB: crea tablas `feedback_metrics`,
   `tenant_health_scores`, `feedback_alerts`.

2. `internal/service/aggregator.go`: goroutine que corre cada hora
   (configurable via `AGGREGATION_INTERVAL`):
   - Consulta `feedback_events` del tenant DB agrupado por module+category
     para la ultima hora
   - Calcula totales, positive/negative, avg_score, percentiles de latencia
   - Upsert en `feedback_metrics` (platform DB)
   - Calcula health scores por tenant (formula descrita arriba)
   - Upsert en `tenant_health_scores`
   - Purga eventos granulares mayores a `RETENTION_DAYS`

3. sqlc queries platform DB: `UpsertFeedbackMetric`, `UpsertHealthScore`,
   `GetLatestHealthScore`, `ListTenantHealthScores`.

4. El aggregator necesita listar tenants — usa una query directa a
   `tenants` en platform DB (SELECT id, slug FROM tenants WHERE enabled = true).

## Decisiones de diseno explicadas

### Por que una sola tabla `feedback_events` y no una por tipo?

Simplicidad. Son 10 tipos de eventos con estructura similar (id, category,
module, user, score, context JSONB). El discriminador es `category`. Si en
el futuro un tipo necesita su propia tabla (ej: usage events que son altisimo
volumen), se puede migrar sin cambiar la interfaz.

### Por que el aggregator corre cada hora y no en tiempo real?

1. El health score es un indicador estrategico, no operativo. No necesita
   ser al segundo.
2. Agregar en tiempo real en platform DB por cada evento NATS generaria
   write contention innecesaria.
3. Una hora da suficiente ventana para detectar problemas antes de que el
   tenant los note.

### Por que no usar Prometheus directamente?

Prometheus es para metricas de infraestructura (CPU, memoria, requests/sec).
El Feedback Service trabaja con metricas de NEGOCIO (calidad de IA, NPS,
feature requests). Son complementarios, no alternativos. En el futuro
(Plan 02 Fase 10) el Feedback Service puede exponer un `/metrics` endpoint
para Prometheus con las metricas de negocio, pero eso esta fuera de scope.

### Por que el consumer esta en feedback y no en cada servicio?

Principio de responsabilidad unica. Cada servicio PRODUCE eventos, el
Feedback Service CONSUME y analiza. Si el chat quiere registrar feedback,
publica a NATS y se olvida. El Feedback Service decide que hacer con eso.
Desacoplamiento total.

### Dual DB (tenant + platform) — por que?

- **Tenant DB (granular):** respeta el modelo de aislamiento. Los datos de
  feedback de un tenant estan en su propia DB. El tenant admin puede ver
  sus propios datos sin riesgo de cross-tenant leaks.
- **Platform DB (agregados):** el platform admin necesita vista cross-tenant.
  Los agregados (numeros, no datos sensibles) van a la platform DB. Nunca
  se copian datos granulares del usuario a platform DB.
