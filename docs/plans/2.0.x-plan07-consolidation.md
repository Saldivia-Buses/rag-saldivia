## Fase 1: Shared schema — migrations centralizadas

**Por que:** la Tenant DB es compartida por 6+ servicios. Tener migrations
en 6 directorios distintos crea dependencias ocultas y schema fragmentado.

**Que hacer:**

```
db/
  tenant/
    migrations/
      001_auth.up.sql          ← de services/auth/db/migrations/
      001_auth.down.sql
      002_chat.up.sql          ← de services/chat/db/migrations/
      003_ingest.up.sql        ← de services/ingest/db/migrations/ (001 + 002)
      004_notification.up.sql
      005_feedback.up.sql
      ...
    queries/                   ← sqlc queries compartidas (si aplica)
  platform/
    migrations/
      001_init.up.sql          ← de services/platform/db/migrations/
      002_feedback.up.sql
      003_credentials.up.sql
      004_intelligence.up.sql
      ...
```

- Mover TODAS las migrations de `services/*/db/migrations/` a `db/tenant/` y `db/platform/`
- Actualizar `db-init` script para leer de `db/` en vez de buscar en `services/`
- Los servicios mantienen sus `db/queries/` y `sqlc.yaml` (queries son per-service, schema es compartido)
- Borrar `pkg/services/` (directorios vacios, reemplazado por `db/`)

- Agregar tabla `schema_migrations` para tracking (evita re-aplicar en DBs existentes)
- Actualizar `migrate.sh` para leer de `db/` y trackear que migrations ya corrieron
- Actualizar los 8 `sqlc.yaml` para apuntar schema a `db/tenant/migrations/` o `db/platform/migrations/`

**Verificacion:** `make migrate` corre sin error en DB nueva y existente. `sqlc generate` funciona
en todos los servicios. Todos los servicios compilan. `db-init` funciona.

## Fase 2: `pkg/config` — centro de control del sistema

**Por que:** la arquitectura LEGO define slots, pipelines, prompts, rate limits
en la DB (agent_config, llm_models, prompt_versions). Pero ningun servicio lee
de la DB — todos usan env vars hardcodeados. El LEGO es config, no codigo.

**Que hacer:**

```go
package config

// Resolver lee configuracion de Platform DB con cascada de scopes.
// Todos los servicios importan este paquete en vez de leer env vars
// para config de negocio.
type Resolver struct {
    pool  *pgxpool.Pool  // Platform DB
    cache *redis.Client  // cache de config resuelta
}

// NewResolver crea un resolver conectado a Platform DB.
func NewResolver(pool *pgxpool.Pool, cache *redis.Client) *Resolver

// --- Config resolution ---

// Get resuelve un key con cascada: tenant:{id} > plan:{plan} > global.
// Retorna el value JSONB ya parseado.
func (r *Resolver) Get(ctx context.Context, tenantID, key string) (json.RawMessage, error)

// GetString/GetInt/GetFloat/GetBool — convenience wrappers con type assertion.
func (r *Resolver) GetString(ctx context.Context, tenantID, key string) (string, error)

// --- Slot resolution ---

// ResolveSlot resuelve un slot a su modelo + endpoint + config.
// slot.chat → "qwen3.5-27b" → {endpoint, model_id, temperature, max_tokens}
func (r *Resolver) ResolveSlot(ctx context.Context, tenantID, slot string) (*ModelConfig, error)

type ModelConfig struct {
    ID          string
    Name        string
    Endpoint    string
    ModelID     string  // ID en el endpoint (HuggingFace ID)
    APIKey      string  // encrypted, empty for local
    Temperature float64
    MaxTokens   int
    Location    string  // "local" o "cloud"
}

// --- Pipeline resolution ---

// LoadPipeline carga un pipeline con todos sus steps resueltos.
// Cada step tiene su slot resuelto a un ModelConfig.
func (r *Resolver) LoadPipeline(ctx context.Context, tenantID, pipelineKey string) ([]PipelineStep, error)

type PipelineStep struct {
    Step      string
    Enabled   bool
    Model     *ModelConfig  // nil si el step no usa LLM
    PromptKey string
    Prompt    string        // contenido resuelto de prompt_versions
    Params    map[string]any
}

// --- Prompt resolution ---

// GetActivePrompt retorna el prompt activo para un key.
func (r *Resolver) GetActivePrompt(ctx context.Context, promptKey string) (string, error)

// --- Tools resolution ---

// GetEnabledTools retorna las tools habilitadas para un tenant segun su plan.
func (r *Resolver) GetEnabledTools(ctx context.Context, tenantID string) ([]string, error)

// --- Rate limits ---

// GetRateLimits retorna los limites del tenant (ya resueltos por plan).
func (r *Resolver) GetRateLimits(ctx context.Context, tenantID string) (*RateLimits, error)

type RateLimits struct {
    QueriesPerMinute   int
    TokensPerDay       int
    CostLimitUSDMonth  float64
}

// --- Cache ---

// InvalidateCache invalida el cache de un tenant (llamado cuando platform
// publica un lifecycle event).
func (r *Resolver) InvalidateCache(ctx context.Context, tenantID string) error

// --- Infra helpers (lo que hoy es env() copy-paste) ---

// Env lee un env var con fallback. Reemplaza las 10 copias de env().
func Env(key, fallback string) string

// MustEnv lee un env var o panic. Para config requerida.
func MustEnv(key string) string
```

**Separacion:**

| Tipo | Fuente | Ejemplo |
|---|---|---|
| Infra (donde conectar) | ENV vars via `config.Env()` | `POSTGRES_URL`, `NATS_URL`, `PORT` |
| Negocio (como comportarse) | `config.Resolver` → Platform DB | Slots, pipelines, prompts, rate limits |

**Sub-fases:**

- **2a:** `config.Env()` y `config.MustEnv()` — reemplaza las 10 copias de `env()`.
  Esto es puro dedup, zero riesgo.
- **2b:** `Resolver` con `Get`, `GetString`, `ResolveSlot`, `GetActivePrompt` —
  las funciones que los servicios necesitan ahora (agent, search, ingest).
- **2c:** `LoadPipeline`, `GetRateLimits`, `GetEnabledTools` — capacidades nuevas
  que ningun servicio usa todavia. Implementar la interfaz pero wiring a servicios
  es fase 6.

**Tests requeridos:**
- `config.Env()` unit test
- `Resolver.Get()` con cascada de scopes (tenant > plan > global)
- `Resolver.ResolveSlot()` con modelo existente y no existente
- `Resolver.GetActivePrompt()` con prompt activo e inactivo

**Verificacion:** `make test-config` green. Cada servicio puede crear un Resolver.

## Fase 3: `pkg/llm` — un solo cliente OpenAI para todo el sistema

**Por que:** hay 3 implementaciones del mismo cliente HTTP:
- `services/agent/internal/llm/adapter.go` — con tools, otelhttp
- `services/ingest/internal/llm/client.go` — sin tools, sin otelhttp
- `services/search/internal/service/search.go:335-377` — inline, sin nada

**Que hacer:**

```go
package llm

// Client es el unico cliente OpenAI-compatible del sistema.
// Soporta: chat, tools, streaming, otelhttp, API key auth.
// Todos los servicios que hablan con SGLang o cloud usan este.
type Client struct { ... }

// NewClient crea un cliente. Usa otelhttp.NewTransport para trace propagation.
func NewClient(endpoint, model, apiKey string) *Client

// Chat envia mensajes con tools opcionales.
func (c *Client) Chat(ctx context.Context, msgs []Message, opts ...ChatOption) (*ChatResponse, error)

// SimplePrompt convenience para un solo mensaje sin tools.
func (c *Client) SimplePrompt(ctx context.Context, prompt string, temp float64) (string, error)

// NewFromModelConfig crea un Client desde un ModelConfig ya resuelto.
// No importa pkg/config — recibe el struct directamente.
func NewFromModelConfig(endpoint, modelID, apiKey string) *Client
```

- Mover `agent/internal/llm/adapter.go` a `pkg/llm/`
- Borrar `ingest/internal/llm/client.go`
- Borrar `search/service/search.go:llmChat()` inline
- Todos los servicios importan `pkg/llm`
- `NewFromSlot` conecta con `pkg/config` — un slot resuelto se convierte en un Client

**Verificacion:** `grep -r "internal/llm" services/` retorna 0 resultados.

## Fase 4: `pkg/nats` — conexion y validacion estandar

**Por que:** 3 patrones de conexion (full config vs minimal), 3 validaciones
de subject (allowlist vs denylist), drain vs close, old vs new JetStream API.

**Que hacer:**

```go
package natspub // ya existe, expandir

// Connect crea una conexion NATS con el patron estandar del proyecto.
// MaxReconnects(-1), RetryOnFailedConnect(true), handlers de log.
func Connect(url string) (*nats.Conn, error)

// IsValidSubjectToken exportado (ya existe como privado).
// Usa allowlist regex: ^[a-zA-Z0-9_-]+$
func IsValidSubjectToken(s string) bool
```

- Exportar `IsValidSubjectToken` (cambiar de `isValidSubjectToken`)
- Cambiar la logica de denylist a allowlist (regex)
- Agregar `Connect()` factory con el patron completo
- Todos los `cmd/main.go` usan `natspub.Connect()` en vez de `nats.Connect()` con config propia
- Todos usan `nc.Drain()` en shutdown (no `nc.Close()`)
- Traces service migra de `nc.JetStream()` a `jetstream.New()`
- Borrar `validateToken` de `agent/service/traces.go` (usa `pkg/nats`)
- Extractor Python: mantener `_SAFE_SUBJECT_RE` (no puede importar Go) pero
  documentar el regex canonico `^[a-zA-Z0-9_-]+$` en el paquete para que coincidan

**Verificacion:** `grep -r "nats.Connect(" services/` retorna 0 resultados (todos usan `natspub.Connect`).

## Fase 5: `pkg/security` — wiring del token blacklist

**Por que:** `TokenBlacklist` esta implementado y testeado pero no conectado
a ningun servicio. Un token revocado sigue funcionando.

**Que hacer:**

- Auth service `Logout()`: llamar `blacklist.Revoke(jti, expiresAt)` 
- `pkg/middleware/auth.go`: despues de verificar JWT, llamar `blacklist.IsRevoked(jti)`. Si revocado, devolver 401.
- Auth middleware necesita recibir el blacklist como dependencia (inyeccion, no global)
- Password change: llamar `blacklist.RevokeAll()` para invalidar todos los tokens del usuario

**Verificacion:** test de integracion: login → get token → logout → usar token → 401.

  10. HTTP server con otelhttp
  11. Graceful shutdown con `nc.Drain()`
