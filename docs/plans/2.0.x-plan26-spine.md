## Problema

NATS hoy es *transporte* — subjects validados + dos JetStream + payloads `any`. La auditoría del 2026-04-15 (60+ publish sites, 5 consumers durables) detectó cinco bugs operacionales que aplican a todos los servicios:

1. **Sin outbox.** Toda publicación es post-commit. Si NATS está caído entre `tx.Commit()` y `Publish()` el evento se pierde silenciosamente. La invariante #4 ("every write publishes") es disciplina, no garantía.
2. **Sin idempotencia.** Una redelivery de JetStream ejecuta el handler dos veces. Notificaciones, costos LLM, jobs de ingest pueden duplicarse.
3. **Sin panic recovery.** Si un handler de consumer panickea, la goroutine muere, el loop se detiene, los mensajes no se ackean. Requiere restart manual.
4. **DLQ logueado pero nunca consumido.** `pkg/nats/dlq.go` empuja a `dlq.{stream}.{subject}` pero ningún servicio se suscribe. Mensajes que exceden MaxDeliver desaparecen del sistema.
5. **Sin envelope ni schema versioning.** Payloads sin `event_id`, `correlation_id`, `causation_id`, `schema_version`, `timestamp`. Cambios de schema rompen consumers en silencio. Trace context sólo en headers, se pierde si alguien re-publica.

### Por qué Spine y no extender traces

Traces es vertical (LLM exec, costos, spans). Su schema (`model`, `tokens`, `usd_cost`) no generaliza a "cualquier evento". Spine es horizontal (envelope + outbox + framework para todos los publishers/consumers). Son complementarios. Cuando Spine esté en producción, traces puede migrar a `spine.Consume[TraceEvent]` y ganar gratis las garantías. Detalle en sección "Nota a futuro: traces".

### Relación con `audit_log` ya existente

`db/platform/migrations/005_audit_log.up.sql` ya define `audit_log` para acciones humanas de admin (`tenant.created`, `module.enabled`, etc.) con `user_id`, `action`, `resource`, `details`. Spine introduce `dead_events` para una intención distinta: eventos NATS que excedieron MaxDeliver. Son tablas separadas con propósitos separados:

- `audit_log` — qué hizo un humano (compliance, trazabilidad de acciones admin)
- `dead_events` — qué evento murió (operacional, replay/drop)

No fusionar. Mantener semánticas claras evita que la misma tabla mezcle "el admin Juan habilitó tal módulo" con "el evento `chat.new_message` murió 5 veces".

## Principios de implementación

### Compatibilidad incremental

`pkg/nats.Publisher` sigue funcionando con `Event` plano y la migración es **atómica por `Type`**, no big-bang global. Los consumers viejos **no** pueden parsear el envelope automáticamente porque leen `msg.Data` como `Event{UserID, Type, Title, Body, Data, Channel}` directo, no como `{id, tenant_id, payload, ...}`. Un `json.Unmarshal` contra envelope devolvería campos vacíos sin error.

Estrategia real:

1. **Dual-subject durante migración por Type.** Mientras un `Type` tenga productor o consumer sin migrar, el productor publica dos veces: al subject legacy (shape `Event` plano) y al subject envelope (shape `Envelope[T]`). La sobrecarga es temporal (horas o un deploy).
2. **Migración atómica por Type en un único PR.** Para cada `Type` (ej. `chat.new_message`): migrar productor + todos los consumers + frontend en el mismo PR. Al mergear, se activa Spine y el publish legacy se elimina en el deploy siguiente.
3. **Invariante durante la transición:** ningún `Type` vive con productor nuevo + consumer viejo. El lint `nc.Publish` y un check del bridge WS garantizan esto.

Esto requiere disciplina pero evita el modo silencioso de fallo ("consumer viejo recibe envelope, unmarshal sin error, campos vacíos, NOP silencioso").

### Tipado por codegen, no a mano

Toda nueva familia de evento se define en `pkg/events/spec/*.cue`. `make events-gen` produce structs Go, types TS, y doc Markdown. Cero drift entre productor y consumer (Go), y entre backend y frontend (TS).

```cue
// pkg/events/spec/notify.cue
events: "chat.new_message": {
    version: 1
    subject: "tenant.{slug}.notify.chat.new_message"  // shape actual de Notify
    payload: {
        user_id:    string  // target user (quien recibe el badge)
        session_id: string
        message_id: string
        title:      string
        body:       string
        channel:    "in_app" | "email" | "both"
    }
    publishers: ["chat"]
    consumers:  ["notification", "ws"]
}
```

### Envelope canónico

```go
// pkg/spine/envelope.go
type Envelope[T any] struct {
    ID            uuid.UUID  `json:"id"`              // UUIDv7 — time-ordered
    TenantID      string     `json:"tenant_id"`       // defensivo, redundante con subject
    Type          string     `json:"type"`            // "chat.new_message"
    SchemaVersion uint8      `json:"schema_version"`
    OccurredAt    time.Time  `json:"occurred_at"`
    RecordedAt    time.Time  `json:"recorded_at"`
    TraceID       string     `json:"trace_id,omitempty"`
    SpanID        string     `json:"span_id,omitempty"`
    CorrelationID string     `json:"correlation_id,omitempty"`
    CausationID   *uuid.UUID `json:"causation_id,omitempty"`
    ActorUserID   *string    `json:"actor_user_id,omitempty"`
    Payload       T          `json:"payload"`
}
```

### Outbox por tenant DB, no platform

Cada tenant DB tiene su tabla `event_outbox`. La razón: el `INSERT INTO outbox` debe estar en el mismo `tx` que el INSERT del mensaje/usuario/job. Esos viven en tenant DB, no platform. Drain es por tenant también.

```sql
CREATE TABLE event_outbox (
    id               uuid PRIMARY KEY,
    subject          text NOT NULL,
    payload          jsonb NOT NULL,
    headers          jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at       timestamptz NOT NULL DEFAULT now(),
    published_at     timestamptz,
    attempts         int NOT NULL DEFAULT 0,
    last_error       text,
    next_attempt_at  timestamptz                      -- NULL = eligible ahora; >now() = bloqueado por backoff
);
CREATE INDEX event_outbox_drainable
    ON event_outbox (created_at)
    WHERE published_at IS NULL;
-- Trigger para NOTIFY
CREATE OR REPLACE FUNCTION spine_outbox_notify() RETURNS trigger AS $$
BEGIN
    PERFORM pg_notify('spine_outbox_new', NEW.id::text);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER event_outbox_notify_insert
    AFTER INSERT ON event_outbox
    FOR EACH ROW EXECUTE FUNCTION spine_outbox_notify();
```

### Idempotencia en tenant DB (por-tenant, igual que outbox)

`processed_events` vive en cada tenant DB (no en platform). Razón: preserva la invariante #1 (tenant isolation at every layer), escala linealmente con el tenant, y permite que el consumer ejecute handler + `INSERT processed_events` en la **misma `tx`** contra la **misma DB**, lo que es lo que da la semántica effectively-once (sin 2PC cross-DB).

```sql
-- db/tenant/migrations/055b_processed_events.up.sql
CREATE TABLE processed_events (
    event_id      uuid NOT NULL,
    consumer_name text NOT NULL,
    processed_at  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (event_id, consumer_name)
);
CREATE INDEX processed_events_ttl ON processed_events (processed_at);
-- TTL job borra > 30 días (mismo horizonte que retention de JetStream)
```

No incluye `tenant_id` — la tabla vive dentro del schema de ese tenant, ya está aislada. La invariante #1 se cumple por construcción (no hay manera de leer filas de otro tenant sin cambiar de pool).

**Consumers sin tenant DB (ej. un consumer que actúa cross-tenant desde platform)** usan una tabla paralela `db/platform/migrations/010_platform_processed_events.up.sql` con `tenant_id` explícito. Hoy ninguno lo necesita — si aparece el caso, se crea; hasta entonces, la tabla platform no existe.

### Métricas estándar

Cada publish/consume emite a Prometheus:
- `spine_publish_total{service, subject, result}`
- `spine_consume_total{consumer, subject, result}`
- `spine_consume_duration_seconds{consumer, subject}`
- `spine_outbox_unpublished` (gauge, scrapeada del worker)
- `spine_dlq_depth{stream}` (gauge)

### Schema versioning policy (cuándo bumpear)

`SchemaVersion` es `uint8` monotónico — no hay semver (no hay major/minor). Cualquier cambio que no sea puramente aditivo requiere `version + 1` y coexistencia temporal.

- **Aditivo puro** (campo nuevo opcional con default sano): mismo `version`. Consumers viejos lo ignoran sin error. No requiere coordinación.
- **Breaking change** (rename, remove, change type, change semantics, opcional→required): bump `version + 1`. Proceso:
  1. Dejar la spec vieja en `pkg/events/spec/` con sufijo `_v1.cue` (ej. `chat.new_message_v1.cue`).
  2. Codegen produce ambas variantes (`Envelope[ChatNewMessageV1]`, `Envelope[ChatNewMessageV2]`).
  3. Productor emite V2 desde el merge; opcionalmente dual-publish (V1 + V2) si hay consumers que no migraron todavía.
  4. Consumers se migran uno a la vez.
  5. V1 se borra cuando métricas `spine_consume_total{schema_version="1"} == 0` por ≥ 14 días consecutivos.

**Regla dura:** nunca cambiar shape del payload manteniendo el mismo `version`. Eso es lo que rompe consumers en silencio hoy. El lint de CUE debe detectar esto (diff de spec con mismo version → error).

### WS hub bridge — coexistencia con envelope

`services/ws/internal/hub/nats.go` se suscribe a `tenant.*.>` (raw, fire-and-forget) y hoy hace `json.Unmarshal` esperando shape `Message{type, channel, data}`. Con envelope `{id, tenant_id, type, payload, ...}` el unmarshal cae al fallback y rompe el frontend. Solución en Fase 1:

- El bridge detecta envelope por presencia de campo `id` + `schema_version` en el JSON.
- Si es envelope: forwardea al frontend `{type: env.Type, channel: parts[2], data: env.Payload}`. El frontend nunca ve el envelope crudo — sigue consumiendo el mismo shape `Message`.
- Si NO es envelope (broadcasts viejos): comportamiento actual sin cambios.
- Test obligatorio en Fase 1: `TestNATSBridge_Envelope_UnwrapsPayload` verifica que el WS client recibe `Message` shape correcto cuando el publisher emite envelope.

Esto preserva 100% compat de frontend mientras Spine madura. Si en el futuro el frontend quiere acceso al envelope completo (causation_id, trace_id), se agrega como campo opcional `Message.envelope`.

### Outbox drainer — detalles operacionales

#### Concurrencia entre réplicas del mismo servicio

N réplicas de `services/chat` comparten la misma tenant DB. Si las N corren drainer, compiten por las mismas filas de `event_outbox`. La solución estándar **no** es un leader lock (fragil, añade zookeeper-like deps), sino `SELECT ... FOR UPDATE SKIP LOCKED`:

```sql
-- Drainer query (ejecutado en tx por cada ciclo)
SELECT id, subject, payload, headers, attempts
FROM event_outbox
WHERE published_at IS NULL
  AND (next_attempt_at IS NULL OR next_attempt_at <= now())
ORDER BY created_at ASC
LIMIT $1                -- batch size (ver backpressure)
FOR UPDATE SKIP LOCKED;
```

Cada réplica bloquea su propio subset, publica a NATS, hace `UPDATE event_outbox SET published_at = now() WHERE id = ANY($1)`, commit. Las filas bloqueadas son invisibles a otras réplicas durante la tx.

**Tradeoff:** pierde ordering global entre réplicas. Dos eventos del mismo `(session, user)` pueden salir out-of-order si las réplicas publican en paralelo. Se compensa con `OccurredAt` en el envelope — consumers que necesiten orden estricto ordenan por `OccurredAt` en la ventana idempotente.

#### Drainer multi-tenant

Cada tenant tiene su propia DB. El drainer mantiene un registry `map[tenantID]*DrainerWorker`. Arranque:

1. Al bootear, el servicio llama a `platform.ListActiveTenants()` y arranca un `DrainerWorker` por tenant (goroutine + pool dedicado).
2. **Suscripción a `platform.tenant.created`** (subject spine): al recibir el evento, hotloadea pool y arranca drainer nuevo. Simétrico con `platform.tenant.deleted` → stop + pool close.
3. Health endpoint `/spine/drainer/status` reporta por-tenant: `{tenant_slug, unpublished_count, last_drain_at, error_rate}`.

**SPOF por tenant:** si el pool de un tenant DB cae, su drainer entra en modo retry con backoff (10s, 30s, 60s, cap 5min). Métrica `spine_drainer_tenant_down{tenant="..."}` dispara alerta a los 2min. Los demás tenants no se ven afectados (aislamiento de pool).

#### Resto de detalles

- **Polling interval:** 200ms con backlog (`unpublished > 0`), 2s idle. Jitter ±100ms entre réplicas para evitar thundering herd.
- **Activación inmediata:** Postgres `LISTEN spine_outbox_new` / trigger AFTER INSERT → `NOTIFY`. Cada réplica que tiene LISTEN activo dispara drain inmediato; `FOR UPDATE SKIP LOCKED` evita que se pisen. Fallback al poll si LISTEN se cae.
- **Ordering:** **best-effort por tenant**, no estricto. Garantizado estrictamente solo cuando hay 1 réplica con backlog chico. Consumers que requieren orden usan `OccurredAt`.
- **Crash entre publish y mark:** `Publish()` a NATS retorna OK pero el drainer crashea antes del `UPDATE published_at`. Al reiniciar, el drainer republica. NATS re-entrega al consumer. Idempotencia (`processed_events`) garantiza no-op. **At-least-once + idempotent consume = effectively once.**
- **Backpressure dinámico:** el batch size se adapta:
  ```
  batch = clamp(unpublished / 100, min=1, max=1000)
  ```
  Con 10k acumulados → batch 100 por ciclo, ~500 msg/s con poll de 200ms. Drena en ~20s en lugar de 200s.
- **Retry exponencial por fila:** si `Publish()` falla, `UPDATE event_outbox SET attempts = attempts+1, last_error = $err, next_attempt_at = now() + backoff(attempts)` donde `backoff = min(60s, 2^attempts)`. La query del drainer filtra por `next_attempt_at` para no reintentar inmediatamente.
- **Alertas:** `spine_outbox_unpublished{tenant} > 1000` por 10min → warning. `> 10000` por 5min → critical.

### Replay no es time-travel

`POST /v1/admin/dlq/{id}/replay` re-publica al subject original con `event_id` nuevo. **El handler ejecuta contra el estado actual de la DB y código.** Si el evento es de hace 6 meses y el schema/lógica cambió, el resultado puede diferir del que hubiera tenido en ese momento. La UI debe mostrar "re-execute against current state" en lugar de "replay" para no engañar al operador. Replay es una herramienta operacional para resolver fallas transitorias, **no un mecanismo de auditoría histórica** (para eso está `audit_log` + futuro audit-store).

**JetStream stream nuevo:** `DLQ` con subject `dlq.>`, file storage, retention 30 días.

**Alta disponibilidad del consumer:**
- Healthwatch corre en **2 réplicas mínimo** (actualizar `docker-compose.prod.yml` y `k8s/healthwatch.yaml` como parte de Fase 4).
- El consumer DLQ usa JetStream **pull-based** con `DeliverGroup="healthwatch-dlq"` (queue subscription). Ambas réplicas compiten por mensajes; cada mensaje lo procesa una sola.
- Si una réplica cae, la otra recibe todos los mensajes sin interrupción.
- Si ambas caen: JetStream retiene los mensajes en `DLQ` stream por 30 días; al volver cualquier réplica drena el backlog.
- Test obligatorio: `TestDLQConsumer_FailoverBetweenReplicas` — arranca 2 consumers contra el mismo stream, tumba uno mientras mensajes entran, verifica zero pérdida y zero duplicación.

**Permissions nuevas:**
- `admin.dlq.read` (asignada a role `super_admin` por default)
- `admin.dlq.replay` (super_admin)
- `admin.dlq.drop` (super_admin)

**Replay semantics:**
- Replay genera **nuevo** `event_id` (UUIDv7 actual)
- Mantiene **mismo** `correlation_id`
- Setea `causation_id = original.event_id` (deja rastro de que vino de un replay)
- Publica al subject original
- Inserta una fila en `dead_events_replays` (tabla aparte):
  ```sql
  CREATE TABLE dead_events_replays (
      id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
      dead_event_id       uuid NOT NULL REFERENCES dead_events(id),
      replayed_at         timestamptz NOT NULL DEFAULT now(),
      replayed_by_user_id text NOT NULL,
      new_event_id        uuid NOT NULL,
      status              text NOT NULL CHECK (status IN ('pending','succeeded','failed'))
  );
  CREATE INDEX dead_events_replays_dead ON dead_events_replays (dead_event_id, replayed_at);
  ```
  (Guardar historial, no solo el último, porque replay es una operación auditable y los operadores pueden necesitar saber cuántas veces se intentó.)
- `dead_events` mantiene un `replay_count` incrementado por trigger, y `last_replayed_at` para display rápido en la lista.

**Concurrencia de replay:**
- `POST /v1/admin/dlq/{id}/replay` toma `SELECT ... FOR UPDATE` sobre `dead_events` row. Dos requests simultáneos: uno gana el lock, el otro recibe `409 Conflict` con mensaje "replay already in progress for this event".
- Si el replay original ya terminó y el operador quiere replayear de nuevo: no se bloquea — se inserta un `dead_events_replays` nuevo. Es intencional (útil para debug), pero el UI muestra warning "este evento ya se replayeó N veces" a partir de 3+.
- `drop` es terminal: si `dead_events.dropped_at IS NOT NULL`, replay retorna `410 Gone`.

**Tests:**
- DLQ consumer recibe message → persiste a `dead_events` con todos los headers
- Replay endpoint genera nuevo envelope y publica a subject original
- Drop endpoint marca `dropped_at` sin republicar
- RBAC: usuario sin `admin.dlq.read` → 403

- [ ] Test E2E manual: forzar un consumer a fallar 5 veces (max deliver), verificar que aparece en `/admin/dlq`, hacer replay, verificar que se procesa OK

## Riesgos y mitigaciones

| Riesgo | Probabilidad | Impacto | Mitigación |
|---|---|---|---|
| Drainer se atrasa, outbox crece sin control | Media | Medio | Métrica `spine_outbox_unpublished`, alerta a 1000, runbook explica cómo investigar |
| Tabla `processed_events` crece sin parar | Alta | Bajo | TTL job nocturno borra > 30 días, alineado con retention de JetStream |
| Codegen rompe consumers viejos en migración parcial | Baja | Alto | `Envelope[T]` siempre se serializa con `schema_version` + payload; consumers que leen `data`/`type` viejo siguen funcionando porque `Event` se incluye literal en payload de envelope V1 |
| Latency extra del INSERT outbox en write paths críticos | Media | Bajo | Medido en Fase 3 con benchmarks; típicamente <2ms en postgres local. Si dolió: mover drainer a tx batched |
| Dependencia nueva (CUE) | Media | Medio | CUE binary distribuido, makefile lo descarga; CI lo cachea. Fase 0 incluye tutorial `docs/conventions/cue.md` + ejemplos commented contra los 4 `Type` iniciales (notify, ingest, auth, platform) para reducir curva. Si después de Fase 1 el equipo encuentra CUE friccional, se evalúa migración a JSON Schema (decisión reevaluable, no atada al diseño). |
| Healthwatch crece más allá de su mandato | Baja | Bajo | DLQ es self-healing por naturaleza; encaja bien con healthwatch. Si crece mucho, escindir a `services/spine-dlq` separado |
| Alguien sigue usando `nc.Publish()` directo | Media | Alto | Lint custom: `golangci-lint` con regla que prohíbe `nc.Publish` fuera de `pkg/spine` y `pkg/nats` |
