# ADR-014: Ed25519 JWT signing

**Fecha:** 2026-04-02
**Estado:** Aceptado

---

## Contexto

The auth service needs to issue JWTs that all other services can verify independently (no auth service roundtrip). In 1.0.x, tokens were signed with HS256 (HMAC-SHA256), which uses a symmetric shared secret.

Problems with HS256 in a microservices architecture:

- Every service that verifies tokens needs the same secret key
- Any service that has the secret can also forge tokens (no signing/verification separation)
- Compromising one service compromises the signing key for all services
- Secret rotation requires simultaneous update across all 8 services

With 8 services + CLI + MCP server all needing to verify tokens, the blast radius of a symmetric key compromise is the entire system.

## Opciones consideradas

- **Opcion A -- HS256 (symmetric):** Simple, fast. Pros: one env var (`JWT_SECRET`), fast verification. Contras: every service can forge tokens, key compromise = total compromise, rotation is all-or-nothing.

- **Opcion B -- RS256 (RSA 2048-bit):** Asymmetric. Pros: only auth service has private key, others only need public key. Contras: RSA keys are large (2048+ bits), signatures are ~256 bytes, slower than Ed25519.

- **Opcion C -- Ed25519 (EdDSA):** Asymmetric, modern. Pros: 32-byte keys, 64-byte signatures, faster than RSA for both signing and verification, constant-time operations (side-channel resistant), Go stdlib support via `crypto/ed25519`. Contras: less widespread than RSA (but supported by all modern JWT libraries).

## Decision

**Opcion C -- Ed25519.** Implementation:

- Auth service loads both `JWT_PRIVATE_KEY` (signing) and `JWT_PUBLIC_KEY` (verification)
- All other services load only `JWT_PUBLIC_KEY` (verification)
- Keys are base64-encoded PEM, passed via env vars (or Docker secrets in production)
- Key parsing via `pkg/jwt.ParsePrivateKeyEnv()` and `pkg/jwt.ParsePublicKeyEnv()`
- Token verification via `pkg/jwt.Verify()` using `golang-jwt/jwt/v5`

## Consecuencias

**Positivas:**
- Only the auth service can sign tokens -- all other services can only verify
- Compromising any non-auth service does not compromise token signing
- 32-byte public key is trivial to distribute (single env var)
- Ed25519 verification is faster than RSA-2048 (~8x on typical hardware)
- Go stdlib `crypto/ed25519` -- no external dependency for the core crypto

**Negativas / trade-offs:**
- Key generation requires a one-time setup step (`openssl` or Go utility)
- Two env vars instead of one (`JWT_PRIVATE_KEY` + `JWT_PUBLIC_KEY` vs single `JWT_SECRET`)
- Ed25519 is less common in enterprise environments that mandate RSA or ECDSA
- Cannot use JWT token introspection (tokens are self-contained, no revocation check) -- mitigated by short access token TTL + refresh token rotation
