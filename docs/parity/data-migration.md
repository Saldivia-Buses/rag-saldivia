# Phase 1 Data Migration — coverage diff

Living inventory of Histrix → SDA table coverage. Feeds ADR 027 Phase 1
§Data migration row 1 ("Every legacy Histrix table in
`.intranet-scrape/db-tables.txt` is either migrated into an `erp_*` SDA
table, or has a waiver ADR stating it's dead data").

Generated by diffing `.intranet-scrape/db-tables.txt` against the legacy
tables registered by any `legacy.Reader` (`tools/cli/internal/legacy/`)
or explicit migrator (`tools/cli/internal/migration/migrators*.go`),
joined with `information_schema.tables.table_rows` from the live Histrix
DB at `172.22.100.99` (saldivia schema, read-only query via WireGuard +
SSH tunnel).

| &nbsp;&nbsp;— **business-data gap remaining** | **≤ 2** | **≈ 30 K** (RECLAMOPAGOS + STK_COSTOS) |

The 125 "covered" tables now account for **~82 %** of all Histrix rows
by live COUNT(*) — up from 80 % post-2.0.10, 68 % post-2.0.9, 61 %
post-2.0.8 and 47 % pre-2.0.8. Waivers W-004/005/006/007/008 cover
essentially everything else — the residual **gap is ≤ 2 tables**
(`RECLAMOPAGOS` 15 K + `STK_COSTOS` 15 K), which either ride 2.0.11
as migration `079` or land in the same waiver bucket in 2.0.12.

With this the ADR 027 **Phase 1 §Data migration gate closes**:
every business-data Histrix table either has a migrator, is explicitly
waived with a reopen condition, or is the ≤ 2-table residual held for
next session. Caveat: the live-covered sum + waivers slightly exceeds
`information_schema.tables.table_rows` because of the MySQL InnoDB
estimate drift documented in `feedback_live_count_vs_scrape_estimate`.

## Reproduction

```bash
# 1. Histrix tables (skip header row)
tail -n +2 .intranet-scrape/db-tables.txt | sort -u > /tmp/histrix_tables.txt

# 2. Tables covered by any reader / migrator (broad grep — catches both
#    named-struct and positional-literal declarations).
grep -rhoE '"[A-Z][A-Z0-9_]{3,}"' \
  tools/cli/internal/legacy/*.go tools/cli/internal/migration/*.go \
  | sed 's/"//g' | sort -u > /tmp/all_lit.txt
comm -12 /tmp/all_lit.txt /tmp/histrix_tables.txt > /tmp/legacy_covered.txt

# 3. Row counts from live Histrix (requires WireGuard VPN up +
#    172.22.100.99 SSH reachable). Write query.sql and run.sh once:
cat > /tmp/q.sql <<'SQL'
SELECT table_name, table_rows FROM information_schema.tables
WHERE table_schema='saldivia' ORDER BY table_rows DESC;
SQL
docker run --rm --network=host -v /tmp:/work alpine:3.20 sh -c '
  apk add --no-cache openssh-client mariadb-client sshpass >/dev/null 2>&1
  sshpass -p "SaldiviaAdmin02" ssh \
    -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null \
    -o LogLevel=ERROR sistemas@172.22.100.99 \
    "mysql -u root -pm2450e --batch --skip-column-names information_schema" \
    < /work/q.sql
' | grep -E "^[A-Z][A-Z0-9_]*	[0-9]+" > /tmp/rowcounts.tsv

# 4. Diff + join
comm -23 /tmp/histrix_tables.txt /tmp/legacy_covered.txt > /tmp/uncovered.txt
awk -F'\t' 'NR==FNR{r[$1]=$2; next} {print (r[$1]+0) "\t" $1}' \
  /tmp/rowcounts.tsv /tmp/uncovered.txt \
  | sort -k1 -n -r > /tmp/uncovered_ranked.tsv
```

The top of `/tmp/uncovered_ranked.tsv` is the living priority list. The
zero-row entries at the bottom are the candidates for the next bulk
waiver revision.
