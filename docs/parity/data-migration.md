# Phase 1 Data Migration — coverage diff

Living inventory of Histrix → SDA table coverage. Feeds ADR 027 Phase 1
§Data migration row 1 ("Every legacy Histrix table in
`.intranet-scrape/db-tables.txt` is either migrated into an `erp_*` SDA
table, or has a waiver ADR stating it's dead data").

Generated by diffing `.intranet-scrape/db-tables.txt` against the legacy
tables registered by any `legacy.Reader` (`tools/cli/internal/legacy/`)
or explicit migrator (`tools/cli/internal/migration/migrators*.go`).

## Totals (2026-04-18)

| Segment | Count |
|---|---:|
| Histrix tables in `.intranet-scrape/db-tables.txt` | 675 |
| Tables with a registered migrator/reader | 97 |
| Tables uncovered (total) | 578 |
| &nbsp;&nbsp;&nbsp;&nbsp;— Histrix infra (HTX*, `*_OLD`) — waived | 36 |
| &nbsp;&nbsp;&nbsp;&nbsp;— Business-data gap to close (migrator or waiver) | 542 |

Covered % of Histrix surface: **14.4 %** by table count (97 / 675).

The "97 covered" figure counts tables that have a declared reader, not
tables whose rows actually migrate cleanly — Phase 0 item 1 covers the
latter. So this is coverage **intent**; Phase 0 integrity queries own
whether each migrator behaves.

## Next ranking signal: row counts on Histrix

Table-count coverage is a weak prioritizer by itself — a 12-row config
table is not equivalent to a 4 M-row movement table. The next improvement
to this doc is to join the uncovered list with `SELECT COUNT(*)` on the
Histrix DB (root/m2450e @ sistemas@172.22.100.99, db `saldivia`, via the
WireGuard VPN — see the `infrastructure-access` skill). Every follow-up
session that writes a new migrator should pick from the top of the
row-count-weighted list, not alphabetically.

## How to extend this doc

Each session that touches a new uncovered area should:

1. Add a per-domain section below (e.g. `## STK* — stock extended`).
2. List the tables it covered, the migrator(s) added, the tests.
3. Waive the rest of the area explicitly (new waiver entry) or leave
   an open `TODO` with the row-count justification for keeping it on
   the roadmap.
4. Update the "Totals" table at the top and tick the ADR 027 Phase 1
   §Data migration row when the uncovered count reaches 0.

## Reproduction

```bash
# 1. Histrix tables (skip header row)
tail -n +2 .intranet-scrape/db-tables.txt | sort -u > /tmp/histrix_tables.txt

# 2. Legacy tables covered by any reader / migrator.
{
  grep -hE 'Table:\s*"[A-Z][A-Z0-9_]+"' tools/cli/internal/legacy/*.go \
    | sed -E 's/.*Table:\s*"([A-Z][A-Z0-9_]+)".*/\1/'
  grep -hE '^\s*\{"[A-Z][A-Z0-9_]*"' tools/cli/internal/legacy/catalogs.go \
    | sed -E 's/^\s*\{"([A-Z][A-Z0-9_]*)".*/\1/'
  grep -hE 'LegacyTable:\s*"[A-Z][A-Z0-9_]+"|legacyTable:\s*"[A-Z][A-Z0-9_]+"' \
    tools/cli/internal/migration/*.go tools/cli/internal/legacy/*.go \
    | sed -E 's/.*[Ll]egacyTable:\s*"([A-Z][A-Z0-9_]+)".*/\1/'
  printf 'HTXPROFILES\nHTXUSERS\n'
} | sort -u > /tmp/legacy_covered.txt

# 3. Diff
comm -23 /tmp/histrix_tables.txt /tmp/legacy_covered.txt > /tmp/histrix_uncovered.txt
```

When the Histrix DB is reachable (corp network + WireGuard), join with:

```sql
SELECT table_name, table_rows
FROM information_schema.tables
WHERE table_schema='saldivia'
  AND table_name IN (<uncovered list>)
ORDER BY table_rows DESC;
```

to get the prioritization signal.
