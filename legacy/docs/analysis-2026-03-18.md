## CRÍTICOS (bloquean producción)

### C1 — Milvus 2.6.5 sparse field incompatibility ⚠️

**Síntoma:** `MilvusException: not allowed to retrieve raw data of field sparse`

**Root cause confirmado:** La colección `tecpia_test` tiene campo `sparse` (BM25 hybrid).
`langchain_milvus._collection_search` usa `output_fields = self._remove_forbidden_fields(self.fields[:])`.
En DENSE mode, `builtin_func=None` → `_remove_forbidden_fields` no excluye nada → `sparse` queda en output_fields → Milvus 2.6.5 lo rechaza.

**Mi error previo:** Parcheé el branch `if enable_dynamic_field:` (línea 1601) pero la colección
tiene `enable_dynamic_field=False` → el código nunca entra ahí. El bug está en el branch `else` (línea 1604).

**Afecta 4 métodos:** `_collection_search` (L1604), `_collection_hybrid_search` (L1729), y dos más (L2749, L2867).

**Fix:** Patch el branch `else` en los 4 métodos:
```python
else:
    output_fields = self._remove_forbidden_fields(self.fields[:])
    output_fields = [f for f in output_fields if f != "sparse"]  # ADD
```

**Aplicación:** `deploy.sh` vía `docker exec` + `sed` (como vlm.py). Se re-aplica en cada deploy.

**Alternativa a largo plazo:** Re-ingestar en colección DENSE pura (sin campo sparse).

### C2 — .env.saldivia NO llega a los containers en runtime ⚠️

**Síntoma:** De los 27 vars de `.env.saldivia`, solo 4 llegan al RAG server en runtime (via `ENV_MAPPING`).

**Root cause:** `deploy.sh` genera `.env.merged` solo con `config.py` → `generate_env()` → `ENV_MAPPING`.
`ENV_MAPPING` tiene 15 vars. Las otras 23 de `.env.saldivia` **no se generan**.

**Vars críticas que faltan en runtime:**
```
APP_VECTORSTORE_SEARCHTYPE=hybrid    # ← Blueprint usa 'dense' por default (por eso funciona)
APP_VECTORSTORE_INDEXTYPE=HNSW      # ← Blueprint usa GPU_CAGRA
APP_VECTORSTORE_ENABLEGPUSEARCH=False
APP_VECTORSTORE_ENABLEGPUINDEX=False
VECTOR_DB_TOPK=100
APP_RETRIEVER_TOPK=25
APP_NVINGEST_CAPTIONMODELNAME=qwen3-vl-8b
APP_NVINGEST_EXTRACTIMAGES=True
APP_VLM_MAX_TOKENS=1024
FILTER_THINK_TOKENS=true
ENABLE_SOURCE_METADATA=true
... (y 12 más)
```

**Consecuencia real:** El sistema funciona en dense (Blueprint default) pero sin NINGUNA de
nuestras optimizaciones de retriever, VLM, NV-Ingest, o features.

**Fix:** En `deploy.sh`, incluir `.env.saldivia` como base de `.env.merged`:
```bash
# Base: .env.saldivia (defaults Saldivia)
cp "${SALDIVIA_ROOT}/config/.env.saldivia" "$ENV_FILE"
# Override: vars del Python config (model names, endpoints)
python3 -c "loader.write_env_file('/tmp/config_env.txt', profile=PROFILE)"
cat /tmp/config_env.txt >> "$ENV_FILE"
# Override: secrets
cat "${SALDIVIA_ROOT}/.env.local" >> "$ENV_FILE"
# Override: runtime paths
echo "PROMPT_CONFIG_FILE=..." >> "$ENV_FILE"
```

### C3 — ingestion-worker en crash loop ⚠️

**Síntoma:** `redis.exceptions.ConnectionError: Error 111 connecting to localhost:6379`

**Root cause:** El worker conecta a `REDIS_URL` que defaultea a `redis://localhost:6379`.
Dentro del container Docker, `localhost` no tiene Redis. El hostname correcto es `compose-redis-1` (o `redis`).

**Fix:** En `compose-platform-services.yaml`:
```yaml
ingestion-worker:
  environment:
    - REDIS_URL=redis://redis:6379
```
O mejor, agregar `REDIS_URL` al `ENV_MAPPING` generado por config.

---

### C4 — nim-llm network alias se rompe en cada redeploy ⚠️

**Síntoma:** `nemotron3-super` no tiene alias `nim-llm` en la red `nvidia-rag`.

**Root cause:** `docker compose up --force-recreate` desconecta y reconecta containers a la red,
perdiendo aliases manuales. El step 6 de `deploy.sh` falla silenciosamente si el container
ya está en la red (sin el alias).

**Fix:**
```bash
docker network disconnect "$NETWORK" nemotron3-super 2>/dev/null || true
docker network connect --alias nim-llm "$NETWORK" nemotron3-super \
    && log "  nim-llm alias connected" \
    || warn "  nim-llm alias failed"
```

## IMPORTANTES (seguridad y correctitud)

### I1 — BYPASS_AUTH=true trata user=None como ADMIN

**Código:**
```python
def filter_collections(user: User, requested: list[str]) -> list[str]:
    if user is None or user.role == Role.ADMIN:  # ← None = admin!
        return requested
```

Cuando `BYPASS_AUTH=true`, `user=None` pasa por todos los checks sin restricciones.
El endpoint `/v1/collections` también expone todas las colecciones cuando `user is None`.

**Fix:** `BYPASS_AUTH` debería estar prohibido en producción. Agregar check:
```python
if os.getenv("ENVIRONMENT", "production") == "production" and BYPASS_AUTH:
    raise RuntimeError("BYPASS_AUTH cannot be enabled in production")
```
Y en `filter_collections`: si `user is None` y no hay bypass, lanzar 401.

---

### I2 — /v1/documents no valida acceso por colección

El endpoint de ingesta verifica el role del usuario pero no qué colección está ingresando.
Un `AREA_MANAGER` puede ingestar en cualquier colección, no solo las de su área.

**Fix:** Extraer `collection_name` del body multipart y verificar permisos.

---

### I3 — Gateway propaga 200 cuando upstream devuelve 400

**Síntoma:** Gateway logs: `POST /v1/generate HTTP/1.1 200 OK` pero `rag-server returns 400 Bad Request`.

El `StreamingResponse` siempre devuelve 200 independientemente del status del upstream.
El cliente recibe un stream con datos de error sin saber que falló.

**Fix:** Verificar el status code del upstream antes de iniciar el stream:
```python
async with client.stream("POST", ...) as resp:
    if resp.status_code >= 400:
        error_body = await resp.aread()
        raise HTTPException(status_code=resp.status_code, detail=error_body.decode())
    async for chunk in resp.aiter_bytes():
        yield chunk
```

### I4 — MCP server usa localhost hardcodeado

```python
resp = await client.post("http://localhost:8081/v1/search", ...)
```

No usa `RAG_SERVER_URL` ni maneja errores de status code. Claude fallará si RAG devuelve error.

**Fix:** Usar env var + check de status.

---

### I5 — Milvus collection no liberada después de stats()

`collections.py:stats()` hace `col.load()` pero nunca `col.release()`.
Cada llamada acumula memoria en Milvus.

**Fix:** Agregar `col.release()` al final o usar try/finally.

---

### I6 — ingestion_worker sin retry logic

Fallos transitorios (timeout de red, NV-Ingest ocupado) se marcan como permanentes.
No hay reintento, el job muere.

**Fix:** Retry con backoff exponencial (3 intentos, espera 10s/30s/60s).

---

### I7 — vlm.py patch usa container name incorrecto

`deploy.sh` busca `--filter "name=nv-ingest"` pero el container real es `compose-nv-ingest-ms-runtime-1`.
El filtro por substring puede matchear mal o no matchear.

**Fix:** Usar nombre exacto `compose-nv-ingest-ms-runtime-1`.

## MENORES (calidad y robustez)

### M1 — config.py ENV_MAPPING incompleto (diseño)
El problema C2 no es solo un bug de deploy.sh — es que la arquitectura
"YAML → ENV_MAPPING → .env.merged" nunca fue completada. La solución correcta
es expandir ENV_MAPPING O abandonar ese enfoque y simplemente incluir `.env.saldivia` directamente.

### M2 — Milvus collection release en stats()
(ver I5 arriba)

### M3 — Job ID truncado a 8 chars
`str(uuid.uuid4())[:8]` — colisión posible con muchos jobs. Usar ID completo.

### M4 — ingestion_worker sin signal handling
`docker stop` mata el proceso a la mitad de una ingesta. Agregar `signal.SIGTERM` handler.

### M5 — Deploy health check solo advierte, no falla
Si después de 2 min los servicios no responden, `deploy.sh` continúa con `warn`.
Debería exitear con código de error.

### M6 — Watch module: _processed set no persiste entre restarts
Si el watcher se reinicia, re-procesa todos los archivos.
Usar Redis para persistir el set.

### M7 — AuthDB crea nueva conexión SQLite por operación
Overhead innecesario. Usar connection pool o singleton.

### M8 — SQLite sin garantías transaccionales
`database.py` usa `with self._conn() as conn:` que hace auto-commit por operación.
Si hay una excepción entre el INSERT y el cierre del context manager, no hay rollback garantizado.
Con requests concurrentes (FastAPI), hay riesgo de race condition al crear users/areas.
**Fix:** Usar `conn.execute("BEGIN")` explícito o `isolation_level=None` con BEGIN/COMMIT manual.

### M9 — validate_config ocurre tarde (solo en deploy.sh)
`validate_config()` se llama desde `deploy.sh` pero no en el startup del gateway ni del worker.
Si alguien arranca un servicio manualmente sin pasar por `make deploy`, la config inválida
se descubre en el primer request (401 en LLM, etc.) en vez de al arrancar.
**Fix:** Llamar `validate_config()` en el `@app.on_event("startup")` del gateway.

### M10 — Audit log se loguea después del action, no en auth failures
`gateway.py` loguea el action DESPUÉS de que la operación exitó. Si falla, no queda registro.
Tampoco se loguean intentos fallidos de autenticación (401/403) — útil para detectar ataques.
**Fix:** Agregar `log_action(action="auth_failure")` en el handler de 401/403.

### M11 — Redis connection pool nunca se cierra
`IngestionQueue.__init__` crea `redis.from_url(redis_url)` (connection pool) pero nunca
llama `self.redis.close()`. En procesos long-running (worker, mode-manager) el pool acumula
conexiones sin liberar. Con suficiente actividad, agota el límite de conexiones de Redis.
**Fix:** Implementar `__del__` o context manager con `self.redis.close()`.

### M12 — mode_manager._wait_for_vlm_healthy usa localhost
El container de mode-manager intenta `http://localhost:8000/health` — debería usar
el hostname Docker del VLM container.

### M12 — compose-overrides.yaml está vacío
`services: {}` no hace nada. Si no se necesita, eliminarlo para claridad.

### M13 — full-cloud.yaml tiene model names incorrectos
Usa `nv-embedqa-e5-v5` y `nv-rerankqa-mistral-4b-v3` — los mismos nombres incorrectos que
ya corregimos en models.yaml.
